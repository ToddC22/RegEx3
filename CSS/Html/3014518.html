How to process a request for a SharePoint Online site restore

This document defines the process to run a Point in Time Restore (PITR) for customer content within SharePoint Online. These are typically the result of a customer opening a support case with the request to restore data to a previous point in time. The process that's defined within this document must be followed for each request. This process will continue to change based on engineer feedback and the evolution of the service. If there are any questions about the PITR process, direct them to GBS SPO Site Restore Help.
Restore your OneDrive and Restore a document library have been made publicly available to customers in SharePoint Online. Please consider reviewing these options with your customer as it may result in a much quicker resolution for the customer. Should these features not resolve the issue, please make sure to collect the details of the failure or why it did not meet the customer's needs and escalate to engineering (SNAP/ContentStore).
If you’re taking a new customer case and based on the initial notes, it seems that data has to be recovered on the site by using any method (recycle bin, versioning, or PITR), we have to act quickly to collect data and set expectations from the beginning.

Examples of such a case might be where a customer calls out that they:
  • Are missing content from their site.
  • Are missing a site or content was purged from the recycle bin that they have to recover.
  • Have deleted columns from a list or library.
  • Overwritten critical data on a site that must be recovered.
Ideally, our first contact with the customer is by using the telephone so that we can collect scoping data and help the customer as soon as possible. If our first contact with the customer is through an email message, here's an email template that you can consider using to set expectations accordingly and to quickly obtain important questions from the customer.

Step 1: Scope the Restore Request

 

Whether it is through the first telephone call or email messages, on a case in which we are investigating a potential restore, acknowledge the request with the customer and then set the expectation that a restore may cause the following behavior:
  • During the restore, the site may temporarily become unavailable.
  • For an whole site collection restore, changes that were made between the requested restore time and present time will be lost.
Advise the customer that you'll be trying to prevent the need for a restore by solving their issue. In order to fully understand the scope of the customer’s restore request, document the answers to the following questions within the support case:
  • What is the URL of the site collection that contains the item, or items, that have to be restored?
  • What is the URL to the item, or items, that has to be restored? If this is unknown, specify a location as close as possible.
  • What is the reason for the restore request? For example, missing item, unwanted changes, and so on.
  • When were the item, or items, last known to exist in the desired state?
Step 2: Exhaust all non-PITR solutions to the issue

 

There are a several reasons that a customer may request a restore of their site collection or subsite. At this point, you should have a good understanding of why the customer is requesting a restore. Based on this understanding, an effort has to be made to resolve the customer's issue without requesting a site restore. The common reasons for PITR requests are documented here together with the steps that should be taken to avoid the PITR, if it's possible.

Be aware that Office 365 Enterprise customers can restore a site collection through organization admin. For more information, go to the following Microsoft websites:
  • How to recover a deleted subsite from the SharePoint Online recycle bin
  • Manage the Recycle Bin of a SharePoint site collection
  • Restore a deleted site collection
  • OneDrive file restore is available to restore files in bulk and can be very helpful to recover from events like Ransomware attacks
Step 3: Determine scope and point in time for the restore


There are three scope options for a Point in Time Restore.

1. Site collection to the same Url

- This is the preferred scope of restore as is a full fidelity restore and supports the widest range of scenarios, however as it is to the same Url the current site will be deleted in order to be replaced with the restored site. There will be downtime with this option and if the target is the root site, all sites will have downtime until the restore is complete.

2. Site collection to a new Url

- This is a new option that is currently in preview. There will be no data loss or downtime impact to any existing sites so is ideal, however it is currently limited to sites less than 50 GB and is a best effort support scenario. If the site is greater than 50 GB or the PiTR should fail while creating the site backup or restoring the site from the backup, another option will need to be considered.

3. Subsite (web/subweb)

- This is a best effort scenario, and may delay the overall restore process. This is not a full-fidelity restore. For example, Workflows, Apps from the App Store, checked-out items, and items in the recycle bin won't be restored. If the restore fails during export or import of the web, the only option is a site collection restore. This option is not available for the rootweb of a site collection.

Based on this information, the scope of the restore will be a URL in one of the following formats:

  • Site Collection: The URL should map to the root of a site collection.
    • Example 1: https://<domain>.sharepoint.com
    • Example 2: https://<domain>.sharepoint.com/sites/TeamSite
  • Subweb or subsite: The URL should map to the subweb that's the target of the restore. This can't be a URL to the root of a site collection.
    • Example 1: https://<domain>.sharepoint.com/subSite
    • Example 2: https://<domain>.sharepoint.com/sites/TeamSite/subSite1/subSite2
After the scope of the restore is set, you must identify the point in time that the data should be restored from. The time should be selected based on the last known time in which the item, or items, were in the desired state, and must be no more than 14 days in the past. For example, consider the following scenario:
The reason for the restore is that a document was noticed as missing on Friday. The last time any of the users are confident that the document existed was the previous Monday. In this scenario, the point in time for the restore should be set to a point before the user confirming the presence of the file on Monday.
The actual point in time of the restore will typically be within 15 minutes of the time specified. A single point in time should be defined by using the understanding that the restore will likely be to several minutes before the specified time. This time has to be confirmed with the customer and provided as a UTC time in the following 24 hour format: MM-DD-YYYY hh:mm:ss. For example, 10/22/2014 4:43:00 UTC.

Note The point in time for the restore requested by the customer will be validated by the escalation resources before the execution of the PITR with the tools documented here. If the point in time given by the customer doesn't match the data returned by the tools available, additional review will be necessary.
Step 4: Establish expectations

 

Now that the scope and point in time are established, expectations around the restore process must be set correctly with the customer. These expectations can vary based on the restore scope determined in the previous step. The following verbatim should be sent to the customer in an email message. Make sure that the correct verbatim is sent based on the scope of the restore and document the customers consent to move forward with the restore within the support case before executing the restore. Whereas it isn't called out directly, the site collection verbatim that's listed here implies that the changes that occurred between the point in time of the restore and the restore execution time will be lost. This supports the fact that a second restore to revert the site to post-restore state shouldn't occur. 

Site collection - to the same Url

The next course of action to move our support case forward is a restore of the site collection located at <Insert Site Collection URL>. In order to complete the restore, this site collection will first be removed and then will be replaced with a copy of the site from within 15 minutes of the established restore point (<Insert Restore Time (UTC)>). This process typically completes within 6 to 12 hours, but could take as long as 72 hours to complete. Considering the amount of time needed for the restore, it's important that there is a high level of confidence with the point in time that was chosen for the restore. If the root site collection of the tenant (ex: https://contoso.sharepoint.com) is the target for the restore, all sites within the tenant will be unavailable after the site is removed and will be inaccessible until the restore has been completed.

Due to the nature of the restore and the site rollback, it's recommended that the following steps are taken prior to moving forward with the restore.
  1. Users of the site collection should be notified that the site is now considered to be in a read-only mode as any changes made after the point in time for the restore will be lost. The users should also anticipate that the site won't be accessible for a considerable amount of time during the restore.
  2. In order to minimize data loss, the items that have been updated after the point in time specified for the restore should be saved to a local computer in order to replace the items upon completion of the restore.
  3. Please stop the synchronization of OneDrive sync clients on any libraries located under the path of this restore.

Microsoft will notify you once the restore completes. Upon completion of the restore, the following steps should take place to complete the restore process:
  1. The users should be notified that the site restore has completed and is again accessible for read/write activity.
  2. Any files that were backed up locally should be replaced within the site as needed.

Once you are ready to proceed with the restore, reply-all to this email confirming that you understand that the site at the URL listed above will be rolled back to the state it was in at the time specified above. In addition, you understand the recommended steps that should be executed before and after the restore. Once you provide confirmation to move forward with the restore, Microsoft will start the restore process.

Site collection - to a new Url

The next course of action to move our support case forward is a restore of the site collection located at <Insert Site Collection URL>. In order to complete the restore, a new site collection will be created from the backup created within 15 minutes of the established restore point (<Insert Restore Time (UTC)>). This process typically completes within 6 to 12 hours, but could take as long as 72 hours to complete. Considering the amount of time needed for the restore, it's important that there is a high level of confidence with the point in time that was chosen for the restore. The Url for the new site is typically the original Url with _Restored appended to the end, however if there are conflicts with that path the new Url will be communicated once it has completed.

Due to the nature of the side-by-side restore, there a few points to note:

  • The new site collection will be set in read-only mode. It is not intended to be a fully functional and utilized site, but rather a way to recover data that is otherwise unrecoverable.
  • This new site collection will be crawled and thus will return in search results and will be accessible by those with access. As such, this may result in the restored items being inadvertently updated. You should plan to remove the restored site as soon as possible.
  • If the restore fails for any reason, it will be immediately aborted. Once aborted, the only remaining option to restore the site collection to the same Url.

Microsoft will notify you once the restore completes. Upon completion of the restore, the new site should be reviewed and content backed up/moved to the target location so that the restored site can then be removed.

Once you are ready to proceed with the restore, reply-all to this email confirming that you understand and agree with the plan outlined. In addition, you understand the recommended steps that should be executed before and after the restore. Once you provide confirmation to move forward with the restore, Microsoft will start the restore process.

Subweb

The next course of action aimed at moving our support case forward is to restore a copy of the subweb located at <insert subweb URL>. The subweb that currently exists at the specified URL won't be deleted. Instead, a copy of the subweb from the provided point in time will be restored alongside the original. The data within that subweb will reflect the state of subweb from within 15 minutes of the established restore point (<Insert Restore Time (UTC)>). This type of restore typically completes within 24 hours, but can take as long as 72 hours to complete.

Due to the nature of the side-by-side restore, there a few points to note:
  • This type of restore isn't full fidelity. Workflows, Apps from the App Store, checked out items, and items in the recycle bin won't be restored.
  • The quota for the site collection will be increased to ensure there is enough space to accommodate the second copy of the subweb. This increase in quota will take from the overall tenant pool of available storage which may prevent the creation of new site collections until this sites quota is reduced at the end of the restore process.
  • This type or restore creates a duplicated copy of the subweb chosen for the restore. This duplicated data will return in search results and will be accessible by those with access, which may result in the restored items being inadvertently updated. Due to this, you should plan to remove the duplicated content as soon as possible.
  • If the first attempt of the subweb restore fails for any reason, it will be immediately aborted. Once aborted, the only remaining option to restore the site will be a full Site Collection rollback.
Microsoft will notify you once the restore completes. Upon completion of the restore, the duplicated content should be removed as soon as possible.

Once you are ready to proceed with the restore, reply-all to this email confirming that you understand information listed above. Once you provide confirmation to move forward with the restore, Microsoft will start the restore process.
Step 5: Execute the PITR


Follow these steps:

  1. Confirm that the pre-restore steps located here are completed, and then follow these steps to verify that the point in time for the restore is before the change(s) the customer needs undone. If you do not have access to either of these links, continue to step 2.
  2. Initiate the restore using the URL and verified point in time 
    1. As a Merlin user, execute the command Start-GridSPOSiteRestore for a site restore or Start-GridSPOWebRestore for a web/subsite restore. Information about execution of the script can be found here.
    2. If you do not have access to Merlin, follow your normal esclation path to engage engineering (via ViewPoint) to initiate the PITR.
  3. Monitor the restore. After you start the restore, monitor the Grid Job that was created. If the job fails or suspends, refer to the PITR Troubleshooting Guide located here.
Step 6: Complete post-PITR activities

 

At this point, the restore has completed and there may be some additional steps. In addition to the final cleanup steps, we also have to help the customer take action so that they can prevent the need for the restore in the future. Depending on the kind of restore that was completed, the final steps differ.

Site collection - to the same Url

Because the whole site collection was rolled back, there's little to do after the restore. If the customer backed up any content that was updated after the restore point in time, they'll have to add that content back to the site at this point.

Site collection - to a new Url

The new site collection will exist at a new URL. The duplicate site has the potential of causing confusion and should be addressed as soon as possible. The customer should take action to recover the intended content and restrict access or delete it as appropriate.

Subweb

The original subweb now exists on a new URL alongside the restore subweb. The duplicate site has the potential of causing confusion and should be addressed as soon as possible. The customer should take the needed action against the renamed subweb (depending on the reason for the restore) and then delete it as appropriate.

General Guidance

The process of restoring a site is time-consuming and causes a disturbance with the user base. Therefore, this is a great opportunity to help the customer avoid future restores. Based on the need for the restore, the guidance here will differ. A list of suggestions for common restore reasons will be maintained here.
