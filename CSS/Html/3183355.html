1. Types of URLs

There are three different types of URL in Microsoft Advertising:

  • Display URL: The URL that is displayed on an ad. This should be either the actual landing page, or a shortened version of the landing page to show the top-level domain. See Display URL - Definition for more information.
  • Final URL: The landing page URL the customer is taken to after selecting the ad copy.
  • Mobile URL: The landing page URL the customer is taken to after selecting an ad on a mobile device.

Note: When creating landing page URLs at the keyword level in the UI, Final URL and Mobile URL options are located together, when editing the Landing Page column.

We also have a legacy option, known as Destination URL. This is being ramped down and can no longer be selected when editing/creating ads but may be seen in older advertising accounts.

Advertisers are also able to set up URL tracking, to find out how search traffic arrived on their site and to analyze the data using third-party tools. See What is URL tracking in Microsoft Advertising for more information.

Note: Advertisers using URL tracking should also consult the help article How do I create an account tracking template.

2. Upgraded URLs (UURLs) 

Upgraded URLs give advertisers and agencies better tracking options. UURLs allow for tracking templates and custom parameters to be used.

The four new fields added with upgraded URLs are:

  • Final URL
  • Mobile URL
  • Tracking template (tracking URL + landing page URL)
  • Custom parameters (eight maximum)

Final URL validation

  • The length of the URL is limited to 2,048 characters.

Note: The HTTP or HTTPS protocol string does count towards the 2,048 character limit.

  • Usage of '{' and '}' is only allowed to delineate tags.
  • Final URLs must each be a well-formed URL starting with either http:// or https://.
  • If you specify FinalMobileUrls, you must also specify FinalUrls.
  • You may not specify FinalMobileUrls if the device preference is set to mobile.

See FAQ: Upgraded URLs for detailed information on this functionality.

2.2 Escaped-unescaped URLs

In computing and telecommunication, an escape character is a character which invokes an alternative interpretation on subsequent characters in a character sequence. Only alphanumeric characters (the special characters "$-_.+!*'(),") and reserved characters may be used unencoded within a URL.

Double-encoding - The way you get, for instance, %2520 is when your URL already has a %20 in it, and gets URL encoded again, which transforms the %20 (=space) to %2520.

Note: Use an online decoder (there are many available online) to understand the URL custom parameters.

A custom parameter = Any URL tag that begins with an underscore.

  • A tracking template is needed for custom parameters to run, as the tracking template allows the system to understand how the parameters need to be appended.
  • Can add any parameter that the advertiser might be interested in tracking (for example, a keyword ID generated by a third-party tool such as DoubleClick).

2.3 UURLs and editorial review

UURLs separate the landing page URL from the URL parameters so if you want to edit your URL parameters, your ad doesn't have to go through another editorial review. It also allows you to define a separate mobile landing page URL if you have a website that is optimized for smaller devices.

2.4 UURLs and auto tagging

Auto-tagging was originally implemented to allow the tracking of Microsoft Advertising text campaigns on analytical tools such as Google Analytics.

To enable the tracking, the most common ways employed by advertisers on Microsoft Advertising are now auto-tagging or using tracking templates via third-party tools such as intelliAd, Kenshoo, Double click Search, Adobe AdLens, or ad-hoc tools built via API's.

Auto-tagging or tracking parameters are making the data readable by external tools﻿﻿.

Data should then be displayed on Analytics as UTM-tags, which allows it to identify the source as Bing and any other parameter dynamically set (such as ad group name).

Auto-tagging is also supported for UURL; however, this might lead to redundant results which the advertiser might want to be aware of, or other issues. For example, a Final URL with hard-coded tags (UCM000000399810, UCM000000407278) http://www.liligo.com.au/?utm_source=bing&utm_medium=cpc&utm_campaign=B-L-AU-FL-Airline-SEA-PC-L&utm_term=all+nippon+airways+australia&k_clickid={ignore}_kenshoo_clickid.

UTM-tags dynamically generated (non-auto-tagging) on the tracking template might cause redundant results (although not delivery issues) – for example, lpurl?utm_campaign={Campaign} + auto-tagging which will generate the same tag.

2.5 UURL troubleshooting

  1. Spoof the ad and get R-link (right click on ad and select copy shortcut).
  2. Confirm that the R-link refers to the affected keyword/campaign by decrypting URL on AdInquiry or checking that the ad in the UI is matching with the one on the SERP.
  3. Decode URL (there’s an embedded URL decoder on Bing, just type “decode url” and search on Bing).
  4. Verify that all tags are firing properly, if not:
    • Check which portion of the UURL is failing (for example, auto-tagging or other dynamically generated parameters).
    • Go back to the UI to verify the setup (https://docs.microsoft.com/advertising/guides/url-tracking-upgraded-urls?redirectedfrom=MSDN&view=bingads-13).

Keep in mind:

  • Auto-tagging is set at account level (options: replace or keep existing tags).
  • Final URL is set at either ad or keyword level.
  • Tracking template/custom parameters are set at either account, campaign, ad group, ad, or keyword level. (For Shopping, it’s only available at product group level. Enabling the tracking template for Shopping at account level is in the Engineering pipeline).
  • Troubleshooting scenarios.

Scenario 1 – Advertiser can’t see tracking information in GA

  • Things to check for prior to escalating to Tier 2:
  • Is the tracking template set at the right level? If the tracking template that is needed for GA is set at campaign level and there is one other tracking template at ad level, then ad level tracking template will be used.
  • Do we have the R-link to the ad? This will be used to understand at what level is the tracking template setup at.
  • Are any unsupported parameters used in the tracking template? If the customers use a tracking parameter that is a valid one for Google Ad words (for example, {feeditem}), but this isn’t supported on our platform yet, so we will ignore the parameter.
  • In GA, do they have columns for the basic five tracking parameters, utm_source, utm_medium, utm_campaign, utm_content, utm_term.

Scenario 2 - UTM Tags not appending

Things to check:

  • How is the auto-tag setup ("ON and override" or "ON and Append") – can be validated from the Accounts & Billing screen.

[Asset 4046049]

  • Check website to know what parameters are used, utm_source, utm_medium, utm_campaign, utm_content, utm_term.

Scenario 3 - Incorrect Campaigns being attributed in the tracking system

  • Things to check:
  • Is the tracking template set at the right level?
  • Is there a hard-coded entry for UTM_Campaign Autotag listed in the tracking template?
  • Check UI campaign name and compare to the website URL to know if they display the same information.

Scenario 4 – Missing tracking info on TP platform such as DoubleClick (DS)

  • Things to check:
  • What element is missing to track? For example, if ad, check that the relevant parameter is passed, such as {AdId}.
  • Check Ad Inquiry to verify at what level the tracking template is set.
  • Verify that the ad or keyword URL are set to Final URL to have the tracking template enabled when serving on the SERP.
  • Decode R-link and verify that there is a redirection to the TP page (for example, typically clickserve.dartsearch.net for DS) along with required params if needed (for example, custom params {_lid} and {_ds_s_kwgid) are typically used for DS).
  • Verify that after the redirection, the advertiser’s landing page is correctly reached with the required parameters passed.

3. Display URL Expansions 

Display URL Expansions enhance ad placements by adding relevant content to an ad’s display URL and further qualifying clicks. This annotation shows users that there’s a match between their query and the content on an ad’s landing page.

The keywords used in Display URL Expansions are derived from the bided keyword for the ad or the user’s search query. They must also meet the following criteria:

  1. The keyword is already not present in the Display URL.
  2. The keyword is present in the Final URL (or redirected URL), but not part of the query string (or).
  3. The advertiser has not used parameter replacement on the keyword.

This feature increases user confidence in the relevancy of ads and is offered at no cost to the advertiser.

When applicable, this external annotation is enabled at run time, creating a more relevant consumer experience, and increasing ad response rates.

You’ll find a quick step by step explanation of how Display URL Expansions work in the sections below.

3.1 Things to know about Display URL Expansions 

  • Advertiser benefits: Initial simulation shows significant CTR gain when showing Display URL Expansions with search ads. Additional data is currently being gathered during the initial flight.
  • Competitive/Parity: It is not confirmed if Google also shows display URL annotations, or if the display URLs are still being dictated by the advertiser.
  • Devices: PC only. Tablet and mobile at a later date.
  • Search/Content Ads: Search only.
  • Mainline/Bottom of Page/Sidebar: Mainline only to start. Bottom of page or sidebar (in markets still supporting this feature) may be available later.
  • Impacted Users/Markets: Bing US/English only at this time. Yahoo releases plans to Microsoft Advertising.

Return to top

3.2 How Display URL Expansions work 

1. Bing retrieves ad landing page URL and searches for content relevant to user’s search query.

2. Bing finds phrases in the landing page URL or content on the landing page that matches the user query.

3. Bing confirms there are none of the following restrictions:

  • The keyword is already not present in the Display URL.
  • The keyword is present in the Final URL (or redirected URL), but not part of the query string (or).
  • Advertiser has not used parameter replacement on the keyword.

4. The text-only annotation appears in the ad’s display URL at run time. When other components of the ad are clicked, the user will go to the ad’s landing page URL.

For more information, see the Display URL Definition help page.

Parallel Tracking

Parallel Tracking allows advertisers to send customers directly to the final URL, without taking the customer through any tracking URLs. Tracking URLs will continue to work via a parallel click process that happens in the background, without affecting the user experience.

This reduces latency and improves the end-user experience.

Because Parallel Tracking does not pass customers through tracking URLs, defining the tracking parameters within the Final URL Suffix will ensure they are passed to the landing page and valuable tracking information is not lost.

Advertisers will need to apply a final URL suffix (and custom parameters, if relevant) to their URL, to ensure their clicks are being tracked correctly. This can currently be done at account, campaign, ad group, or keyword level.

For more information, see Parallel tracking gets customers to your landing page more quickly.

You can also share the Parallel Tracking Technical Resource guide with customers to help them with Parallel Tracking setup.

Note: Parallel Tracking is not supported in all browsers. For unsupported browsers, we will apply the standard sequential tracking logic. (For example, the Tracking Template will be applied to the standard URL and customers will be taken through this, rather than passed directly to the landing page first.)

5. URL mismatch policy 

Display URLs in a single ad group must use the same top-level URL domain. For example, your ad group can feature display URLs with multiple folders, such as Microsoft.com/Office and Microsoft.com/Skype; however, it may not feature multiple display URLs with different top-level domains, such as Microsoft.com and Contoso.com.

The display URL must match the landing page URL.

  • Disapproved example:

    Display URL:     www.contoso.com
    Landing Page URL:     www.contoso.us
  • Disapproved example:

    Display URL:     www.contoso.com
    Landing Page URL:     www.floral.com

Dynamic keyword insertions could trigger a mismatch. If you use keyword insertion, ensure that the display and landing page URLs match.

  • Disapproved example:

    Display URL:     www.contoso{keyword}.com
    Landing page URL:     www.contoso.com
    Keyword: woodwork
    (display URL will result in     www.contosowoodwork.com)

Acceptable mismatches

Limited to the following circumstances, non-matching display and landing page URLs are acceptable.

  • Tracking URLs with redirects. Final URLs with redirects not matching the display URLs are acceptable provided that the top-level domain of the display URL and the top-level domain of the landing page URL match.
  • Approved example:

    Display URL:     www.Contoso.com
    Final URL: contosotracking.com/...
    Landing Page URL:     www.Contoso.com
  • Descriptive deep links. Subdomains or folders can be used in the display URL for descriptive purposes that do not match the subdomain or folders of the landing page URL provided that the top level domain match, and that the subdomain or folder adequately describes the portion of your site users will land.

The display URL must be of valid format.

The display URL cannot be used in a misleading manner.

Display URL cannot be used as another line of text.

Approved example

www.contoso.com
contoso.com
http://floor.contoso.com

Display URL: www.contoso.gov
Landing page URL: www.contoso.gov

Display URL: www.contoso.com/woodplans
Landing page URL: www.contoso.com/woodplans

Disapproved example

http:// Buy floor here
www.contoso
floor@contoso.com

Display URL: www.contoso.gov
Landing page URL: www.xyz.info

Display URL: call us now
Landing page URL: www.contoso.com/woodplans
  • Hosting sites. You may be allowed to use a descriptive display URL that does not match the landing page URL for hosted domains. If you believe you have a legitimate use case fitting this category, you may appeal the rejection.
    • Approved example:

      Display URL:       www.Contoso.com
      Final URL: contoso2.hosted-domain.com
      Landing Page URL: contoso2.hosted-domain.com
  • Brand considerations. You may be allowed to use a display URL that does not match the landing page URL for brand consideration purposes. For example, a well-recognized product brand can be used as display URL to direct users to the generic sale portal of the product owner. If you believe you have a legitimate use case fitting in this category, you may appeal the rejection.
    • Approved example:

      Display URL: office.microsoft.com
      Final URL: products.office.com/?office...
      Landing Page URL: products.office.com/?office...

6. Display URL deduping/duplicate ads policy 

Microsoft Advertising maintains a quality user experience by ensuring users are only served one ad per display URL by deduping eligible ads (preventing delivery of duplicate display URLs).

The main objectives of this functionality are to:

  1. Deliver a high-quality user experience to maintain and grow a valuable audience for advertisers.
  2. Create and maintain a fair marketplace for all advertisers.

Because only one ad for a given display URL will be served per search engine results page (SERP), if two or more ads contain similar display URLs and are eligible to serve, only the highest performing ad will serve and all other ads will be de-duped.

Note: Deduping will occur across multiple accounts/customers. Affiliate advertisers may experience more impact if they are using the brand owner's display URL.

Examples:
A URL is comprised of a top-level domain and a root domain (in addition to other components). Together, they are sometimes referred to as the domain. Microsoft Advertising considers display URLs to be similar and will be deduped if both the root domain and top-level domain are the same between both ads.

Root domain

Example: The root domain for http://www.microsoft.com is “microsoft” and the root domain for http://www.contoso.gov is “contoso.”

Top-level domain

Example: The top-level domain for http://www.brand.com is “.com” and the top-level domain for http://www.contoso.gov is “.gov.”

 

The following display URLs would be considered similar to “www.brand.com” because they contain the same root domain and top-level domain and would cause an ad to be deduped:

  • www.brand.com/anything
  • www.anything.brand.com
  • http://anything.brand.com
  • brand.com

The following display URLs would be considered different from “www.brand.com” because either the root domain, top-level domain, or both are different. This would not cause an ad to be deduped:

  • www.brand.anything.com (Root Domain is different)
  • brand.anything.com (Root Domain is different)
  • www.brand.net (Top-Level Domain is different)
  • http://anything.com (Root Domain is different)
  • anything.net/brand (Top-Level Domain and Root Domain are different)

Deduping impact on brand owners and affiliates

The best-performing display URL could potentially come from a brand owner or one of its affiliates.

  • The best-performing ad will continue to be shown and no preference will be given to favor either the brand owner or affiliate.
  • Brand owners are responsible for managing their affiliate agreements, including trademark and display URL usage.
