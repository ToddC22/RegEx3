Overview

Azure Cost Management (ACM) helps Enterprises, Cloud Partners and other Azure customers effectively plan and control their cloud cost. ACM also helps analyze anomalies spent in the cloud as well as optimize cloud spending.

This TSG provides guidance on how to assist customers who contact support with questions/issues related to ACM.

Azure Cost Management is available for the following:

Change of Channel Partner (COCP)

Partners markup the prices of services and features they provide differently from one another. Therefore, Azure blocks the customer view of costs until the next annual term. This prevents indirect customers from seeing differences in price markups if they change partners.

Example: It appears to an indirect customer that their term starts April 1. If the COCP happened in January, we would have dropped their cost view Jan – Mar.

TAGS

  • Resource group tags are not supported and only supports resource tags.
  • Resource tags are applied as usage is emitted from each service and aren't available retroactively to historical usage. If they don't show up when the usage shows up, it's an issue with the service.
  • Each service is responsible for sending tags with usage. If it doesn't show up with the usage, the service didn't send it. How and when the service sends usage will impact what you see.
  • This is the list of Tag support for Azure resources in ARM https://docs.microsoft.com/en-us/azure/azure-resource-manager/tag-support.
  • Classic resources do not support Resource Group and/or Tags. See TSG 4456278.
  • Tags are case insensitive. This may cause tags entered with capital letters to be displayed in all lowercase in the Cost Analysis section of the portal. See Resource Naming Restrictions for more information on the format of tag and resource names.
  • Tags updated in the portal take 24 to 48 hours to be applied. Since the published data is for the previous day’s usage, these changes will not appear in the usage report for two days.
  • When grouping costs by a specific attribute, the top 10 cost contributors are shown from highest to lowest. If there are more than 10 cost contributors, the top nine are shown plus the Others group, which represents all remaining groups combined. 
  • When grouping by tags, an Untagged group appears for cost contributors that don't have a tag key applied. Advise the customer to switch to the table view and change granularity to None to see all values ranked from highest to lowest cost.

Note:

  • If a resource was moved from one resource group to other, the tag will not gather the billing information before the movement.
  • If a resource was added to a tag, the previous cost of this resource will not appear in the tag. The cost is effective only after the movement date.
  • If a tag was "recently" created, it will not show the billing information previously to its creation date.

Spot Instances: Workflow and User Experience  

A Spot Instance is a Modern Asset that lets customers take advantage of unused virtual machine capacity in Azure. It has Low Priority VM Anatomy and has no fixed pricing model. Spot Instances are available at up to an 85% discount compared to on-demand prices. Customers are billed for Spot Instances in USD and their pricing fluctuates on a monthly basis.

To create a Spot Instance, the Deploy as an Azure Spot Instance: Spot option must be enabled during the Create VM workflow.

Eviction Types     

  • Up to On-demand: evicts the VM when Azure needs capacity for on-demand workloads
  • Max Price: evicts the VM when the platform price is greater than the customer’s Maximum Price
  • Eviction Policy: Stop/Deallocate or Delete (gives a 30 second warning prior to deallocating the Spot Instances)

Pricing Models

  • On-Demand: customer pays for use on an hourly basis
  • Reserved Instance: customer makes a larger upfront payment to receive a discount.
  • Spot Instance: customer negotiates price with a Maximum or On-Demand cost for Spot.

Spot Instances support all payment types. The Set Maximum Price blade compares prices in nearby regions to help customers select the lowest-cost region.

Spot Instances in ACM   

  • Multi-Currency in ACM lets customers select their local currency to analyze costs.
  • To analyze the cost in ACM, customers must filter using Pricing Model: On-Demand, Reservation, or Spot.
  • Budgets are supported after applying the Pricing Model filter.
  • Alert and Export are supported.

Usage Details and Invoice Experience 

  • Meter ID: unique identifier of the resource
  • Part Number: attribute of a particular product as per the available price sheet on Azure resources
  • Market Price: Spot Instance price
  • Unit Price: negotiated price (Maximum Price quoted by customer)
  • Effective price: lowest of Market Price and Unit Price
  • Pmax: Maximum Price set by customer for the Spot Instance

The Maximum Price is charged if the Spot Instance exceeds the threshold and is not evicted. Invoices are grouped at the meter description and then the Service Family. 

Pay-As-You-Go Customer Experience

As of 3/4/2020, customers on the Pay As you Go Subscriptions (3P) who purchase Spot receive a separate invoice for their Spot Charges as Spot is a Modern product and will be rated and invoiced in the modern systems by calendar month.

These customers have the following experience in Azure Cost Management:

  • Charges accrued against Spot VMs appear under the Invoices (Marketplace and Spot) view in the date picker, not under Invoices (Azure), because charges against Spot VMs are invoiced by calendar month and not by billing period.
    [Asset 4550575]
  • Spot charges are also available available if any of the options under Relative dates or Calendar months are selected, since these options are not Invoice specific.
  • For supported currencies, Spot charges are displayed in the Local or Billing currency.
  • For unsupported currencies, Spot charges are only displayed in USD.
    Note: other Azure consumption charges are still displayed in Billing Currency.
  • The Usage Details download file is structured in the same way: Spot charges appear in the Marketplace services file.

Cost Allocation 

Cost Allocation is a feature that allows Billing Account Owners or Enrollment Administrators in charge of payments to redirect the costs of shared services back out to internal departments in their organization at the Subscription, Resource Group or tag level.

This feature is s is managed at the ARM level, so CST will not show any data around this. Should you receive a case related to this feature, share the public facing document with the customer. If the customer needs further assistance, leverage the ACM Swarming Channel: ACM Cost Allocation for additional guidance.

  • In order to create a budget, the customer needs to have access to the cost data (i.e. AO/DA views should be enables and markup should be published). See Asset Unavailable section in this TSG.

  • Resource groups with little or no usage are not shown in budget filters
  • Budget names are only unique within a scope
  • When an account / subscription / resource group is deleted the corresponding budget is not deleted automatically. The customer will need to delete it manually.
  • You can set a budget alert to any email account, including distribution lists, shared mail accounts and personal addresses.
  • In case subscription offer type changes (and, hence, historical data is not available) query and budgets will have data only from the day of transfer and budget alerts would be based on new data only
  • In case of EA transfer, budgets created on subscription and bellow scopes should not be affected. Budgets created on higher levels should be recreated by customer (for example, budgets on a billing account level).
  • The language of the enrollment COUNTRY impacts the language of alerts sent and currently we are tracking this as an enhancement request that we need to develop. We currently don't support changing this as it is not possible to select/Change the language of the budget email alerts. The system takes the language from the Locale set on the Azure account/EA account.
  • Budgets only evaluated in the billing currency
  • In case budget currency is not matching the chosen currency, they would be hidden from UI. In other words, if customer has budget in currency other than USD, those budgets would be hidden when he switches to “Only USD” view
  • Budget uses “Evaluated” spend which happens every 12 hours (updated on July 2020) while Cost Analysis calls queries directly to get current cost, hence there can be a cost discrepancy between budget evaluated cost and actual spend.
  • Common Schema Alert is not currently supported with actions groups. The customer will need to set up an action group without common schema in order to receive budget alerts.
  • Push Notifications on IOS. Azure Mobile does not currently support budget alert push notifications.   • Annual Budget would only send a notification once per year. If the Budget threshold is reduced, ACM will not send any notification until the next billing period.
  • ACM features will not work when Azure offer is upgraded to an offer not supported in ACM.
  • Email alert notifications are sent from azure-noreply@microsoft.com. Verify with the customer that the address is not blocked by spam filters before escalating.
  • In order to disable alerts to a specific recipient, the user need to do the following:
    1. Navigate to the Budget blade.

EA status and availablility in ACM

Extended – At the end of the agreement date, enrollments are opted into indefinite Extended Term by default. The enrollment status changes from Active to Extended. This enables EA customers to use Azure indefinitely at the end of their enterprise agreement. Usage is charged as overages and customers continue to be invoiced. Usage details need to be shown on the EA portal for these customers. If a customer opts out of the extended term once they reach their end date, the status becomes Expired.

Expired Status – Enrollments expire and all associated services are disabled between 30 and 90 days for EA customers who opt out of the Extended Term at their enterprise agreement end date. Services are not terminated immediately during expired terms; some customers may still incur charges unless it is manually disabled by Azure Enterprise Operations (EA Ops). Customers cannot be invoiced after their term expires; therefore, all overage invoices are cancelled. The EA portal does not display usage for expired terms.

Terminated – This status indicates that the agreement is cancelled on the Microsoft Service Level (MSL) side by ROC. However, the process does not contact EA Ops to also terminate the agreement in ASMS. These terminations are not completed, and backend usage is generated as subscriptions are still active. Since termination was submitted by the customer, ASMS cannot invoice them and usage after termination does not display in the EA portal.

Manually Terminated – Initiated by EA Ops. Enrollment is terminated and does not generate charges. All accounts and subscriptions are inactive when terminated. Accounts or subscriptions that remain active are signs of a system bug.
 

Data availability in ACM

  • For Active and Extended status continue to return data (past and present) to the customer.
  • For Expired, Terminated, Manually Terminated, and Transferred status return data for past dates. Queries for data after the end date will return an HTTP 400 “bad Request” error.

According to BizOps, if enrollment expires, no usage or invoice or bill is generated and the customer cannot see data past the expiration date of enrollment.

Forecast

Currently, linear model is used for forecast. Forecast takes a week to adjust the projections based on the changes made by the customer. The model is not reactive to immediate changes.

Additionally, the model also does not account for purchase and RI discounts separately, currently it considers usage, purchase as one cost. if you have one off purchase in the past and lower usage every subsequent days the model believes that the usage is on a downward trend and will go to 0 on some days, the model takes up to a month to adjust to this one time high usage/purchase before it returns to normal trend.

Forecast can be unavailable in case there is not enough historical data, or in case EA number changed recently.

Azure Advisor

EA customers get recommendations based on their negotiated on-demand rates

The savings numbers in advisor factors in the utilization pattern that the user had over last 30 days. If at any time the utilization of RI would have been less than 100%, then it reduces the saving%. In the RI purchase UI, the number is based on the assumption that utilization will be 100, we don’t consider the past trend.

RI recommendations data have cost savings in Customer pricing currency

Cost Management for Partners (Formerly CSP)

  • ACM is enabled for CSP for customers onboarded to MCA. This can be verified in CST. Customer-facing info is available on Get started with Azure Cost Management for partners.

  • ACM is not supported for CSP legacy offers MS-AZR-0145P, MS-AZR-USGOV-0145P, or MS-AZR-DE-0145P (see list of supported/unsupported offers in this TSG). Support for Legacy CSPs is in the roadmap.

  • For the rest of the CSP customers, ACM is available as a web app that allows tracking and management of Azure usage and costs. It is usable through the Partner Center API. Details are available on Azure cost management app for Azure CSP partners.

  • Only users with Global Admin and Admin Agent roles can manage and view costs for billing accounts, billing profiles, and customers directly in the partner's Azure tenant.

  • In general, CSP customers can access data related to usage, billing, and commerce from the Partner center portal associated with the CSP Reseller tenant. They cannot access billing information on the Customer tenant. Billing and account roles are managed separately from scopes used for resource management, which use RBAC.

  • When CSP customers are redirected to the Azure portal, they land on the Customer tenant which is for service management and not cost management. The customer must ‘Switch Directory’ on the Azure portal to CSP Reseller Tenant > Cost Management + Billing > Billing Account/Billing Scopes > [Select the MCA]. This will show the CSP Billing details under the Billing blade in ACM. This blade belongs to ModernGTM, so we must engage ACE for any assistance.

  • Management Groups are not supported for MCA (end customers and Partners)

  • Purchases in ACM are shown in Partner tenant and not available in Customer tenant

  • Azure Lighthouse currently supports posibility to see Azure Cost Management data for partners and from other subscriptions (that have been onboarded for delegated admin access), with a few caveats:
    • CSP accounts are not supported (Susbcriptions hsould be MCA).
    • Cost Analysis scope can only be set to one tenant. You can’t build a cost analysis graph showing two different subscriptions.
    • Access to billing data is not supported.
  • For end customer, partners should enable the cost-visibility policy (see 3 steps to enable cost management)
  • Enabling Cost Management for end customers involves 3 steps for the  provider (see TSG 4525789):

    1. Confirm acceptance of the MCA on behalf of the customer. Present the MCA to the CSP customers and, once they've agreed to it, confirm the customers’ official acceptance in the Partner Center or via the API/SDK.

    2. Transition the customers to Azure plan. The last step for the partner to view and manage costs in the Azure portal is to transition existing CSP offers to an Azure plan. They'll need to do this once for each reseller and direct customer.

    3. Enable ACM for the customers. In order for the customers to view and manage costs in ACM, they need to have access to view charges for their subscriptions. This can be enabled from the Azure portal for each customer. It shows them their cost based on pay-as-you-go prices and does not include any applicable discounts or custom prices.

Support Scenarios for Partner led (CSP)

  •  End customers have access only to a subscription scope. Supporting Management groups is in the roadmap
  • Partner should enable cost policy for every customer separately
  • The tenant name is not available in Cost Analysis filter in the customer domain. The domain name is available only for Invoiced data and is not available for open period.
  • Cloudyn and ACM are independent services. If a customer migrates to MCA and uses ACM, it has no effect on their access to Cloudyn. The customer can continue to use both, but the recommendation is to transition fully to ACM, as this is where new features will be added, and Cloudyn will eventually be retired.
  • Purchases are available in Partner tenant level only. End customers and resellers views for purchases are in the backlog.
  • Admin Agent can’t create Export for the customer. -Exports do not support dumping storage files across tenants, the subscription for storage account should be in the same tenant as the user creating the export. -Admin Agent accessing customer tenant as foreign principal, Exports are not currently supported for this case. Backlog item exists but no eta available -Workaround:
    1. Adding Partner user to the customer’s tenant. In order to create Export user should have non-foreign login for that tenant. This can be not practical as Partners can have hundreds of end customers
    2. Providing customers with instructions on setting up their own exports
    3. Creating exports within Partner tenant for each customer. After implementing mechanism to deliver the exported data to the customers
  • Cost discrepancy issues: - Costs for end customers is calculated on base/PAYG price of the meter. Any discounts or Partner Earned Credits (PEC) are not available in that view. - Partners can see Partner Earned Credits (PEC) and other global discounts for some meters (those are tier specific discounts and others). This means that partner can receive more than PEC discount from Market price for the meter. The cost that partner sees when logged to his tenant should be similar to Recon file. - There is a known issue with minor cost discrepancy between Cost Analysis and Partners Recon file due decimal numbers. It can be explained by truncation logic for Charges by invoicing team. In ACM there is no truncation for decimal digits

Note: On ACM, Partners who have started onboarding End customers to MCA can still view the costs of Marketplace purchases made by Legacy customers.

 

Offers Supported in ACM

ACM includes all usage and purchases, including reservations and third-party offerings for Enterprise Agreement (EA) accounts. Microsoft Customer Agreement (MCA) accounts and individual subscriptions with Pay-As-You-Go rates only include usage from Azure and Marketplace services. Support and other costs are not included. Costs are estimated until an invoice is generated and do not factor in credits.

  • Marketplace offers are supported in ACM, but Support plans are not.
  • Internal offers are supported
  • Visit the Supported Microsoft Azure offers webpage for more details.
  • ACM supports only Partner (former CSP) subscriptions who have onboarded their customers to the Modern Customer Agreement
  • AIO customers can see the data in ACM from Oct 2, 2018 and onwards. ACM found a discrepancy between the work around provided and the info displayed within ACM. If you receive a case regarding this matter, share the below statement to the customer:

We apologize for any inconveniences that may have arisen as a result of this discrepancy. The newly released data solution is more accurate than the workaround and will allow you to take full advantage of Azure Cost Management features, including Cost Analysis, Budgets and Exports.

Engagement and Escalation Path

ACM cases are part of ASMS scope. Every ACM engagement must be vetted by a Technical Advisor (TA) or for the suppliers by one of the supplier Team Leads (TLs). Once vetted, your TA/TL will reach out to the ACM aligned TAs for assistance via the Azure Cost Management Support Teams channel.  

  1. Check that the issue is related to ACM and not the Billing blade (see Overview section of this TSG for further details).
  2. Collect the following details to provide to your TA/PTA
    • Issue description
    • Time when customer experience the issue
    • Subscription GUID
    • EA number (if applicable)
    • Offer type
    • User ID experiencing the issue
    • Screenshots
    • Network trace file (for dashboard issues please ask the customer to download dashboard as JSON file)
    • Export name and scope for export issues / Budget email alert for budget alert issues / ARM template for budgets created via ARM cases / Detailed usage file in case of discrepancy / other documents that can be relevant
    • TSG 4474330 followed? Yes / No. If No - why?
  3.  Once you have collected all of the customer details, do the following:
    1. Engage your TA/TL and provide them with a brief description of the customer's request, along with the details collected from above.
    2. Your TA/TL will review the request and assist you accordingly.
  • If your TA/TL needs additional guidance they will then reach out to the ACM Support Teams channel.
  • The following is a list of additional swarming team channels. Please make sure you post in the correct channel as PMs monitor relevant channels and assist with questions.
  • ACM Budget and Alerts
  • Data Quality (Accuracy, Reliability, Performance
  • ACM Scheduled Reports and Export
  • ACM X-Cloud
  • ACM Showback
  • ACM Forecast
  • Azure Pricing Calculator and Pricesheets API
  • ACM Power BI
  • ACM Advisor Recommendations
  • ACM for CSP and Partners
  • Cost Analysis
  • Once next steps are provided, your TA/TL will then update the Service Desk Task accordingly.

Note: Should a collaboration task need to be submited, only a TA/TL should submit such engagement to the ACM aligned TAs.
