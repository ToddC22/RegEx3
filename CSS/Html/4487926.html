1. Overview blade

The Overview tab captures the new customer’s account to organization hierarchy. Here’s an overview of the tab:

  • CRID – Unique Customer Root ID
  • Billing Account ID - Unique identifier that associates the Billing Account
  • Billing Account Name – Name Customer has identified for the Billing Account
  • Tenant ID – Unique ID that is tied to the tenant.
  • Billing Profile ID- Unique ID that associates the account to the Billing Group
  • Billing Profile Name – Name the customer/partner created for the Billing Profile
  • Cloud Instance - (True; False) Outlines weather or not the account is a cloud instance
  • Is JEDI – (True; False) Outlines weather or not the account is a JEDI account.
  • Is Match? - This feature will highlight the Organizations that Match the search criteria you used in the Search Customer Blade. Is Match? Applies to Invoice ID or Asset ID Only. 
    • For example, if you searched by Asset ID, the Is Match Field will mark the ones that have that Asset ID ties to the Organization.

4. Billing Accounts (LE) Blade

What is a Billing Account?

  • Account is the MCAPI root object that represents the scope of commerce interactions.​ It is a container for instances of the other platform objects.
  • Allows customers to relate their AAD tenant(s) with a single Account.​
  • It helps organizations to define their commercial scope, independent of the AAD scope.

This section displays all of the Customer’s Billing Account Details that are associated to the selected Billing Profile from the Overview Blade. 

Customer Details

  • Customer Account ID – Unique ID that is generated when the Customer Account is created
  • Segments – Indicates which Segment the Customer is in (Commercial, Government, etc.)
  • Cloud Instance – (True/False) Indicates if the customer is a cloud instance

Billing Account Details (Already defined in this document)

[Asset 4576675]

Agreements

In this section you will see the Customer’s new Modern Agreement, which is called the MCA.  If you are retrieving Partner data, you would then see the Partner’s new Modern Agreement, which is called the MPA.

MCA (Microsoft Customer Agreement).  One Microsoft Agreement that all Customer’s sign. This makes the licensing transaction process easier, more efficient and simpler for our customers. The MCA allows new and renewing customers to sign up for Azure through a seller using Quote Center.

MPA (Microsoft Partner Agreement).  One Microsoft Agreement that all Partner’s sign. This also makes the transaction process easier, more efficient and simpler for our partners.

In the example below, you are seeing a Customer’s New Modern Agreement data.

  • Agreement Number – Unique identifier that ties the agreement to the customer (hidden for confidentiality)
  • Status – (Active; Inactive) States whether or not the agreement is active
  • Start Effective Date – The date the agreement is active
  • End Effective Date – The date the agreement expires
  • Acceptance Mode – How the customer accepted and signed the agreement (Click to Accept)
  • Document URL – Link to a physical copy of the MCA

[Asset 4576676]

Policies

  • Tenant ID – Displays all of the Tenant ID’s associated to this Billing Account
  • Policy Name – Name of the Policy (ex. AllowSelfServicePurchase)

  • Product ID- Unique ID associated to the Product
  • Product Name – Name of Product the Policy is associated with
  • Status – Enabled; Disabled

RI – Transfer To Modern (GTM)

The following steps walk you through how to transfer RI to a Modern GTM Customer.  PLEASE NOTE – The process for RI Transfer to Modern has not changed.

  1. Log into CST; Navigate to “Assets + Subscriptions” Blade and Select the Reservations Tab
  2. Select the Active Reservation you would like to transfer (Example shown below)[Asset 4576708]
  3.  Select the “Manage RI” Tab from the left-hand pane and then “RI Transfer to Modern (GTM)”[Asset 4576709]
  4. Enter the following details to ensure a successful transfer:
    1. Case ID – Free form field to track the transfer
    2. Transfer From
      1. Email address that is associated with the RI you would like to transfer
      2. Tenant – Tenant that is associated to the Email address and RI
    3. Transfer To
      1. Email- Email Address that you would like to transfer the RI to.
      2. Tenant – Select the Tenant that is associated with Email Address that you are transferring the RI to
      3. Destination Project – Select the Destination Project that you are wanting to Transfer the RI to
      4. Destination Rating Asset – Rating Asset that is associated to the Destination Project
      5. Enter Comments
      6. Select Next
  5. Confirm all required fields are accurate before submitting RI Transfer.[Asset 4576710]

Upgrading Developer Support Plan (Upgrade Asset)

An agent has the ability to upgrade a Developer Support Plan from Standard to Pro Direct OR Downgrade from Pro Direct to Standard in CST.  If this action is performed, the customer will be charged to the pro-rated amount from the date the upgrade OR downgrade is effective.  To perform this action, please follow these steps:

  1. Log into CST; Search for Customer’s Billing Account
  2. Navigate to “Assets + Subscriptions” Blade; then “Plans + Others” Tab
  3. Select the Developer Support Plan you would like to perform either action (shown below)[Asset 4576730]
  4. Select “Upgrade Asset” from the action bar[Asset 4576731]
  5. Ensure the following fields are accurately filled out
    1. Case ID – To track the action
    2. Following Asset Will Be Converted (upgrade/downgrade) - Verify correct
    3. Select the Plan to Convert – Upgrade to Pro Direct OR Downgrade to Standard
    4. Comments – Required Field
  6. Select Next; Confirm to ensure correct Select Submit[Asset 4576732]

11. Lockbox blade

UBS signed an agreement with Microsoft that contains a special clause requiring customer’s approval for change of subscription ownership. Microsoft will obtain UBS approval before performing the transaction. The agreement impacts Subscription Ownership Transfer (SOT) feature in CST and Commerce Management Agent Tool (CMAT). Both CST and CMAT need to implement the lockbox feature for UBS and other customers that require preapproval.

By implementing this feature, Lockbox blade will retain UBS as a Customer and Reduce customer pain time (CPT), improving customer experience (CSAT).

Supported channels

  • EA
  • Web Direct
  • Field Led

Even though the Lockbox requirement is for UBS only, there are future deals in the pipeline that will also leverage this feature. This blade will display the customer’s Lockbox information as follows:

Approval process:

  1. Through agreement with the customer, an approver or approvers are designated and assigned in CST.
  2. Once a SOT is performed, an email from Microsoft is sent to the approver.
  3. The approver has five days to approve the SOT transfer:
    1. If approved, no action from agent is needed, auto complete.
    2. If not approved a new request is sent from Microsoft after five days.
