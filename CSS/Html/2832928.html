ASMS – Azure Active Directory (AAD)

Submit a ModTrack request to update this TSG.   

Overview  

This TSG provides guidance on how to assist customers with their Azure Active Directory (AAD) subscriptions and services.  

AAD is a REST-based service that provides identity management and access control capabilities. Customers can have one identity service across Azure, Microsoft Office 365 (O365), Dynamics CRM Online, Intune and third party cloud services. Azure Active Directory provides full support of third party identity providers and allows customers to easily integrate their on-premises AD deployments. 

Policy  

Use the guidelines below to determine what actions should be taken based on the scope of the scenario. 

  • If you receive an issue that is outside of ASMS scope, escalate to your TA who will work with the aligned TA's and internal teams to resolve the issue. If it is determined that the issue should be addressed by the AAD team, you can transfer the entire case to them. 

  • If you receive a request in regard to licensing or tenant issues (disabled, hit a limit, getting reports, enablement of AAD Premium, credential rejections, etc..), escalate to your TA for next steps as they may need to engage the aligned internal team.  

  • If you receive a request in regard to resending emails to get the Azure Active Directory activated, escalate to your TA so they can engage the internal team to assist. ASMS does not have the ability to resend these emails. 

  • Always engage ASE first when you are asked to engage ACE (unless there is an emergency).

  • Do not use the ASE template as a way to open ICM tickets to any other team. ASE is not the same as ICM.

Change Subscription Directory in the Azure Portal

Customers can edit their directory via the Ibiza portal, using the Change Directory button. More information about this feature can be found in the following links:

  • How to: Associate or add an Azure subscription to Azure Active Directory
  • Change the subscription directory in the Azure portal 

Edit directory changes for customers can be seen in the KpiMetric MDS table for the last 30 days. The table provides:

  • Time
  • Subscription ID
  • User that initiated the directory change
    • OID
    • TID
    • PUID
    • Session

Important Notes:

  • Only users who have the RBAC role of “Owner” can edit the directory.
  • The user editing the directory must have the appropriate access to the destination directory before making the change.
  • Changing a directory causes all RBAC roles to be removed, as the user who makes the directory change becomes the new Service Administrator.
  • Internal subscriptions cannot perform a Change Directory.
  • Change Directory will remove all RBAC users.
  • Change Directory will effect KeyVault and user would have to change the Key Vault tenant ID before resuming operations for KeyVault.
  • Resource Locks are removed when a Change Directory is performed. Locks and role assignments are examples of authorization resources that do not get migrated when tenants change, because they're partitioned on tenant ID.
     

Escalation path: Most RBAC issues should be worked by the Identity team. Follow the engagement process in TSG 4025433.

Change Directory of a subscription 

ASMS  

Any email ID with OWNER role can change directory of an Azure subscription. It used to be available only to Service Admins with Microsoft accounts, but now it's available to users with AAD accounts as well.

In some scenarios, customer would not be able to access the subscription after performing change directory operation. This is due to delay in synchronization of tenant ID in ARM.

To confirm the tenant ID in RDFE: AzureRT > Subscription management > Get subscription tenant ID.

To confirm the tenant ID in ARM: Azure Resource Manager > Subscription State Management > Get Subscription.

If the above operation result in different tenant ID and the tenant ID in RDFE is the desired one after change directory, then perform the following operation : Azure Resource Manager > Subscription State Management > Synchronize subscription from RDFE.

When the directory of a subscription is changed, it will affect the users, services, applications and resources associated to the current directory of a subscription. Please make sure that the customer is educated before making any changes.

What are the types of accounts available in Azure?  

There are two types of account authentication available. Namely a Microsoft account and a Work account.

[Asset 4078461]

Work accounts are created by an organization’s administrator to enable a member of the organization access to all Microsoft cloud services such as Microsoft Azure, Microsoft Intune or Office 365. A work account can take the form of a user’s work email address, such as username@orgname.com, when an organization federates or synchronizes its Active Directory accounts with Azure Active Directory.  

 Microsoft accounts are created by user for personal use, is the new name for what was previously referred to as “Windows Live ID”. The Microsoft account is the combination of an email address and a password used to sign in to all consumer-oriented Microsoft products and cloud services such as Outlook (Hotmail), Messenger, OneDrive, MSN, Windows Phone or Xbox LIVE. If a an email address and password are used to sign in to these or other services, then the user already has a Microsoft account—but the user can also sign up for a new one at any time.

What is the relationship between Work accounts and O365 subscriptions in Azure: 

Office 365 and Azure use the Azure AD service to manage users and subscriptions. The Azure directory is like a container in which you can group users and subscriptions. To use the same user accounts for your Azure and Office 365 subscriptions, you need to make sure that the Azure subscriptions are created in the same directory as the Office 365 subscriptions. Keep in mind the following points: 

  •  A subscription is created under an account associated to a directory. 
  • Users may belong to multiple directories.
  • A subscription lands in the default directory of the user who creates the subscription, tying the Office 365 subscription to the same account as the Azure subscription.
  • Azure subscriptions are owned by individual users in the directory.
  • Office 365 subscriptions are owned by the directory itself. Users with the right permissions within the directory can manage these subscriptions.
