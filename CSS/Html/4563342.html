ASMS – EAO Engagement Path

Submit a ModTrack request to update this TSG.

Overview

This TSG provides guidance on how and when to engage the Enterprise Azure Operations (EAO) team.
Details about which Support Scenarios are owned by ASMS and which are owned by EAO can be found in TSG 2872061.

Policy

SLA

The following SLAs have been put in place for when ASMS engages the EAO team. This applies to cases of all severities (A, B, C).

Note: There are a few exceptions to these SLAs, particularly in instances where the case/task is handled by a specific region in which the initial response SLA is 1 business day. These exceptions include:

  • Credit Issues (US Only) 
  • RI Cases (US only) 
  • Language Cases (Singapore Only)

Case Ownership

Refer to TSG 2872061 for a list of scenarios owned by ASMS and a list of scenarios owned by EAO.
  • If the case is in scope for ASMS but requires EAO assistance as mentioned in the respective TSG, send a task to the EAO team.
  • If the case is related to any of the scenarios as part of the EA Scope Transition project, engage the ASMS EA ringfenced team via a task.  For a list of scenarios see TSG 4553068. 
    • MSaaS ASMS EAL MCA (Global English and non-APJ LL; includes USG)
    • MSaaS ASMS EAL MCA - APJ Local Language (Mandarin, Korean, Japanese for Wave 1 Scenarios only)
  • If the case is submitted by an Internal Microsoft Employee and it reaches the ASMS EAL MCA queue, transfer the case to the MSaaS Azure EA Help - Internal Audiences queue for the EA Portal team to handle the case, irrespective of the scenario/scope.
  • If the case is out of scope for ASMS, transfer the entire case to the EAO team.
    • MSaaS Azure EA Portal Support
    • MSaaS Azure Government EA
  • If you are unsure who should own the case, engage your TA/PTA.

Note: Cases that need to be transferred to another team should include case notes on why the case is being transferred.

Queues

Cases will be handled in the following queues:  

  • MSaaS ASMS EAL MCA (Global English and non-APJ LL; includes USG)
  • MSaaS ASMS EAL MCA - APJ Local Language (Mandarin, Korean, Japanese)
 

Escalation Paths

Standard Escalations

A standard escalation can be initiated when SLA has not been met (see SLA table above ) and there is an immediate need for follow-up for one of the following reasons:

  • Initial Response missing
  • No status updates
  • Case/task closed prior to resolution confirmation from customer/task owner

In these situations you should use the teams channel ASMS & Enterprise Azure Ops to escalate the issue. When doing so, keep in mind the following protocols:

  • If the case has an owner, include an @ to the Vendor Manager and the EA/ASMS team POC.
  • If the case is new and needs immediate attention, check the region of the case and @ the region’s EA/ASMS team POC.
  • If the escalation is related to ASMS credit TFS @ Louie Torelli.

If you do not receive a response within 24 hours of posting in the Teams channel, reach out to the respective POCs listed below (per regional availability ).

Critical Escalations

A critical escalation can be initiated when CPE or a customer’s production is being impacted.

  • Examples include: access issues, system errors, production down, outage, etc.

You can leverage the Cloud and AI Customer Portal to see a list of “High Priority” customers to help prioritize your case.

To have a case/task prioritized, email the aligned POC listed below (per regional availability). During the weekends, leverage the On-call TA list in TSG 4313497.

Email communications should include the following information:

  • Subject line: Critical Escalation (Case Number)
  • Summary: Brief description of issue, Enrollment number, action required (i.e. Resolution/To be on Call/ Status Update, case should be handed over to next region).
    • Be sure to provide a clear customer impact statement. This should include how the issue is impacting the customer, who is escalating the case, and the severity of the case (A, B, C).
    • Mention the TA On-call if required for more clarification.
