What is this article about?

This article covers Service Request (SR) creation guidelines: When to create a new SR or edit an existing SR, how to work with an SR the Advanced Capability Advocates (ACA) has been engaged with, and what to do when a customer cites multiple issues on the same call.

1. Overview

Note:

As a rule, you should record every customer interaction except for in a few scenarios. Recording the contact may mean creating a new SR, editing an existing SR, or adding case notes, depending on the scenario. The exceptions to this rule are described below.

 

If a customer has multiple issues on the same call and the issues are very different from each other, multiple SRs should be created during a single interaction.

If the customer is contacting support again about an existing issue, the goal is to continue working from an existing SR, if one is available. Advocates who receive the return contact from the customer should always take ownership of the existing SR. SR status and age are not factors when editing a previous SR. Reviewing previous SRs should be done during each customer interaction to determine whether the customer previously contacted us.

Customer contacts must be tracked for all customer interactions, including outbound call-backs, successful or not.

2. Contact definition

  • A contact is created when:
    • An advocate creates a new contact in the SR to track an inbound call and selects Complete.
    • An advocate creates a new contact in the SR to track an outbound call and selects Complete.
    • An advocate creates a new contact in the SR to track a chat and selects Complete.
    • An advocate sends an email to a customer after selecting Complete.
    • An advocate edits an SR and adds case notes after selecting Complete.
    • A customer replies to an email sent from an advocate.
  • A contact is not created when:
    • An advocate edits an SR prior to selecting Complete (for example, when adding additional notes or changing taxonomy).
    • An advocate saves an SR periodically prior to selecting Complete.
    • An advocate sends an email to a customer prior to selecting Complete.

Note: When you edit or take ownership of an SR, enter detailed notes explaining any actions you have taken. Do not use any symbols to bypass a comment box or required fields

3. Preferred email guidance

If a customer provides an email address other than the Microsoft account email, remind them that they can add that as a contact email address at https://account.microsoft.com/billing/addresses -> Billing email, and offer to add it to their billing profile today to make sure they receive notifications from Microsoft going forward (Update via MST -> Update Profile).

Note: This field should only have one email address, not multiple.

4. Scenarios in which an SR will not be created

  • Misrouted customers that can be identified quickly.
    • If you need to do significant troubleshooting to determine if the customer needs a transfer, the customer was not misrouted, and an SR should be created.
    • Support for a non-Microsoft product.
    • Support for a Microsoft product that you do not support (for example, Surface or Microsoft 365).
    • The issue is clearly outside your support boundaries (for example, console failure received by an Accounts and Billing advocate).
  • Customer doesn't want to continue with the call and hasn't provided any contact details.
    • Failure to provide details may be intentional (does not want to disclose) or unintentional (does not know).
    • Failure to provide details may also be caused by line issues (for example, audio quality, ghost calls, dead air, or quick disconnects).
  • If a customer is asking why their account was suspended/banned or their gamertag was changed by the Xbox Enforcement team (Chat advocates should still associate, as customer will still receive a survey).
  • Prank calls.
  • Tools fail during call and that prevents SR creation (MST/ASD failure does not mean no SR).
  • RRT calls in which the customer is in the middle of an emergency scenario (for example, untreated injury or active threat to personal safety).
  • Ad hoc scenarios set and advised by Microsoft Service Delivery/CSS Command Center (CSS CC).
  • If a customer is asking for a refund and they have yet to complete a refund request via xbox.com/refunds (this excludes subscription scenarios, see KB 4338529 Guideline: Microsoft Store: Exception based cancel and return for recurring subscriptions).
    • Note: This will only apply to locales where Refund Webform is live. These locales can be found in the Overview section in KB 4485105 Procedure: Xbox: Refund Webform Process.
    • If a customer needs to be redirected to our partners in Russia for assistance, due to the current changes in the Russian market, as follows: changing currency from RUB to USD for Xbox One and 360 Games and CSV, Stop Sell for Xbox One Subscriptions and (legacy) Xbox 360 Subscriptions. Full details can be found in the XCC post https://microsoft.sharepoint.com/teams/XboxCommunicationsCenter/SitePages/Updated--CSS-FAQ---Russia-Changes.aspx.

5. Scenarios for which an existing case will be edited

  • Customers calling during an active MS ACA engagement. You should Edit Comments Only on the existing child SR. You will only be able to add comments. This is by design, as the ownership of the child SR remains with the advocate engaging MS ACA.
  • Customers calling again about a pre-existing issue. Evaluate the issue carefully to determine if it is the same issue. Questions about different transactions would be considered different issues, even if they are similar questions. Questions about the same transaction would be the same issue even if the questions are weeks apart. If the SR was created by a site that isn't your own, see section 8.
  • When you create a service request, never include high business impacting information in the service request notes. See KB 3064766 Procedure: Privacy: Handling Microsoft customer personal data.

6. Scenarios in which an existing case will NOT be edited

  • Customers calling about a closed SR that was engaged with MS ACA. You cannot re-engage MS ACA using the closed service request. You must create a new child service request and follow the engagement process again.
  • Customer calls about a new issue or an issue that's not the same as a previous SR.

7. What to do when a customer has multiple issues during a call/chat

  • Gather all the customer's issues.
  • If the issues are related, create only one SR and log the primary pain point for the customer. Examples:
    • One issue created the other.
    • Both have the same root cause.
  • If the issues are unrelated, create a different SR for each issue. Examples:
    • The issues deal with completely different functions with the console.
    • The issues have completely different root cause.

8. How to take ownership of a service request

If you need to edit a previous SR, do the following:

  • Check which queue the SR is currently set to.
  • If the queue is one of your sites already, select Edit and work the case as normal. If the editing comments only warning message appears, cancel out of the case and use the Take Ownership button.
  • If the issue is owned by a site that isn't yours, select Take Ownership and work the case as normal. If the Take Ownership button does not appear, check the queue again to verify that the case is not currently owned by MS ACA. If it isn't owned by MS ACA, contact your supervisor to report the problem and create a new SR to assist the customer.
  • If you need to take ownership of an SR, review any open follow-ups to make sure that they are either cleared out or updated based on the next action needed.
