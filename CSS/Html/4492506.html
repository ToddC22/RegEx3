Overview

This TSG provides information about assets and how they behave in Modern Commerce.

An asset represents a footprint of ownership and the management of their associated child assets, typically acquired through purchase. It represents the result of a transacted purchase line item. ​ The diagram below illustrates assets and how they are related to the rest of the Modern Commerce API (MCAPI) resources:

Asset Overview

[Asset 4492500]

  • Every asset has a single catalog BigID.
  • ​Every asset is always associated with one project or invoice section as its authorization container.
    • ​When purchased, the recipient project is the project first associated with this asset.
  • ​An asset may have one or many behaviors. An asset is a composite resource that is represented by a single base asset and multiple behaviors associated with it.
    • Behaviors are identified in the BigCat asset instruction section.
    • Each behavior contains a set of closely related attributes that define a unique commercial aspect of an asset.
    • Behaviors include:
      • Ratable
        • Indicates that this asset is a blanket container of which service usages are rated together. This is typically used in a consumption scenario where quantities of usages are unknown ahead of time.
          Example: Azure Access Pass asset has this behavior.
        • Assets with this behavior are frequently called Rating Asset Direct (RAD) or Rating Asset Linked (RAL).
        • Access Passes will be available first to field-led customers on Field Led Azure Modern (FLAM). (WD), Cloud Solution Provider (CSP), and Enterprise Agreement (EA) customers will have an Access Pass created behind the scenes for them for a normalized system architecture. Customers will not be exposed to this concept and will continue using Azure Subscriptions as their billing scope.
      • Quantifiable
        • Indicates that the asset has a known and limited quantity of licenses, products, or benefits​ chosen by a customer at the time of purchase.
          Example: Per-seat-based Software-as-a-Service (SaaS) models like Office 365, Minecraft, and Azure Reservations have this behavior.
        • Quantity helps define the legally binding license count a customer can choose. A software publisher might choose to do a hard enforcement of their quantity (don't issue more if the quantity cap is hit), or soft enforcement of their quantity (true-up for an over-used portion). The former is currently being supported in the system, while the latter is in the roadmap to build.
        • Quantity can accumulate through multiple purchases.
      • Term-Based (Renewable)
        • ​Indicates whether the asset has a limited duration for its license, benefit, or product.
          Example: Azure Reservations have this behavior because they can be a 1-year or 3-year offer.
        • Some products allow customers to turn on auto-renew to extend the term, hence this is the attribute where "auto-renew" can be toggled on or off. The IsRenewable flag configured in the catalog tells the system whether the product can be renewed or not.
          Example: Office 365 would allow AutoRenew, while Azure Reserved Instance would not.
      • Future Billing
        • Stores a set of settings (such as BillingGroup) that systems can use to charge a customer's renewal fee or consumption overage fee.
          Example: Access Pass (Azure Access Pass) and SaaS subscription offers will have Future Billing behavior on it.
      • Usage
        • This indicates this asset has a 1-to-1 mapping to an Azure Subscription. This is a behavior only used for connecting Azure consumption with the Rating asset.

Asset Usage Accrual

Usage assets connect Access Passes (commerce) with AzureSubs (deployment or management).

What is Azure Access Pass? ​

  • Some parts of it are very similar to legacy AzureSubs:
    • The customer gets rated and billed under each Access Pass.
    • The customer can choose what service they want to use based on their elastic needs.
    • The customer only pays after usages. (They can pre-buy benefits like Reserved Instance.)
  • However, it offers richer functionality than AzureSubs:
    • AzureSubs can be kept as a deployment and management container.
    • Multiple AzureSubs can be billed against a single Access Pass.
    • Usages can be accrued cross AzureSubs for better pricing benefits (like graduated pricing).

There can be multiple types of Access Pass, such as Dev/Test, Standard, and Fairfax. Typically, each Billing Account has one Dev/Test and one Standard Access Pass.

Access Pass is only visible to Partner Led CSPs.

How Does Usage Get Accrued to an Access Pass? ​

Each AzureSub will have a corresponding usage asset created. Each usage asset remembers the AzureSubscription ID it is associated with. When usages flow into the MCAPI system, it then knows which usage asset gets affected and hence the Access Pass to use for rating. ​

​A single Access Pass can have one or more usage assets from the platform perspective. An Azure Subscription (Usage Asset) is only related to one Azure Access Pass.

Asset Life Cycle

An asset has its own life cycle driven by many actors in the system.

  • Purchase
    • Creates a new asset.
  • Recurrence Processor
    • Auto expires term-based assets at their end-of-life cycle.
    • Renew assets where auto-renew is turned on.
  • Dunning
    • Mark asset as delinquent and activated as part of the cash collection process. Service provider decides actions to take against these signals.
  • Cancel and Return
    Note: Can be driven by Customer Service and Support (CSS), Operations, or the customer.
    • Deletes an asset.
  • Asset Self-cleanup
    • Abandon an asset that shouldn’t be created – system error, rollback, etc.

Asset Delegation​

Asset Management Capability Can Be Delegated, but the Bill Is Still Aggregated

Asset owners can choose to delegate other parts of the organization to manage their assets, but the owner still pays for all bills associated with the assets. This action is called delegation. Delegation results in an additional asset that is linked with the original asset, to represent this relationship. The original asset is usually referred to as the direct asset, and the delegated asset is usually referred to as the delegated asset or the linked asset.

Entitlement Asset Delegation

It is usually easier to start with an entitlement asset. These are assets that the user knows the quantity to buy ahead of time, in contrast to the usage model used for Azure Pay-As-You-Go offerings.

The following diagram illustrates an example where 1,000 Office 365 assets (seats) get purchased, and 200 of those assets get delegated to User 2 for Project 2. As a result, all charges for the 1,000 seats will be charged against Asset 1, with the billing group specified on Asset 1, but user 2 can manage their delegated 200 seats.

Asset Transfer

The Asset Transfer operation gives customers the ability to:

  • Transfer the right to manage a product​.
  • Transfer the financial responsibility to own a product​.
  • Transfer a partial quantity of a purchased product.
    • Coming soon. This is not in scope for the 2/15/2019 release.

The following table outlines what types of asset transfers are possible.

Account Type Subscription Type Transfer to Legacy Account Transfer to MCA
Modern Modern Not allowed N/A
Modern Legacy Allowed via CMAT only. Allowed via CST only.
Legacy Legacy Allowed via CMAT only. Allowed via CST only.

Rating and Consumption Asset Transfer Policy

  • The user initiating the usage asset transfer must have access rights in the source and destination projects. ​
  • The user initiating the usage asset transfer doesn’t need to have access rights in the destination billing group.

Entitlement Assets Transfer

  • The user initiating the entitlement asset transfer must have access rights in the source and destination projects. ​
  • The user initiating the entitlement asset transfer doesn’t need to have access rights in the destination billing group.

Client Obligation​

  • “Transfer_To” entity exists before transition:
    • When there is a transfer to FLAM, the legacy system (namely, Outsourcing Management System (OMS)) will take care of that portion of the operation. ​
    • When performing a usage asset transfer​, the system confirms that the requestor has the proper access in both the source and target.
      • This confirmation is handled by the Azure Commerce Engineering (ACE) team’s Associated Asset Transfer Service. ​

Transfers will be blocked if:​

  • It is rejected by transfer eligibility API. ​
  • There are associated asset transfer initiations.
  • The transfers are backdated. ​
  • The transfers are of partial quantities. ​
  • The transfers are cross-cloud in scope.
     

​​​​Access Pass Delegation

Access Pass delegation is very similar. However, instead of delegating a fixed set of entitlements, the owner allows other users to set up Azure Subscriptions and let usages be pumped and billed against an original Access Pass asset:

  • ​RAD (Rating Asset Direct) is similar to Access Pass Direct. The Access Pass was directly purchased and has a billing group with it.
  • RAL (Rating Asset Linked) is similar to Access Pass Linked or Delegated. The Access Pass was created by delegation steps.
