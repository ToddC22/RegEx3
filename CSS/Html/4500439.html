Summary

Always On VPN provides a single, cohesive solution for remote access and supports domain-joined, non-domain-joined (workgroup), or Azure AD–joined devices, even personally owned devices. With Always On VPN, the connection type does not have to be exclusively user or device but can be a combination of both. For example, you could enable device authentication for remote device management, and then enable user authentication for connectivity to internal company sites and services.

Scoping


1.     What type of VPN client is being?

        a.Microsoft or third-party VPN client? 

        b. If third party, UWP APP based or 'fat' client?

        c. If Microsoft, how configured? Via SCCM, Intune, CMAK, GPO or manually...?

2.     User tunnel and device tunnel or just one of them?

3.     What type of VPN Server is used (NLB, Hardware LB?)

4.     What type of Radius server is used?

       a. If NPS, are there any extensions used?

5.     What authentication methods are being used?

6.     Are there any error messages seen? If so, what are the error messages what is the error code?

7.     Who is affected?

          a. All users or just some users (specific group/ OU/ setup?) 

          b. How often does this issue occur? 

          c. What is the actual business impact?


8.     Was this working previously, or has it never worked?

9.     If working previously, did anything change prior to the issue (i.e. was there a patch installed; was there a change to the infrastructure; etc.)?

10.  Do you have the internal and external NICs on the VPN server configured correctly? Are they in different subnets? Does the external NIC connect to the correct interface on your firewall?

11.  Check to see whether users can establish VPN connectivity

    1. Not all VPN problems involve connection failures. Sometimes, users can connect, but they can't access network resources. Determining whether the user can establish VPN connectivity will help you narrow down the areas in which you should be looking for problems.

12.  Check for or policies preventing connectivity:

          a.Check the Active Directory Users and Computers console to verify that the user has been given permission to log in remotely respectively if ‘’Ignore user account dial-in properties’’ is configured on the NPS Network Policy. 

          b. Likewise, some VPNs are designed so that users can log in only during certain times of the day;


13.  Can you access the VPN server from an external network? Consider opening Internet Control Message Protocol (ICMP) to the external interface and pinging the name from the remote client. After a ping is successful, you can remove the ICMP allow rule.

 

Data Collection



1) Download the latest TSS tool from here:

https://github.com/CSS-Windows/WindowsDiag/tree/master/ALL/TSS

2) On the Windows client machine, launch a CMD session as admin and change the directory to where you have downloaded the tss tool

3) Launch the following command:  Tss ron vpn Video for standard VPN troubleshooting

    3.1) If the issue is related to a more complex authentication/TLS issue use the following command: Tss ron vpn auth Video   

4) If you need data from the NPS Server launch the following command: tss rOn General NPS

5) If you need data from the RAS Server launch the following command: tss rOn RAS

6) Follow the instructions on the CMD session and reproduce the issue

7) After determining the proper troubleshooting scenario and having reproduced the issue, stop the data collection process with the following command: Tss Off

8) When prompted with the question if you want to collect an SDP (support diagnostic package) type: Y on the client, the RRAS server and NPS server

9) The resulting logs will be saved under the following location C:\MS_DATA 

10) Upload the resulting logs from C:\MS_DATA to the DTM secure transfer space.

Observation: There will be situations where the customers might not be able to launch the TSS tool due to various reasons (company security policy, OS issues, etc.). Please refer to the below section for “Manual tracing Steps” .

Restore Default IKEv2 config

In the process of testing it may be necessary to restore the default IKEv2 configuration on both the client and the server. This can be accomplished by running the following PowerShell cmdlets

Server –  Set-VpnServerConfiguration -RevertToDefault

Client –  Set-VpnConnectionIPsecConfiguration -ConnectionName [connection_name] -RevertToDefault -Force

Manual tracing steps (non TSS)

On the NPS Server save this script and launch it via a PowerShell ISE session as admin:


----------Start-trace.bat-------------------------- 

Echo Off 
wevtutil sl /e:true Microsoft-Windows-CAPI2/Operational /ms:15728640
nltest /dbflag:0x2080ffff 
netsh nps set tracing *=verbose  
netsh ras diag set trace enable 
netsh ras diag set trace clear 
netsh nps show config > %windir%\tracing\%computername%-NPSConfig.txt   
logman create trace "ds_security" -ow -o %systemroot%\tracing\ds_security.etl -p {91CC1150-71AA-47E2-AE18-C96E61736B6F} 0xffffffffffffffff 0xff -nb 16 16 -bs 1024 -mode Circular -f bincirc -max 4096 -ets 
logman update trace "ds_security" -p {37D2C3CD-C5D4-4587-8531-4696C44244C8} 0xffffffffffffffff 0xff -ets 
logman update trace "ds_security" -p {1F678132-5938-4686-9FDC-C8FF68F15C85} 0xffffffffffffffff 0xff -ets 
logman start trace "EapHostAuthr" -o %systemroot%\tracing\EapHostAuthr.etl -p {F6578502-DF4E-4a67-9661-E3A2F05D1D9B} 0x4000ffff 0 -ets 
netsh trace start capture=yes overwrite=yes tracefile=%windir%\tracing\NPS-SRV.etl netconnection globallevel=0xff 
Echo Please Reproduce the issue and then 
Pause 
REM STOP Tracing 
Netsh trace stop 
logman stop "ds_security" -ets 
logman stop "EapHostAuthr" -ets 
netsh ras diag set trace disable 
netsh nps set tracing *=none 
wevtutil sl /e:false Microsoft-Windows-CAPI2/Operational /ms:15728640 
wevtutil epl Security %systemroot%\tracing\%computername%_security.evtx /ow 
wevtutil epl System %systemroot%\tracing\%computername%_System.evtx /ow 
wevtutil epl Microsoft-Windows-CAPI2/Operational %systemroot%\tracing\%computername%_CAPI.evtx /ow 
copy %windir%\debug\netlogon.log %systemroot%\tracing\ /y 
reg export HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecurityProviders\Schannel %systemroot%\tracing\SCHANNEL-REG.txt 
----------Start-trace.bat  end --------------------------




On the RAS Server save this script and launch it via a PowerShell ISE session as admin : 

----------Start-trace.bat-------------------------- 
Echo Off 
wevtutil sl /e:true Microsoft-Windows-CAPI2/Operational /ms:15728640 
netsh ras diag set trace enable 
netsh ras diag set trace clear 
netsh ras dump > %windir%\tracing\%computername%-RASConfig.txt 
netsh trace start capture=yes overwrite=yes tracefile=%windir%\tracing\RAS-SRV.etl VPNServer,netconnection globallevel=0xff 
netsh wfp capture start file = %windir%\tracing\wfpdiag 
Echo Please Reproduce the issue and then 
Pause 
REM STOP Tracing 
Netsh wfp capture stop 
Netsh trace stop 
netsh ras diag set trace disable 
wevtutil sl /e:true Microsoft-Windows-CAPI2/Operational /ms:15728640 
wevtutil epl Security %systemroot%\tracing\%computername%_security.evtx /ow 
wevtutil epl System %systemroot%\tracing\%computername%_System.evtx /ow 
wevtutil epl Microsoft-Windows-CAPI2/Operational %systemroot%\tracing\%computername%_CAPI.evtx /ow 
reg export HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecurityProviders\Schannel %systemroot%\tracing\SCHANNEL-REG.txt 
----------Start-trace.bat  end --------------------------

In Case of Windows Server 2019 please make sure that capture=yes is not used in netsh command as that might disconnect the already connected user.
