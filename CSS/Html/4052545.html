ASMS – Azure Billing APIs

Submit a ModTrack request to update this TSG.  

Overview 

The Azure Billing Application Programming Interfaces (APIs) are designed to allow Azure customers to get information about the pricing and usage of various Azure services.

Applies To: This TSG applies to customers using the public Azure Billing APIs only. Enterprise Agreement (EA) customers, Cloud Solution Provider (CSP) partners, and customers looking for assistance using PowerBI should be handled as directed in the “Policy” section below.

Policy 

As there are three different Billing APIs available (Invoice, RateCard, and UsageAggregates) please confirm which Billing API a customer is experiencing trouble with, before starting to troubleshoot the problem.

Enterprise Agreement (EA) Customers -  For EA subscriptions, only the consumption.azure.com APIs are applicable. EA Customers using https://ea.azure.com may be using Consumption APIs, not the public Azure Billing APIs.  

The only exception is that EA customers may use the Public UsageAggregate API to track usage beyond EA commitment amounts.  The Ratecard API is not applicable to EA customers as the Ratecard data is obtained via EA Billing APIs only, not the public Billing APIs.

To learn more about Consumption APIs click reference the following public facing pages:

  • Consumption API Overview

  • Reporting APIs for Enterprise Customers

Note: All Management API issues are handled by Engineering. 

Case handling for EA Customers 

  • EA Consumption Billing API and Ratecard scenarios are handled by the Azure EA Portal Support Team. Engage this team via a collaboration task. 
    • SR Routing Product – Azure EA Portal (Non – GBS)
    • Target Queue – Azure EA Portal Support.
  • EA Management APIs are handled by ACE via IcM

Cloud Solution Provider (CSP) using Partner Center APIs and CREST - Partners using https://api.cp.microsoft.com are using the Partner Center APIs, not the public Azure Billing APIs, and are CSPs. Partners in these spaces will get errors when running the APIs and this is expected. 

Case handling for Cloud Solution Providers (CSPs) using Partner Center APIs and CREST - CSP (Partner Center and CREST) partner cases should be routed to the Global PC Partner EN queue.

 Use the following routing path:

  • Transfer reason – Misroute
  • SR Routing Product – Partner Center

However if customer also mentions comparison of PowerBI data with Azure Cost Management or ACM data, then a [Asset 4498193] should be sent to the Azure Cost Management team instead.

Customer Experience & Provisioning

Currently Azure is offering the following APIs:

Invoice

This API allows you to Access your Azure invoice in PDF Format.

RateCard

This API can be used to get the meter / resource metadata information along with prices.

UsageAggregates

This API allows you to get consumption data for an Azure subscription. It returns all the usage data incurred on an Azure subscription and NOT JUST the usage data that was considered for billing – this might include any usage at the end of the billing cycle that was discarded due to lateness.


Invoice

The information returned by the invoice API is the exact same PDF invoice file that is available in the portal.

RateCard

The RateCard API allows you to query for the resource/meter metadata and related prices used in a given subscription by:

  • Offer ID
  • Currency
  • Locale
  • Region

Sources for parameters for Ratecard

There are presently no APIs which will get the parameters for an existing subscription.  The customer should plan to obtain these parameters manually for input into the APIs. 

  • OfferDurableId: https://azure.microsoft.com/en-us/support/legal/offer-details/ lists all offers. You need to add "MS-AZR-" to the beginning of the offer number from the table. (Example: MS-AZR-0003P) 
  • Currency: https://en.wikipedia.org/wiki/ISO_4217 lists ISO currency codes.  Not all countries may be represented in this API. 
  • Locale: https://msdn.microsoft.com/en-us/library/ee825488(v=cs.20).aspx 
  • Region Info:  https://en.wikipedia.org/wiki/ISO_3166-2

UsageAggregates

The UsageAggregates API allows you to query aggregate Azure subscription consumption data by:

  • Start and end date/time
  • Aggregation granularity (ie: daily, hourly)
  • Instance level detail (ie: for multiple instances of the same resource)
     

Support Scenarios

Support cases which are routed to the Billing APIs may fall into one of two categories: Billing and Technical. ASMS handles the Billing questions, while the CIP-Cloud Cost Management API (CIP-CCMAPI) and Azure Developer Team handle the Technical cases. 

The CIP-Cloud Cost Management API (CIP-CCMAPI) team supports the following Billing API scenarios:

  • API issues
  • Exports
  • Issues not in any other category

Additionally, all EA Billing Consumption API issues should be sent to Azure Ops EA Team.

Note: The EAO Team only supports Billing Consumption API's.

API-specific Scenarios

Invoice

The information returned by the invoice API is the exact same PDF invoice file that is available in the portal. There should be no difference regarding the use of this API compared to the information in the normal monthly invoice and any discrepancies should be treated as regular billing questions.

What permissions do I need to access Invoices? The Azure Billing REST APIs are implemented as a Resource Provider as part of the Azure Resource Manager and share its dependencies. Access control for Azure Resource Manager uses the built-in Owner, Contributor, and Reader roles via the Role Based Access Control (RBAC) feature in the Azure Preview portal. You must make sure that the logged-in account is assigned one of those three roles for the specified subscription. By default, all service administrators and co-administrators are assigned the Owner role, and have implicit access. For details, see Role-based access control in the Microsoft Azure portal.

To allow caller for Invoice API:

  1. The Account admin must allow other users to download invoices.
  2. Adding the user to the role.

If this does not work then this would be an incident on billing extension/billing API team Azure Commercial Experiences, Signup and Provisioning BMX (BillingExtension).

For more information about the Invoice API, see: https://azure.microsoft.com/en-us/blog/azure-billing-reader-role-and-preview-of-invoice-api/

Note: For questions regarding the Invoice API, submit a ticket via this IcM. 

RateCard

Missing Meters:

The most common cause of a meter missing from the RateCard responses is that the meter’s Public Disclosure Date (PDD) is future-dated, either intentionally or accidentally. The PDD updates are owned by the Azure BizOps team and any update would need to run through their team. If there is a report of a meter not showing up in RateCard, contact the Azure BizOps team and ask them to check on the meter's PDD.

  • If the meter's PDD is future-dated, they can either update appropriately or communicate that the meter is indeed meant to be hidden.
  • If the meter's PDD is back-dated and it is still not showing, engineering investigation/effort will likely be required. In these case, contact the RateCard engineering contacts for assistance.

Server-Side Errors:

  • Any server errors should be escalated to the engineering team via ICM
  • The most common error is a 502 BadGateway, this will occur when requests are not being served within 100 seconds.

Other Questions:

Q: Is it possible to derive or automatically supply the RateCard parameters (offer ID, currency, locale, region)?

A: We don't currently have a way to do this in RateCard, and as such all parameters are required.

Q: Where can I get a list of offer Durable Ids?

A: https://azure.microsoft.com/en-us/support/legal/offer-details/ lists all offers. The offer number from the table must be prepended with "MS-AZR-" Eg: MS-AZR-0003P.

Q: I'm calling ratecard through ARM (direct) with a CSP subscription and I'm getting an error?

A: CSP subscriptions must use partner ratecard to access ratecard. This is supported by the Partner Center support team.

UsageAggregates

Common parameters

The following information is common to all tasks that might be performed related to getting usage details:

  1. Replace {subscription-Id} in the request URI with your subscription identifier
  2. Replace {api-version} in the request URI with 2015-06-01-preview
  3. Set the Content-Type request header to application/json
  4. Set the Authorization request header to an OAuth bearer token formatted as a JSON Web Token, which you obtain from Azure Active Directory. For more information, see Authenticating Azure Resource Manager requests.

These are the parameters used for the UsageAggregate API. Notice the Replace commands which are common to API terminology. The customer may contact ASMS because they aren’t getting expected results from their UsageAggregates API query. Typical issues are typos in the Subscription ID, API-version incorrect, Content Type is not JSON, or Web Token is not formatted as JSON.

Parameter-related cases should go to the POD Azure Developer team for more information.

Billing Issues

In general, billing issues include anything having to do with general questions about what APIs are available or with the output of the bill:

  • What APIs are available for getting pricing and usage information from Azure?
  • Can I download an invoice?
  • How do I calculate my bill using the Ratecard and Usage APIs?
  • Why does my Usage report not match up with my Invoice?
  • Why is the MeterId missing/wrong/different than what I expect?
  • Why are there multiple outputs for a resource for a single day?

Note: Answers for most of these questions can be found here: https://msdn.microsoft.com/en-us/library/azure/mt219001.aspx#Anchor_2

General API FAQ:

Q: The API can’t be consumed for Multi-Enrollment scenarios?

A: As of this time, that option is not available.

Q: I need to associate Billing data and Monitoring data via the Billing API.

A: The option we have is via Microsoft’s recent acquisition, Cloudyn, because it’s a mix of Billing API and Monitoring API functions.

Q: Could we perform resource sizing via the API without using third-party software?

A: As of now this is not possible without using Cloudyn features.

Q: Tags applied to Resource Groups are not showing in billing API report?

A: This is by design. Only tags applied to resources are visible in the report pulled via the Billing API.

Q: My refresh is taking forever.

A: Refresh in PowerBI will pull all the data from scratch and not just the data since the last import. This is time consuming and can take up to an hour if the volume of data is large.

Q: I’m unable to download EA MarketPlace cost via the Billing API.

A: Both API v1 and v2 do not have one-time charges enabled. The product team is considering this feature but do not have a timeline available.

Technical Issues

The agreement is that 100% of Billing API support topic cases will come to ASMS queue, we will do initial triage of the case, and for the small percentage of cases that are coding/API we will transfer the case to the technical team. Technical issues include anything having to do with making actual calls to an API, such as:

  • Errors when calling an API
  • How do I authenticate to the API?
  • How to register a Billing API application in the Azure AD
  • How to write the Powershell/REST/C#/Python code to call an API
  • What is a ContinuationToken and how do I use it?
  • Why is the Billing API call returning http 503 Technical issues, please set the correct Support Topic and route technical cases to the Pod Azure Dev queue. Do not override the Support Topic.
  • All EA related API issues should be sent to Azure Ops EA Team.
