2. Guidance

When there is no content in Service Desk Knowledge (SDK) or an SDK article with instructions on how to escalate a platform, billing or subscription issue, an escalation needs to be sent to Commerce Engineering Tier 3.

Ensure that personal information of the customer is updated and has completed in CMAT as part of the troubleshooting when encountering portal issues. Ensure that every line item in CMAT’s account tab is filled out when encountering portal issues. A portal error is common when personal information is not updated in CMAT and left blank. 

Note: All portal issues escalated to Tier 3 require the trace information.

Scenarios that do not require a POD TEAMS Consultation
  1. Unknown charge inquiry (after exhausting all possible resources) – make sure the escalation template in Rave has PI details and POP before escalating.
  2. Portal issues (if no reported/emerging issue, should have complete troubleshooting/repro steps outlined) - advocates need to make sure the profile in CMAT is showing a valid profile, always include the error message and know when to go to Fraud/RiskOps vs. Tier 3.
  3. Emerging issues (recognized by T3 but no internal action/workaround yet implemented).
  4. Missing invoice / invoice not generated without any subscription activity – Make sure tenant anniversary date is looked at before escalation.
  5. Payment not posting in CMAT despite payment having cleared on customer’s end (credit card or invoice).
  6. Unban credit card.
  7. Previous assistance request recommendations did not work for customer.

3. Procedure


Note: This procedure must be followed by Tier 1 and Tier 2.
a. Basic troubleshooting steps

 

  1. Verify if the customer’s alias has a valid first and last name and the customer has a valid and completed address.
    • If yes, continue with escalation.
    • If no, have the customer add valid first and last name as well as a valid and completed address, and attempt the action once again.

    Note: Customers cannot proceed with their action if the first and last name fields are empty. For example, when the customer is making a purchase, the customer needs to enter valid first and last name instead of the company name. Otherwise, it throws an error.
  2. Verify if the customer is using the correct credentials (Global or Billing Admin).
  3. Verify if the customer is using the correct portal or URL (Microsoft 365 admin center or IAP). 
  4. Verify if the customer is using Internet Explorer 11 or higher. For Office 365 browser support scenarios, see here.
    Note: If the customer is not using Internet Explorer 11 or higher, then the following procedure steps might not work.
  5. Make sure that the Office 365 Admin Portal is not on Preview mode. If so, ask customer to turn off the Preview version.
  6. Repro the issue after adding the site to the browser’s Trusted sites.
  7. Repro the issue from an "In Private" browser session (CTRL+SHIFT+P from Internet Explorer).
  8. Repro the issue in "Compatibility Mode" (Internet Explorer Settings>Compatibility View Settings).
  9. Enable third-party Cookies (Internet Explorer Settings>Internet Options>Privacy>Advanced Privacy Settings).
  10. Clear Cache and Cookies (CTRL+SHIFT+Del from Internet Explorer).
  11. Log out of Microsoft Admin Center (MAC) and log back in.
  12. Has another admin attempted to reproduce the error?
    • If yes and successful, have the admin that cannot perform the action update the browser settings to match.
    • If yes and not successful, continue with escalation providing trace logs and error messages.
  13. If the issue persists, assist the customer in retrieving complete trace information. See the information on Portal issues and Using Problem Steps Recorder under Supporting documents section below.

    Note: The source code/trace logs are only good for 3 days. “TraceInfoDiv” is needed for the Engineering team’s investigation. If customer is unable to provide the “TraceInfoDiv”, there should be at least a complete repro steps or PSR.
b. Escalation steps:

EMEA

  1. Please add in RAVE on the last note before escalating the required escalation template and the logs gathered during the troubleshooting. Please see Section 4. Escalation Templates.

    Note: For certain scenarios that needs GFS (Global Financial Services), Risk Ops, BG or Volume Licensing team attention, there will be internal articles instructing the advocates to complete specific templates. Please make sure those templates are documented according to the scenario you are investigating, and they correspond to internal articles.
  2. All data must be accurate. Repro steps, troubleshooting steps, and the time and date with the time zone of any error message and error code must be included in the escalation.
  3. For portal errors, attach the trace information word or text document provided by the customer. Retrieve the error information (i.e. error code, error message details) as well as the trace (view source info) from the error page. Always check if in the file you find the PUID, TID, CID, SID, or GEO info. If not, then troubleshoot browser ‘Security settings’ (i.e. Trusted sites and Restricted sites).

    Note: If the customer does not want to send the entire trace out of security concerns, explain to the customer that the web page source does not include any data that will compromise their PII. It simply captures the HTML code that is running in the back, and it contains what parts of the code were running or throwing an error so that we can find the correlating information in our server logs to determine what went wrong.
  4. Escalation is done by submitting a level 2 assistance request in Rave. 

    Note: DO NOT     close a Rave case that is escalated to Tier 3. If a case needs an update or to be closed, the TL/SME should ping the level 2 assistance request owner (Tier 3).
  5. Ensure that the proper Customer Type is selected. Only ONE selection is allowed.
    [Asset 4506205]
     
  6. Document the Problem Scope field. If an escalated case is connected to a former one, this information (along with the previous case # and SEE owner) should appear in the Problem Scope filed in order for the SEE who previously handled the case, if available, to acknowledge the new case as well or, if not available, for the other SEEs to have all needed information in regards to previous troubleshooting when acknowledging the new case.
     
  7. Select the correct POD Tech and the Topic.

    Invoice & Billing - This POD handles any escalation related to Credits and Refunds, Payment issues, Tax information and Invoice related questions.

    Licenses & Subscriptions – This POD handles topic related with the purchase and activation of subscriptions, remaps, lifecycle notifications or cancellations.

    Portal, Accounts & Trials - This POD handles any issues related to Portal bugs, issues with trial subscriptions and tenant related questions.

  8. Based on which option the Ambassador/SME chooses in the POD Tech section, they will see different lists of Topics. Please make sure the topics are correctly selected.

 

NOAM

Note: The inquiry should be coordinated with T3 Engineering via the TEAMS POD Consult. If Tier 3 advised through POD to create an assistance request, follow the steps below:

  1.  Please add in RAVE on the last note before escalating the required escalation template and the logs gathered during the troubleshooting. Please see Section 4. Escalation Templates.

    Note:     For certain scenarios that needs GFS, Risk Ops, BG or Volume Licensing team attention, there will be internal articles instructing the advocates to complete specific templates. Please make sure those templates are documented according to the scenario you are investigating, and they correspond to internal articles.
  2. All data must be accurate. Repro steps, troubleshooting steps, and the time and date with the time zone of any error message and error code must be included in the escalation.
  3. For portal errors, attach the trace information word or text document provided by the customer. Retrieve the error information (i.e. error code, error message details) as well as the trace (view source info) from the error page. Always check if in the file you find the PUID, TID, CID, SID, or GEO info. If not, then troubleshoot browser ‘Security settings’ (i.e. Trusted sites and Restricted sites).

    Note: If the customer does not want to send the entire trace out of security concerns, explain to the customer that the web page source does not include any data that will compromise their PII. It simply captures the HTML code that is running in the back, and it contains what parts of the code were running or throwing an error so that we can find the correlating information in our server logs to determine what went wrong.
  4. Submit an assistance request in Rave.
    Note: DO NOT close a Rave case that is escalated to Tier 3. If a case needs an update or to be closed - the TL/SME should ping the case owner (Tier 3) using POD Teams.
If consultation about the error is needed, the consultant needs to add an event to the case so that others can read the ticket and understand the next steps to resolution.

5. Severity levels

a. Sev A – Immediate SLA

 

For service or tools Sev A issues, that are impacting customers, contact CSS Command Center (CSS CC), csscc@microsoft.com , and loop in your local Tier 3 via TEAMs.

 
b. Sev B – 11 business day TTR

 

Between the hours of 6 A.M. and 6 P.M. (UTC/GMT -8 hours), Monday through Friday, Tier 3 will action the case within 4 hours of being transferred.

Criteria:

At least 1 customer is blocked without a workaround.

For example: Customer is unable to purchase additional seats on a paid subscription.
c. Sev C – 11 business day TTR

 

Important issues that do not have a significant current service or productivity impact for the customer.

6. Supporting documents

  4564762 - Informational: M365: Assistance Request V2 for Broad Commercial support
Portal issues

 


The Microsoft 365 admin center errors are easily identified by the presence of a Correlation ID and Error Code.

Please complete the following steps and add the output to the template when escalating a portal error:

Note: These steps must be completed while the error is on the customer’s screen.
  1. Ask the customer to add "microsoftonline.com" to the compatibility list in Internet Explorer.
    From Internet Explorer:
    1. Click Settings.
    2. Click Compatibility View Settings.
    3. Enter microsoftonline.com in the blank box.
    4. Click Add.
  2. Using the section Using Problem Steps Recorder below, have the customer reproduce the error and/or issue using the same steps and account that was previously used. Make sure the customer is logged into the portal.
  3. Have the customer reproduce the error and attach the output from the Problem Steps Recorder (PSR) to the work item.

    Note: If the customer is unwilling to complete this step, please indicate this under the troubleshooting section of the template.
  4. Document all of the repro steps from login to error message.
  5. Request that the customer right-clicks anywhere on the O365 ERROR PAGE and select the view source item from the context menu.

    Note: If the customer does not want to send the entire trace out of security concerns, explain to the customer that the web page source does not include any data that will compromise their PII. It simply captures the HTML code that is running in the back, and it contains what parts of the code were running or throwing an error so that we can find the correlating information in our server logs to determine what went wrong.
  6. Ask the customer to select all of the text in the newly opened window by pressing CTRL+A.
  7. Copy the highlighted text to the clipboard (CTRL+C).
  8. Paste (CTRL + V) the text into a text document and email to the advocate using the case or attach to the case.
  9. Add the collected repro steps to the completed Engineering Template.
  10. Include the text file in the work item to Tier 3.
Using Problem Steps Recorder


If the Problem Step Recorder (PSR) tool is to be used, please follow 4464265 - How to use Problem Steps Recorder.
