The Svchost.exe process may end unexpectedly on a Windows Server 2003-based computer

Microsoft Internal Support Information

BUG #: 36934 (Content Maintenance)

You may experience one or more of the following symptoms on a Microsoft Windows Server 2003-based server:
  • Many applications and services do not work correctly because the Svchost.exe process ends unexpectedly. Errors that are similar to the following are logged in the Application event log:

  • The following error message is logged in the System event log:

    Event Type: Error
    Event Source: Service Control Manager
    Event Category: None
    Event ID: 7023
    Description:

    The Server service terminated with the following error:
    Access is denied.
    Note A text-based version of the System event log contains the following error message:
    The Server service terminated with the following error: %%5
  • The Server service stops and does not restart. The following error message is logged in the System event log:

    Event Type: Information
    Event Source: Service Control Manager
    Event Category: None
    Event ID: 7036
    Description:

    The Server service entered the stopped state.

    Event Type: Error

    Event Source: Service Control Manager
    Event Category: None
    Event ID: 7023
    Description:
    The Server service terminated with the following error:
    The server is in use and cannot be unloaded.
    Note A text-based version of the System event log contains the following error message:
    The Server service terminated with the following error: %%1811
    Error 1811 maps to the "ERROR_SERVER_HAS_OPEN_HANDLES" error. The server is in use and cannot be unloaded.

    Additionally, the Windows Management Instrumentation (WMI) service stops, and the Server service ends unexpectedly. The following error message is logged in the System event log:
    Event Type: Error
    Event Source: Service Control Manager
    Event Category: None

    Event ID: 7032
    Description:
    The Service Control Manager tried to take a corrective action (Restart the service) after the unexpected termination of the Windows Management Instrumentation service, but this action failed with the following error:
    An instance of the service is already running.
  • On a Windows cluster node, the cluster file server resources cannot come online. Additionally, the server may log the following error message in the System event log:
    Event Type: Error

    Event Source: ClusSvc
    Event ID: 1068
    Description:
    Cluster file share resources <ResourceName> failed to start with error 2114

    Event Type: Error
    Event Source: ClusSvc

    Event ID: 1055
    Description:
    Cluster File Share resource <ResourceName> has failed a status check.

    The error code is 2114.
  • The Srvinfo.exe tool does not work correctly. (This tool is included in the Windows Server 2003 Resource Kit.)
  • When you manually try to start the Server service by using the Computer Management snap-in or by typing net start server at the command prompt, you may receive the following error message:

    Could not start the Server service on Local Computer.
    Error 5: Access is denied.
    Notes
    • This behavior occurs even though the user account is a member of the local Administrators group and the Server service is configured to start by using local system credentials.
    • Typically, a restart of the computer temporarily fixes the issue.
This problem may occur if the computer experiences a low-memory condition. When the computer experiences a low-memory condition, memory corruption may occur in the private heap of a dynamic link library (.dll) file. These random heap corruptions may cause programs to crash.


Memory corruption may occur when you run memory-intensive programs on a computer that has the Intel Physical Addressing Extensions (PAE) specification enabled. PAE enables a computer to support more than 4 gigabytes (GB) of installed memory.

The Server service runs on a Windows Server 2003-based computer together with some other services in a Svchost.exe process. If the Svchost.exe process ends unexpectedly in a low-memory condition, the process cannot restart.
To work around this problem, turn off the PAE specification. You must either remove the /PAE switch or include the /NOPAE switch in the Boot.ini file.

Notes
  • When you remove the /PAE switch from the Boot.ini file, the computer cannot use all its available memory if it is equipped with more than 4 GB of RAM.
  • Some servers have a "hot plug memory" feature that enables PAE even when the /PAE switch is not set in the Boot.ini file.
To verify that the /PAE switch is enabled on the server, examine the following registry subkey:
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management

Key Name: PhysicalAddressExtension

Type: DWORD

Value: 1
Note Even if you set this registry value to 0, it may automatically reset to 1 after a restart, unless the /NOPAE switch is set.
To resolve this problem, use one of the following methods:
  • Update the system to Windows Server 2003 Service Pack 1 (SP1).
  • Install hotfix 895575.
    For more information, click the following article number to view the article in the Microsoft Knowledge Base:
    895575 A process that runs in the Physical Address Extension (PAE) kernel may experience memory corruption in Windows Server 2003

Note Hotfix 895575 is recommended for Windows Server 2003-based server clusters and for Microsoft SQL Server 2000 installations. For more information, click the following article numbers to view the articles in the Microsoft Knowledge Base:
895092 Recommended hotfixes for Windows Server 2003-based server clusters

  • The RTMGDR version contains only security hotfixes and software updates that address widespread critical issues, such as security vulnerabilities, and that are widely released by Microsoft.
  • The RTMQFE version contains security hotfixes, GDRs, and cumulative hotfixes.
To verify that the correct version of hotfix 840987 is installed, type the following command at the command prompt:
filever /v %systemroot%\system32\file_name
Note file_name is one of the following files:
  • Ntkrnlpa.exe on PAE-enabled systems.
  • Ntkrnlmp.exe on multiprocessor systems.
  • Ntkrpamp.exe on multiprocessor PAE-enabled systems.
Locate the line that starts with "FileVersion." The correct version shows the qfe string as follows:
FileVersion 5.2.3790.177 (srv03_qfe.040410-1236)

Troubleshooting

To look for a possible memory leak, run the System Monitor tool. The Server service will not start in a low-memory condition.

Search for and save existing User.dmp and Drwtsn32.log files so that these files are not overwritten during another process crash.
For more information, click the following article number to view the article in the Microsoft Knowledge Base:
836080 Drwtsn32.exe does not create a User.dmp file or a complete log entry on a Windows Server 2003-based computer

If Windows Error Reporting does not detect the dump file, set up the Userdump.exe tool or the ADPlus tool to monitor the Svchost.exe process.
For more information, click the following article number to view the article in the Microsoft Knowledge Base:
241215 How to use the Userdump.exe tool to create a dump file


Note If a Windows Server 2003-based computer is working under a heavy load or if many threads are open, Windows Error Reporting may not create a dump file when a program or a service crashes.

Install hotfix 837018. This hotfix is described in the following Microsoft Knowledge Base article:
837018 Windows Error Reporting may not create a dump file in Windows Server 2003


To enable a full crash dump type, follow these steps:
  1. Click Start, click Run, type DRWTSN32, and then click OK.
  2. On the Crash Dump Type row, click the radio box for Full, and then click
    OK.
To change the recovery option for the Server service, follow these steps:
  1. Click Start, point to
    Administrative Tools, and then click
    Services.
  2. In the right pane, right-click Server, and then click Properties.
  3. Click the Recovery tab.
  4. In the First failure list, click
    Restart the Service.
  5. In the Second failure list, click
    Restart the Service.
  6. In the Subsequent Failures list, click
    Restart the Service.
  7. Click Apply, and then click
    OK.
  8. Right-click Server, and then click
    Restart.
The ADPlus tool lets you obtain a memory dump file from a process that is causing problems. To obtain the tool, visit the following Microsoft Web site:
http://www.microsoft.com/whdc/devtools/debugging/default.mspx
After you have installed the Windows debugging tools, follow these steps:
  1. Create a folder that is named C:\Adplus.
  2. Open a command prompt, and then move to the folder where you installed the debugging tools.
  3. At the command prompt, type CD "C:\Program Files\Debugging Tools for Windows", and then press ENTER.
  4. Type "cscript adplus.vbs -crash -p
    PID -fullonfirst -o c:\adplus"    , and then press ENTER.

    Notes
    • PID is the process ID for the Svchost.exe process for LanManServer. Use the TLIST -s command to determine the Svchost.exe process ID.
    • Do not interfere with the Command Prompt window that opens.
    • You cannot log off the system when you are monitoring.
  5. After a crash occurs, put the contents of the Command Prompt window in a .zip file, and then send them to the C:\Adplus folder.
For more information, click the following article number to view the article in the Microsoft Knowledge Base:
286350 How to use ADPlus to troubleshoot "hangs" and "crashes"

Isolate Svchost services

Important This section, method, or task contains steps that tell you how to modify the registry. However, serious problems might occur if you modify the registry incorrectly. Therefore, make sure that you follow these steps carefully. For added protection, back up the registry before you modify it. Then, you can restore the registry if a problem occurs. For more information about how to back up and restore the registry, click the following article number to view the article in the Microsoft Knowledge Base:
322756 How to back up and restore the registry in Windows


For troubleshooting purposes, you can isolate Svchost services into a svchost process. To do this, follow these steps:
  1. Start Registry Editor.
  2. Locate and then right-click the following registry subkey:
    HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\SvcHost
  3. Point to New, and then click Multi-String Value. Add the following information:

    Name: TempGrp
    Type: REG_MULTI_SZ

    For the data, enter the names of the services that you want to isolate. This information will be similar to the following:
    TempGrp REG_MULTI_SZ LanManServer

  4. Locate and then right-click the following registry subkey:
    HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\SvcHost


    Point to New, and then click Key.
  5. For the new TempGrp key, create the following entries.

    Note These entries are different for Windows Server 2003 and for Windows 2000. Windows Server 2003.

    Windows Server 2003 and Windows XP
    NameAuthenticationCapabilities
    TypeREG_DWORD
    Value3020 (hexadecimal)
    NameCoInitializeSecurityParam
    TypeREG_DWORD
    Value1
    Windows 2000
    NameCoInitializeSecurityParam
    TypeREG_DWORD
    Value1
    To create the CoInitializeSecurityParam entry, follow these steps:
    1. Locate and then click the following registry subkey:
      HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows NT\CurrentVersion\SvcHost\TempGrp
    2. On the Edit menu, point to
      New, and then click DWORD Value.
    3. Type CoInitializeSecurityParam, and then press ENTER.
    4. On the Edit menu, click
      Modify.
    5. Type 1, and then click
      OK.
  6. Locate and then right-click the following registry subkey:
    HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\LanManServer
    Or, click the services that you are adjusting, and then change the ImagePath value to:
    %systemroot%\System32\svchost.exe -k TempGrp
To simplify this task, you can use the IsoServ.exe utility. To obtain this utility, visit the following Microsoft Web site:
http://toolbox/sites/23251/default.aspx

      OK.
    2. In the right pane, right-click Server, and then click Properties.
    3. Click the Log On tab, and then click
      Local System account.
    4. Click OK.
  3. Verify that the Local System account has Full Control access to the %systemroot%\System32 folder.
  4. To obtain the security descriptor and then run it in security descriptor definition language (SDDL), type SC SDShow LanmanServer > c:\SD.txt at a command prompt. The output will be similar to the following:

    Start, click Run, type
