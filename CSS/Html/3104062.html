ASMS – Misroutes and Duplicates

Submit a ModTrack request to update this TSG.

Overview

This TSG provides guidance on how to handle case incorrectly sent (misrouted) to ASMS, that should be supported by another team. It also explains how to handle duplicate cases.  

Important: It is crucial ASMS engineers reroute technical misroutes to the correct support team through this Misroute process as soon as possible once the case is assigned. It is also important to update the case's Routing Product to be the correct non-ASMS product and support topic. We must be sure to give the support team enough time to assign the case and meet FQR before it expires.

Policy

  • Duplicate Cases

    • If you receive a duplicate case, you should follow the Quick Close scenario steps for duplicate cases outlined in CSS’s Commercial Case Management Standards. 

  • Misroutes 
  1. If you receive a case that is a technical misroute, collect as much information as possible about the request from the customer and document it in the case. Be sure to also include why this case is being transferred and do not accept ownership of the case. 
    • If you have already taken ownership of a case and sent FQR but later find that it is technical in nature, continue to collect as much information from the customer. If the case verbatim does not clearly indicate which Azure technology the customer needs assistance with, ask the customer for a brief description of the issue so that you can send the case to the right team. Add a detailed case note explaining why this case is no longer in scope for ASMS. 

  2. Use the MSaaS Queues Portal to determine which queue the case belongs to.

    If you cannot confirm the correct queue and routing information even after searching in the Queues Portal for at least 5 minutes, email the Azure DMs (wadm@microsoft.com) and ask them to help you find the correct queue for that case. Include all the details of the request you collected from the customer, so that the DMs can use those details to find the correct queue.

  • Queues Portal Search Tip: Use keywords #AzurePOD or #AzureSupport for Azure Support Teams.
  • You do not need to use the WhichAzureTeam alias. The Duty Managers (DMs) will do that if needed.
  • Service Impacting Events (SIEs) have separate queues depending on what service is affected. Let the DMs figure which queue to use.

  1. Update the Routing Product and Support Topic of the case. When selecting the Support Topic, select the lowest level and most accurate topic (avoid Advisory) based on what you know about the customer's request.

    Note: Some teams will have multiple queues in which a case can land in, so accuracy is important. Do not transfer the case using the "Manual Override" feature.

    Be sure to also do the following before transferring : 

  • Change the title of the case by adding “MISROUTE” to the beginning of the title.
  • If the user has Basic or Developer level support, adjust the severity to Sev C.
  • If you accepted the case and sent FQR, then you will need to follow some additional steps:
    • Create a new Task titled “Customer Callback Needed.”
    • In the case notes, indicate the status of the issue and the customer expectations.

  1. Once you have transferred the case, let the customer know (using the preferred contact method indicated in the case) that you have transferred the case to the correct queue.

    • Note: If you receive pushback from one of the Azure Technical teams that they won't accept a support case opened as a Billing/Subscription case, please contact kimlftedr@microsoft.com with the case number and ask for assistance.

  2. If the customer emails you back or creates a new ticket with ASMS stating that they did not receive a response on the technical ticket within the expected SLA, do the following: 

    • Email the DMs at wadm@microsoft.com and ask them to check on the transferred case, as the customer has yet to be contacted. Be sure to provide them the case number of the transferred case.

    • If discovered that the case has not been assigned, email azuresvcsic@microsoft.com so that the technical DMs can assign the case. 

  • Cases transferred to ASMS when another team has partial ownership and has already engaged the customer:

    • If you receive a case from an Internal CSS or Non-CSS team (example - MPN, EA, AAD, VM) that has partial ownership of a case and they have already worked with the customer, you should not accept the entire case. The team who initially contacted the customer, should enagge us via a task (following the One Stop policy on Collaboration).

    • If you receive a case from an Internal CSS or Non-CSS team who has no ownership over the scenario, and it is determined that ASMS should own the case, the internal team should follow the One Stop Misroute process to send ASMS the entire case.

    • Per the Transfer Ownership policy for incoming case transfers, we should check if any part of the case is owned by the sending team. If so, have them follow the collaboration policy, not the misroute policy. You should not accept the case unless the SE has spoken with the other team prior to the handoff and everyone is following the correct OneStop procedures.

  • Initially misrouted cases that come back to ASMS for additional assistance:

    • Follow-up task for ASMS sometimes get transferred to the original engineer, or to that engineer’s team. This happens if a misroute is sent to a technical team who then needs additional assistance from ASMS. 
    • Leverage TSG 3189895 to ensure these follow-up tasks are routed to the correct ASMS queue, and not back to the original team that handled the misroute initially.
  • Multiple Spam or Auto-generated Cases
    • If you receive multiple spam cases from a customer or customers that have invalid contact information (email/phone) or indecipherable (gibberish) verbatims, file an incident. 

Azure Virtual Training

Microsoft Azure Virtual Training Days are not a part of Microsoft Certification, Courseware, Imagine Academy, Azure Dev Tools for Teaching, or other World Wide Learning or developer programs. It is a separate Microsoft initiative, so do not reroute such cases to the Technical Skills for Business Team.

  • The Technical Skills for Business Team is only involved in these types of cases if the customer becomes eligible for the Microsoft Certification exam discount by attending various Microsoft events, which may include Azure Virtual Training Days.
  • Then, and only if the customer has issues claiming the Microsoft Certification exam discount from Microsoft Learning Exam Registration, should the case be rerouted to the Microsoft Certification program. All other cases will be bounced back.

Note: “Microsoft Certified Professional" refers to Service Desk.

  • The Technical Skills for Business Team does support issues related to Azure Virtual Training Days. But it does not have their queue in Service Desk, so those cases cannot be rerouted to the team. Instead, escalate them internally to the team's alias email account at techskillssupport@microsoft.com. 

Note: DO NOT share this alias email address with customers!

Support Scenarios

Below is a list of common support scenarios that require special case handling. Refer to each scenario to learn how to best handle these cases. 

BizSpark Technical Misroutes

For BizSpark related issues refer to TSG 3070105.

HIT Related Duplicate Cases

If you receive a duplicate HIT case, you should do the following:

  1. Accept the case.
  2. Send FQR to the customer stating the issue will be managed in SR# XYZ (the original case) and that this new case will be transferred over to the owner of their existing case for administration.
  3. The duplicate case owner should then send the duplicate case over to be owned by the original case owner. The duplicate case owner should notify the original case owner via email that they are sending over this duplicate case for their ownership.
  4. The original case owner will keep both cases open until the issue is fully resolved, and then close all related cases at the same time.

Note: If a Premier SE receives a duplicate case which is already worked and owned by a Professional SE, the duplicate case should be sent to the original case owner, regardless of support level.

Note: If HIT says to stop working with a customer and the customer continues to file more duplicates, follow HIT's instructions in the ticket. Additional guidance on handling HIT cases can be found in TSG 2658538.

Microsoft Security Response Center Misroutes

If ASMS receives a Security misroute, set the SAP to Azure\Subscription management\Security and compliance requests for Subscriptions\Guidance for security advisories or vulnerabilities.

Note: MSaaS will then route the case accordingly.

If the customer has a security issue, advise the customer to submit abuse report at http://cert.microsoft.com.

MPN Misroutes

For more MPN related scenarios refer to TSG 2836909.

Non-Cloud Misroutes

Non-cloud cases are cases not related to Azure, M365, or any other cloud offer. If you receive such a case do the following: 

  1. Send an email with the following details to NOAM CSS Comm PostSales DP. 
    The subject line of your email must include Project Orion – SR#< > - <customer issue>: 
    • Customer contact information (email and/or phone number)
    • Microsoft product
    • Version info (if known)
    • Problem description
    • Service Desk case ID
  1. A tech router will receive this email and determine which team this issue should be routed to. Once the team is determined, the case will be taken over by that team. At that point, no further actions will be required of you as the ASMS SE.

O365/M365/D365 Commerce, Technical, Home, Concierge Support

For issues you assess as either O365/M365/D365 Commerce, Technical, Home, or Concierge, follow the engagement processes outlined in TSG 4506349.

Payment Central Login Issues

Microsoft Payment Central is the centralized onboarding and account management tool for partners/suppliers/payees receiving payment from Microsoft.

Sometimes customers experience Azure AD login issue, which prevents them from accessing Microsoft Payment Central. In these instances, The Payment Central AP Ops team will create and transfers cases to Azure AD Synchronization and Account Management teams to obtain further assistance for the following scenarios: Synchronization Issues, Viral Tenant, and Abandon Tenant.

For Microsoft Payment Central login issues, route the case to the AAD team based on the related scenario and provide the following details.

Scenarios:

  • Synchronization – Azure Active Directory\Active Directory to Azure AD Synchronization\Other failures regarding Active Directory to Azure AD synchronization
  • Viral Tenant – Azure Active Directory\Domain Name Management\Other questions regarding custom domain name management
  • Abandon – Azure Active Directory\Domain Name Management\Other questions regarding custom domain name management

Case Title: Microsoft Payment Central: AAD Support: CRM Ticket #

Case Details:

  • High Level Description:
  • Error messages received (if available):
  • Additional Background Information:
  • AP Ops Engineer Contact:
  • Customer Domain:
  • Closing Contact - Identity SEs need to send email to the contact (a DL) once case is closed.

Note: The AP Ops will use this email to confirm the issue is resolved and close their ticket, since they use a different CRM tool.

Sev A issues: Escalation AP Ops should email Azure Identity Tech Advisors at cloudidpodta@microsoft.com so they are aware of the case and take appropriate steps to ensure a resolution for the customer.

Other Common Issues

  • Users or IT admin do not know how they got onboarded to Azure AD.
  • IT admin left their organization and the new IT admin does know how to support Azure AD.
  • IT admin tried performing Azure AD onboarding on a trial basis but did complete.
  • Not all user accounts from the organization are synchronized with Azure AD.
  • Custom STS page errors/time out.
  • Suppliers do not know they have to use a corporate account with corporate credentials.
  • Suppliers do not know they should use domain\account format in their STS page.
  • Not all suppliers enabled a password reset experience or capability.

Note: The bulk of issues forwarded to CSS fall within the Sync category.

Premier, ProDirect, or Partner Support Misroutes

If you are assigned a non-technical ASMS case that is covered by a higher level of support entitlement (e.g., Microsoft Unified Support, Partner, Premier, or ProDirect), route it to the ASMS Premier Accounts and Portal queue. Please refer to TSG 2786284 for more information on identifying and routing these cases.
Public Sector (Project Five)

Public sector cases related to Australia, New Zealand, Canada, the United States, and the United Kingdom require special security requirements when handling their cases.

For more (optional) information, see the Project Five SharePoint.

While these cases should not come to ASMS, misroutes are always possible. In these instances, manually override the case routing and send it to the MSaaSFVEY queue if any of the conditions are true:

  • Project Five is part of your case title.
  • Your customer requests or mentions Project Five as a support option.

If you are unsure whether your case qualifies for this type of transfer, engage your TA/PTA.
Publisher Support

If you receive a case related to Publisher Support (i.e. General marketplace help, How To questions, or Deployment issue), transfer the case to the CSS PE Global Sonata Redmond/Hyderabad Marketplace EN T1 queue.

SAP Cloud Platform

SAP Cloud Platform is an open business platform for developing applications across multiple clouds (for example: GCP, AWS, and Azure). It enables customers to build applications and easily transport them between cloud providers.

Currently, there is no scope for ASMS as it relates to SAP. All SAP issues should be treated as a misroute.

  • Transfer SAP Cloud Platform technical support tickets to the SAP Embrace technical queue.
  • Transfer SAP Cloud Platform tickets with Accounts and Billing Scope to the EAO team.

Technical

Scenario 1: Customer with no technical support files a billing case for a technical issue

Do not accept ownership of the case. Update the Routing Product/Support Topic of the case to point to the correct technical subtopic (not to ASMS). Add “MISROUTE” to the case title and change the title of the case to accurately describe the issue. Send the case to the correct technical queue per the Queues Portal.

Note: Make sure severity aligns with support level. No Sev A or Sev B. Don't use the Advisory support topic as these users are not entitled to advisory support.

Scenario 2: Customer with technical support files a billing case, but is able to file a technical support case from the portal

Do not accept ownership of the case. Update the Routing Product/Support Topic of the case to point to the correct technical topic (not to ASMS). Add “MISROUTE” to the case title and change the title of the case to accurately describe the issue. Send the case to the correct technical queue per the Queues Portal.

Note: Make sure case severity aligns with support level. No Sev A or Sev B for Basic or Developer level support. Be sure to use the most accurate support topic when transferring. Don’t use the Advisory support topic unless it’s a Pro-direct or Premier customer and the issue is truly advisory.
 

Scenario 3: Customer with tech support files a billing case because they cannot file a technical support case from the portal

1. Keep the original case and use it to work with engineering to fix the entitlement in Ibiza. Be sure to collect screenshots from the customer.

2. Create a second case for the customer (using the Grace Case TSG 2895582) with their technical issue and send it to the correct technical queue.

Note: Make sure that severity aligns with support level. No Severity (Sev) A or B for Basic or Developer level support. Be sure to use the most accurate support topic when transferring. Don’t use the Advisory support topic unless it’s a Pro-direct or Premier customer and the issue is truly advisory.

Volume Licensing (VL)

For cases you access as VL scope, follow the guidance outlined in TSG 4506349.
