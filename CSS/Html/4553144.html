Microsoft Teams is built on top of Existing Microsoft Technologies:

• Office 365 Groups
• Office 365 Azure Active Directory
• Exchange
• SharePoint
• OneNote

Topic

Information

Office 365 Groups

 

 

• Microsoft Teams uses group membership as the access control list to Files and Notes tabs

 

• Owners of existing Groups can move them over to Microsoft Teams

 

• Team creation settings are controlled through the admin portal where you control group creation settings.

Azure Active Directory (AAD)

 

 

• Microsoft Teams uses your existing O365 Azure Active Directory credentials to sign in

 

• Your Identities are stored in Azure Active Directory

 

• Can leverage Azure Active Directory Modern Authentication for enhanced security

Exchange

 

 

• A group mailbox and calendar gets created for the Team

 

• Meetings created within Microsoft Teams gets pushed to your Exchange calendar

 

• Meetings created in Exchange get synced to Microsoft Teams

SharePoint

 

 

• Microsoft Teams create a SharePoint site for each Team

 

• Each Channel within a Team will get a folder on this SharePoint site

 

• Files shared within the Team are stored on SharePoint

OneDrive for Business

 

 

Files that are shared during private chats are hosted on the user’s

 

OneDrive for Business storage

 

• Navigating to the Files icon on the left side of the application gives

 

you instant access to your OneDrive for Business files

OneNote

 

 

• Notebooks are created for each Team

 

• Shared Notebook is hosted on SharePoint site

 

• Sections are created for each Channel

 

• Accessible from OneNote applications

Diagnostics In RAVE

As of today, we have following diagnostics available in Rave. These diagnostics are built to help engineers quickly troubleshoot scenario/issues where we have high volume of cases. The purpose is to help troubleshoot the issue faster and narrow down to actual problems.

Refer to the topics and description below.

[Asset 4553143]

Sample diagnostics

Teams Auto Attendant Not Receiving Calls
The Teams Auto Attendant Not Receiving Calls diagnostic will run through several checks related to the Auto Attendant and the Resource Account associated with the Auto Attendant, to identify issues with the setup that will cause a failure with calls to the Auto Attendant. The diagnostic will also provide direction on how to correct any issues that are found. The diagnostic currently performs 24 checks against the Auto Attendant and the associated Resource Account that is required by the Auto Attendant and will identify if any of those checks are returning issues. Action steps will be given if any check fails or if all checks succeed.

Teams Meeting Outlook Addin is Missing (Service Side checks)
We have a new diagnostic that checks the service side Teams upgrade mode and Meetings policy for an impacted user.  If the diagnostic succeeds, the user is in a state that *should* show the Teams meeting addin in Outlook.  If the diagnostic does not detect any problems, proceed to troubleshoot the local client.
https://dev.azure.com/Supportability/UC/_wiki/wikis/UC.wiki/290105/Diagnostic-Teams-meeting-add-in-is-missing
 
Note:  This diagnostic can ONLY check Service-side policy settings to confirm a given user is eligible for the Teams Meeting Outlook addin.  It cannot do any client side troubleshooting. For more information check out.

TSG: Teams Outlook Addin for Meetings is Missing

Teams Outlook Addin install should no longer require admin permissions on the client machine; regsvr32.exe should not be needed.
Review relevant Troubleshooting Guides (TSGs) in the wiki especially Teams meeting add-in Windows client side troubleshooting

Teams user is not able to sign-in.
This diagnostics will detect missing configuration on the customer's deployment to allow Teams usage: Teams user not being properly provisioned e.g. Teams license missing. The right next steps to troubleshoot the issue based on the specific Identity model like "Federated" (ADFS) or "Synchronized (DirSync)" or pure "Cloud" identity as seen by the service. In the spec here we highlight other scenarios we plan to cover in coming months as we see the current usage and areas where more deeper checks are needed. 


A Teams user is unable to communicate with a federated contact using Teams
his diagnostics will detect missing configuration on the customer's deployment to allow external communications, Teams user still homed on-premises for Skype for Business or user with external access policy not being in sync across the data stores (AD, AAD, BVD). Such misconfiguration(s) lead to Teams users not being able to communicate with federated contact using Teams.

Client Logs and Tools

Customer facing documentation for log collection:
https://docs.microsoft.com/en-us/microsoftteams/log-files

If the issue is reproducible, just occurred, or there is a threat of the issue reoccurring, please advise the customer that collecting log files from the time of the incident may expedite resolution of their support request. To collect log files please provide the following instructions to the customer: 

Web logs: best logs to start with; most Teams client activity stored here

To collect the logs:

  1. On a Windows machine, press CTRL + ALT + SHIFT +1 on the client to download the logs (file path: %downloads%\MSTeams Diagnostics Log <timestamp>.txt)
  2. On a Mac machine, press Command + Option + SHIFT + 1 on the client to download the logs (file path: Downloads\MSTeams Diagnostic Log <timestamp>.txt)

Desktop logs

Desktop logs contain:

  • Framework and bootstrapping info
  • App Bootstrap process
  • Plugin initialization
  • Update management
  • SSO/ADAL Sign-in


To collect the logs:

  1. On a Windows machine, navigate to %appdata%\Microsoft\Teams\log.txt
  2. On a Mac machine, navigate to ~Library/Application Support/Microsoft/Teams/logs.txt

Media Stack

To collect the logs:

On a Windows machine, navigate to: %appdata%\Microsoft\Teams\media-stack\*.*

On a Mac machine, navigate to: ~Library/Application Support/Microsoft/Teams/media-stack/*.*

For more information on the different types of log files used to troubleshoot Teams, click here.

How to Read Client Logs

Use any text editor to read the client logs. Note: Some content is in JSON format, so an app with JSON format extension is recommended. Some examples are: Sublime, Chrome, or Visual Studio Code

Troubleshooting Client HTTPS Connectivity

TEAMS use HTTPS protocol (Rest API). When troubleshooting https connectivity, use Fiddler (Windows) or Charles Proxy (Mac). You can filter by client process with Fiddler and you can use one of two views with Charles Proxy: structured or sequence. Both require an installed root certificate in the Trusted Root Certificate Store User or Computer from Fiddler/Charles and that https decryption is enabled.

For documentation on troubleshooting connectivity issues with the Teams client, click here.

Troubleshooting Client Media Traffic

To troubleshoot client media traffic, customers need to capture a network trace. Customers can use Netmon, Wireshark, or Message Analyzer. 

Troubleshooting Guest Access

Guest access can be enabled from four different locations but must include the setting in Azure Active Directory for it to work properly.  The four locations are: Teams, SharePoint, Office 365 Groups, and Azure Active Directory.

For instructions on how to turn on Guest Access, click here.

To Troubleshoot Guest Access, click here.

Troubleshooting a Missing Dial Pad: 

  • Make sure the user has been assigned a Teams license.
  • Make sure the user has a Calling Plan assigned.
  • Enable the users for Enterprise Voice.

Troubleshooting Teams Sign-in

  • First, make sure the Microsoft Teams service is healthy.
  • Then check for any common error codes and review Why am I having trouble signing in to Microsoft Teams? 
  • You may also need to review Identity models and authentication in Microsoft Teams.
