Overview

This TSG provides guidance on how to assist customers who have Azure in Open subscriptions.

Azure in Open (AIO) is an offer that allows customers to make monetary commitments through Microsoft Open Programs (Volume Licensing (VLSC) programs). It is available in unlimited quantities with additional credits available for purchase at any time. The offer activates with Online Service Activation (OSA) keys, which are valid for 12 months and are applicable to any Azure service that uses monetary commitment.

Policy

  • AIO keys are valid for five years if unredeemed. Once redeemed, they are valid for 12 months. After expiration or subscription cancelation, the unused credits disappear.
  • AIO accounts do not use a payment instrument (PI) (only tokens), so the rated usage only calculates for the current month.
  • The customer cannot view any details in the following columns in the usage reports downloaded from CMAT or the Account Portal: “List Price,” “Consumption Price,” “Consumption Discount,” and “Monetary Credit Amount”.
  • AIO is only available to OMS accounts, not MINT accounts.
  • SEs support AIO customers, not distributors. Distributors are a channel between Microsoft and the customer.
  • Support plans are not part of the AIO offer. Customers will need to purchase a support plan using a credit card; the offer does not accept reseller credits. You can find this information on the website at Get Started with Azure in Open Licensing under the “Getting Support” section.

Note:  AIO subscriptions that are moved to a different directory will not be able to add credits and will encounter the below error:

Something went wrong, Please try again.

Customers will need to move the subscription back to the default directory to be able to add credits from the Ibiza Portal.
 

Customer Experience and Provisioning

Notify the Customer/Partner of Low Balance

Customers are responsible for monitoring their subscription usage to help determine when to purchase more tokens. Monitoring usage is not always an easy task, so an Alerts feature was created to notify customers/partners of low balances.

Email notifications are automatically sent to the customer when the account balance reaches 30% of the initial purchase amount. If the customer wishes to monitor the balance closer or view balance consumption, they can create up to five additional custom alerts.

Alerts can also be set up at the subscription-level and configured to send a notification to two email addresses. During which, the customer can also enter the partner’s/reseller’s email address to receive an email alert about the remaining credit on the subscription. Alerts are accessible in the Usage and Billing portal under the Alerts tab.

Setting up alerts from the Azure Accounts Portal

Send customers the following steps to set up alerts:

  1. From the Accounts Portal, select the AIO subscription and click on the Alerts tab.
  2. Select Credit Remaining from the Alert For drop-down menu and insert a dollar amount in the Amount Remaining field.
  3. After you save the alert, you will get an email stating, “Your alert is enabled. If your credit remaining for Azure in Open drops below $XX, we'll notify you”. 

Note: The system sends a separate email for each of the OSA keys associated with the subscription. Receiving multiple emails on the same day is normal if you have multiple OSA keys linked to the subscription and billing alerts enabled.

Note: Customers receive their "low credit" Alert emails from billing-alerts-noreply@mail.windowsazure.com. If there are issues regarding it, engage ASE.

Viewing OSA Key Information

Under the Credit History tab, customers can view when an OSA key is set to expire. Keys are valid for 12 months and are applicable to any Azure service that uses monetary commitment.

Purchasing Third-Party Services

  • Azure Marketplace, StorSimple, Azure Support Plans, Active Directory Basic and Premium are prohibited while using Azure in Open SKUs.
  • In the Azure Marketplace, customers in Open licensing can transact all bring-your-own-license (BYOL) offerings and will need to associate a new subscription with a credit card for usage-based offerings
  • Customers can purchase some Azure services through Open Licensing as per-user licenses on an annual term, separately. Azure Active Directory Basic and Premium as well as Enterprise Mobility Suite (EMS), are available in Open Licensing and are examples of a per-user license purchased annually and separately. Some services are not available in all Datacenter locations. Azure Status displays the services offered in each Datacenter.
  • Customers can use AIO credits to deploy any VMs including third-party as long as it’s Bring Your Own License (BYOL); i.e., credits used for computer and infrastructure costs, but not for licensing.

Note: Customers can use their credits to pay for third-party services, but not at a discounted price.

  • Customers cannot use AIO credits for third-party services which are usage-based. Therefore, the customer needs to purchase a separate Pay-As-You-Go (PAYG) subscription in order to set up those services. In addition to this, AIO will only display marketplace items that are deployable.
    • Example 1: For third-party VM images like Red Hat, billed as a Linux VM + Red Hat image
      • The customer can use a BYOL version of the Red Hat image, and they will receive a bill for the Linux VM through their AIO credits.
    • Example 2: Oracle, which bills as a Windows Server VM + Oracle software
      • The customer can use the BYOL version of the Oracle software (through Oracle License Mobility Policy), and they will receive a bill for the Windows Server VM through their AIO credits.
    • Pre-requisites: The customer should have the licenses already purchased separately with the respective 3rd party service provider (Red Hat/Oracle) .

  • Customer purchases licensing seats. Always check if the user has available tokens in CMAT if a customer has a question about licenses or seats (tokens purchased from the Volume Licensing Team). Seats are equal to $100 per seat and redeemable with AIO subscriptions. Engage your TA, and they will work with VL to resolve the issue if you don't see any available AIO tokens.

Support Scenarios

Resend activation email

After a Distributor processes a customer’s purchase request, the customer will receive an automated email with the following details:

  • How to activate Azure in Open.
  • Purchases and the associated quantities.

If a customer needs the email sent again:

  1. Suggest the customer to check their junk mail folder in case the email sent as spam.
  2. If the email is not in their junk mail folder, redirect the customer back to his or her distributor.  

Note: If the customer is still unable to receive the email, engage your TA to get the email resent. For guidance on escalating internally to the One VL TA, reference TSG 2929022.

Customer is unable to find OSA key in Volume Licensing Service Center (VLSC)

Customers receive a one-time use OSA Key when signing up for AIO. These keys cannot be used on multiple subscriptions or accounts. Once used, the status of the OSA Key changes to "used". You can double-check the status of an OSA key in CMAT.

If customers are unable to locate their 5x5 OSA key in the VLSC portal, redirect them to their Distributor. The Distributor will also be able to assess any initial OSA key issues that occur when the customer first signs up.

OSA key does not work

In some cases, a customer’s 5x5 OSA key may not work properly during sign up or top off. The following issues may have occurred:

  • The distributor provided the customer with the wrong key.

  • A system issue prevented Azure from working properly with the Electronic Communications Information Technology (ECIT) token system.

In these instances, you should do the following to assist the customer:

  1. Verify the key status in CMAT by clicking Quick Access then Lookup Prepaid Card.
  2. If the key is valid and not yet redeemed, escalate to your TA to generate a new key for the customer.

Note: Because VLSC Support does not offer 24x7 support, the customer may need to wait until VLSC Support can respond. ASMS should proactively follow-up with VLSC Support on behalf of the customer. See the “Engagement and Escalation Path” section of this TSG for guidance on contacting VLSC.

  1. If the key is valid, but already redeemed, redirect the customer to their Distributor. There are two ways to determine if a key was redeemed:
    1. Use the Lookup Token feature in CMAT. This will show the date the token was redeemed, and the VL Tenant globally unique identifier (GUID) of the entity that redeemed the token. You can then use the VL Tenant GUID to find more information about the redeeming entity by switching CMAT to the O365 view, clicking VL Information, then clicking Tenant GUID, then entering the GUID in the textbox and clicking Search.
    2. View Monetary Credit Top-Off History and see if the customer already used that OSA key.
      • If the key is invalid, then redirect the customer back to their Distributor.

Note: For cases where we redirect the customer back to the distributor for OSA key issues, if the customer urgently needs a key and cannot wait for the distributor to respond, then engage your TA who will work with the Visual Licensing Team to evaluate the situation and provide a replacement OSA key. Be sure to inform the customer that VLSC does not provide support 24/7 so there will be some wait time.

Customer wants to extend the OSA key expiration date

If a customer wants to extend the OSA key expiration date, please inform them of the following:

  • OSA keys are valid for 12 months and expire after this timeframe.
  • Any unused balance is forfeited.

Customers cannot extend the expiration of an OSA key, so they should carefully forecast usage when purchasing keys to minimize the risk of forfeited funds.

Customer cannot login to VL portal to purchase/view OSA key

A customer might experience difficulty logging into the VL portal when trying to purchase or view an OSA key (e.g., for topping off the subscription). Engage the One VL TA. They will evaluate the situation and determine how to assist the customer.

Monetary credit amount contradicts the amount purchased

In some situations, a customer may find that the monetary credit applied to a subscription does not correspond to the amount the customer intended to purchase. Reasons for this mismatch may include a human error (e.g., the distributor mistakenly provided an incorrect OSA key) or a system bug.

If this scenario occurs do the following:

  1. Ask the customer to verify the amount documented in the customer’s purchase confirmation email.
  2. Verify the key status in CMAT.
  3. If the key is valid and shows the incorrect intended amount in CMAT, engage your TA, who can evaluate the situation and remedy it.
  4. If the key is invalid, redirect the customer to their distributor.

Note: In cases where the redirected customer urgently needs the OSA key and cannot wait for the distributor to respond, engage your TA, who can evaluate the situation and attempt to expedite a solution.

Monetary credit amount contradicts currency exchange rates

Microsoft Azure is available in 89 countries; however, we currently support approximately 25 global currencies. Some customers credit amounts might not display in the currency in which they made the purchase. For example, a customer in the United Arab Emirates purchases 1000 AED worth of monetary credit, but the amount displays in USD (approximately $272) since Azure does not directly support AED.

The number of supported currencies is expanding; reference the Azure Purchase FAQ for a current list of supported currencies. Help the customer understand how the foreign exchange rate applies to their situation if a customer contacts Azure Support in this scenario.

Customer runs out of monetary credit and needs help with topping off

When a customer’s monetary credit runs out or is running low, the customer may need assistance with “topping off” the subscription.  In these instances, there are a few options to consider for assisting the customer:

  1. Have the customer contact their distributor for a new OSA key to top off the subscription.
    Note: This may take several days but is recommended because discounts might apply.
  2. Have the customer add a credit card to their account to top off immediately.

4015634

  1. You can issue a one-time courtesy $100 USD token (use the PID tool) for top-offs for customers that can't add a credit card (it's a small list of customers from Azerbaijan, Belarus, Brazil, Canada, Japan, Kazakhstan, Korea, Russia, Taiwan, and Ukraine). Customers in all other countries that support AIO should be pushed back to add a credit card and top off themselves.
  2. Work with your TA, who can issue a CPE credit in CMAT using the Misc. Credit feature (not a key).

Note: Some customers may request to see a listing of top offs applied to a subscription during a specified timeframe. You can direct the customer to the Azure Account Portal to retrieve top off information. This information is also readily available in CMAT, to find it click the clock icon which appears next to the Remaining Monetary Credit, and then review it with the customer. 

Note: AIO top-off purchases are documented as receipts without the customer's VAT ID.

Credit/Refund Request

Credits and refunds for Azure in Open are identical to what’s in place for all other offers, therefore engineers have permission to issue credits as per the standard policy:

  1. Engage your TA to either generate a Product ID (PID) key for the customer to add or apply the AIO credits via CMAT.

Note: Applying an AIO credit via CMAT is the same as processing a credit to a standard Microsoft Online (MOSP) subscription and follows the same ASMS/BG empowerment rules. For guidance on handling credit requests and refunds, reference TSG 2646823.

  1. For time-critical AIO credits (such as "top-off" credits where a customer has approached their distributor for a new key but are at risk for subscriptions shutdown before the distributor will be able to provide them the new token), do the following:
    1. Instruct the customer to add a credit card and use the credit card top-off feature (see the AIO credit Top Offs section of this TSG).
    2. You can assist on top-offs if the customer is located in a country that does not support credit-card AIO top-off using the same process as above.
    3. If you need to engage your TA, the TA will generate a new key for the user if necessary using TSG 3188143.

Note: Keep in mind the key generation tool is tracked very heavily, so do not misuse.

In some cases, a customer might not be able to reach their distributor for a key and therefore proceeded to purchase monetary credits using their credit card. Afterward, the customer contacts the distributor and receives a key, and wants a refund for the credit card purchase.

Refunds in this scenario are discouraged but may be possible. It’s discouraged because Microsoft prefers sales through distributors as the method of purchase. A best practice is to recommend that customers not purchase more on the credit card than they are likely to consume in the short term (i.e., when they can get a key from their distributor). If a credit card refund is deemed appropriate, follow the standard credit process outlined in TSG 2646823.

Customers may ask for a refund for AIO tokens that they have purchased. If you receive such a case, advise them to engage the party that they purchased the AIO tokens from.

Note: Microsoft cannot process refunds for AIO tokens.

Customer wants to know how much they spent during previous months

For Azure in Open, rated usage only displays for the current month. Once the current month closes, the customer can only see the usage for a service without a rating. The customer cannot reconcile their usage to the remaining credits. The Universal Store Team (UST) does not store rated usage for closed months in any of their systems. In the example below, you can see the past billing statements show “0” rated usage.

If the customer experienced a Service Interruption Event (SIE) and credit is needed, you will need to reach out to AEO to access the previous months’ bills, or the customer can self-serve using ACM.

AiO rating is now self serve via Azure Cost Management (ACM)

If you need to get the users usage rated, you can reach out to Azure Enterprise Operations (AEO) and they can run it for you. 

Customer needs a VLSC portal purchase invoice (This is for purchases made via a reseller only, not via Microsoft Credit Card (CC) top off)

When customers make a purchase on the VLSC portal, they will not receive an invoice automatically. Some customers will wish to have an invoice, and contact support for assistance.

If the customer purchased through a Distributor, then direct the customer back to the Distributor. Otherwise, we are unable to provide an invoice for the customer.

Issues with the PDF receipt (this is for purchases made via CC directly from Microsoft only)

Opening the PDF requires a password, which is the customer’s Subscription GUID, as displayed in the portal in all lower-case letters. If the password fails, send a [Asset 4498193] to the “Azure in Open Escalations” queue:

  • The Customer doesn’t receive an email with their PDF attached or wants it re-sent:
    • Send a [Asset 4498193] to the Azure in Open Escalations queue for manual receipt/invoice generation requests.
  • The customer has a dispute on the receipt itself, such as tax amount, incorrect customer information, etc.:
    • Send a [Asset 4498193] to the Azure in Open Escalations queue if you are unable to solve the problem.

Customer wants to transfer a balance to a new offer

Customers may have multiple subscriptions and may want to transfer monetary credits from one subscription to another. Unfortunately, we cannot fulfill this request due to system limitations. Use the standard processes to evaluate and issue a credit/refund, if applicable. Refer to TSG 2646823.

Customer wants to transfer balance from AIO to EA

If a customer wants to transfer an AIO balance to an EA subscription, file a ticket to the EA team, using Credit Scenario 37. For more guidance, reference TSG 4094770.

Customer applied AIO key to wrong AIO subscription

If you need to suspend an account in this scenario, follow the steps below:

  1. Before suspending the "wrong" subscription, take a screenshot of CMAT showing the pending credits which the TA can reference.
  2. Have the customer cancel the "wrong" subscription they used to apply the token.
  3. Engage the TA to apply a credit to the "right" subscriptions up to your empowerment.

Note: you will need approval for anything over your empowerment per standard process.  

  1. When you engage your TA for credits, work with the customer to get the “wrong” subscription de-provisioned at the same time. This is to ensure they don’t submit another service request to re-enable it and use the credits.

Customer purchases AIO token from the reseller and incorrectly spins up a new AIO subscription

Customers may contact ASMS to transfer the credits from the newly created AIO subscription to the pre-existing AIO subscription (since the production deployments reside under the pre-existing subscription).

In these cases, we will guide the customer to cancel the newly created AIO subscription after verifying the remaining credits and work with the TA to apply the remaining credits on the correct AIO subscription. Follow the steps below to complete this action:

  1. Once the TA applies the credits to the correct AIO subscription, you must deprovision the canceled AIO subscription.
  2. Inform the customer you will be deprovisioning the incorrectly created AIO subscription.
  3. Follow the standard process as outlined in TSG 2646764 to engage your TA to properly deprovision a subscription.

Note: The Suspend - Immediate Cancel action in CMAT can also be used to cancel the subscription. The customer will not have to wait seven days to sign up for a new subscription. Upon performing the Suspend - Immediate Cancel action, add the following note in your case so the subscription is not re-enabled in the future:

DO NOT re-enable this subscription <Sub GUID>. Cancellation was performed as the customer had activated the subscription incorreclty, and a credit was applied back to the actual subscription.

Subscription Ownership Transfer (SOT, currently not supported by the Azure platform)

Customers may approach ASMS to move their AIO subscription from one Azure account to another Azure account. Currently, the SOT process of AIO subscriptions is not a support option under the Azure platform.

One potential workaround is for the customer to create a new, empty AIO subscription in their desired destination account and then the customer and SE to work together to perform Data Migration from the source subscription to the destination subscription. For guidance on how to perform data migration, reference TSG 2856526 for simple migrations and TSG 2892044 for forced migrations.

Subscription transfer from a MINT account

If the customer already has a MINT account associated with their ID, they can’t use that ID to obtain an AIO subscription . In these instances, we will need to do one of the following:

  • Backfill the MINT account to become an OMS account.
  • Suggest that the customer use a new ID and create an OMS account.

ART disables the customer’s subscription  

In some situations, the Account Research Team (ART) may disable a customer’s subscription after discovering suspicious activity. In these instances, engage ART for assistance using the guidance in TSG 2658538.  

Note: Azure in Open customers pre-pay for services, and their monetary credit remains valid during a dispute. If ART approves the customer’s account, the subscription reenables, and the customer can continue using their monetary credit.

Customer sees different data on different Azure portals

Some customers might see different monetary credit information on the account.windowsazure.com portal vs. the portal.azure.com portal.

You can verify the monetary credit status in CMAT. However, if you too notice different data than what the customer reported, engage PayOps for further evaluation per the standard process. Reference TSG 2651579 for guidance.

Engagement and Escalation Path

For Azure in Open scenarios that require additional assistance regarding pricing, OSA keys, top offs, or billing issues, you can engage the VLSC Support team directly in addition to engaging your TA:

  • Reference the VLSC Contact Us page to locate the support contact in your region, or
  • You can send a [Asset 4498193] to the Azure in Open Escalations queue for assistance.

The 4 core scenarios for AIO Escalation queue to respond to are:

  • Customer did not receive their top-up invoice or are requesting previous invoices
    • Same topic is where customer’s are unable to unlock their invoices using the subscription GUID and an unlocked invoice is being requested

      • For this scenario, the core data required for any escalation task include:
        SubGUID
        Acct Owner ID:
        Date of top-up purchase(s) customer is seeking the invoices on

  • Customer activated token (credits) on incorrect account or subscription
    • For this scenario, the core data required includes:
      Acct and SubGUID credits were redeemed against
      Date of credit activation
      Remaining balance of credit
      Desired Acct and Subscription to align the credits to

NOTE: DO NOT cancel the current subscription until permission to transfer the credits is given. There are scenarios in which this request would be denied and therefore approval is required before any sub is cancelled

NOTE 2: Upon receiving approval, current subscription must be cancelled to prevent the double the amount of credits from being consumed. In this scenario we are issuing a new token against the desired acct/sub which means the original token needs to be cancelled. This can only be done via suspension of the subscription.

NOTE 3: If customer is unwilling to cancel the subscription, then credits cannot be transferred as there is no way to reduce the credit value manually

  • Customer is requesting a detailed usage report

    • This scenario is already documented. This is an ASMS task to download the usage from CMAT.

      Note: If you need AIO usage rated, work with the AEO team.

  • Customer is requesting a credit for impacted service (i.e. system bug)

    • For this scenario, the core data required includes:
      Acct Owner ID
      SubGUID
      IcM / Technical ticket that investigated the issue – must include Root Cause Analysis (RCA) findings
      Justification for credit
      Credit policy the issue falls under
      Date Range of Impacted Service
      Impacted Service
      Value of refund with calculation file:
      Calc file should be isolated down to date range and for specific service only                             Calc file should include all raw data for usage but be pivoted/filtered by the above criteria

For all other scenarios, follow standard TA escalation paths.

Additional Billing Alert Notification Information

By default, low account balance notifications are not sent to the Account Administrator.

As part of the AIO signup process, the email entered in the “Partner Contact Email” field is the only email address which receives low subscription balance notifications.

  • The customer can add additional email IDs via the Accounts Portal / Alerts tabs.
    • These emails are notified of credit remaining and expiry.
    • Each alert can be sent to two email addresses.
    • These alerts are created at the subscription level.

Azure in Open Billing alerts include:

  • Token Credit Remaining: Remaining token balance is less than or equal to 30% of the initial token balance.
  • Token Expiry: 30 days until token expiration date. After the expiration date, any unused credit will be lost.
  • Subscription Credit Remaining: Total subscription credits remaining is less than or equal to the configured amount.

More information can be found in the FAQs at the bottom of the Get Started with Azure in Open Licensing page.
