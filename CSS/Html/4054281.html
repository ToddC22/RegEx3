Overview

Reserved Instance (RI) is a service that allows users to pre-pay for long-term virtual machines (VMs), SQL compute, or software, providing both a significant discount on compute costs as well as prioritized capacity when the scope selection is a single subscription.

Policy

Region Specific

  • Reserved Instance for Pay-As-You-Go (PAYG) is not available in the following countries​​​​​​: Brazil, India, China, Taiwan, Russia, Korea, Argentina, Hong Kong, Indonesia,  Liechtenstein,  Malaysia, Mexico, Saudi Arabia, South Africa, Turkey. 
  • Region enablement and quota request should not be sent to Capacity Management if they are in the list of countries not available. 
  • If the SKU is not in the drop down, then tell the customer that RIs are not supported for that region/SKU combination
  • Regions will be available to the customer based on availability. All datacenters may not be available for this service.
  • If capacity is unavailable for a region, customers will not be able to purchase RI in that region.
  • Fairfax customers who want to purchase RI for DoD (Department of Defense) regions must first be approved and whitelisted for DoD regions.
    • If a customer has already been approved and whitelisted they may proceed with the purchase.
    • If the customer needs to be whitelisted for DoD, please follow the guidelines in TSG 3005247. 

Offer Specific Restrictions

  • Services can be purchased only using Fairfax, EA, CSP, and Pay-As-You-Go subscriptions.
  • Certain services can be purchased using monthly payments. See the documentation for more information.
    • Customers who want to change from monthly to upfront payments can do so by using the exchange feature. It is not a good idea for them to cancel and and repurchase, as this action will impact their resources.  

      Note: Customers paying month-to-month may experience fluctuations in their charges. This is due to changing exchange rates, which are dependent on customers' currencies.
  • RI benefits can only be applied to VMs under Fairfax, EA, CSP, Azure Plan and Pay-As-You-Go subscriptions.
    • Any VM type that is not listed at the time of purchase is not available for RI. 

      Note: A series and G series VMs are not offered for RI. You should not submit an IcM to engineering for these scenarios. 
  • For CSP subscriptions, reservation purchases can be enabled in the Partner Center and the Azure Portal.
  • The Customer Permissions feature for CSP Partners allows you to give customers more self-service options with Azure resources. By turning on permissions for the customer, you can allow them to buy their own resources, such as Azure reservations.
    • More information on this feature can be found in the public-facing documentation.
  • CSP administrators are not allowed to transfer RIs between partners. If the destination Partner wants to purchase RI they will need to do so on behalf of their end user via the Partner Center portal.
  • For EA subscriptions, reservation purchases can be disabled from the EA Portal, by default the reserved instances are set to on.
  • For EA customers the scope must be under "Shared" for it to apply to the enrollment. This is very similar to how EA subscriptions enable Marketplace purchases.
  • Reserved Instances are not available for the MPSA EA Enrollment type. You can identify a MPSA enrollment by the enrollment number format (xxxxx_yyyy).
  • CSP customers can't transfer a RI from a PAYG subscription to a CSP subscription. They can cancel the existing RI and repurchase it on the CSP.
  • If a customer is a member of a management group and inherits Owner access by being part of the group, it does not allow the customer to add new Reserved Instances. The customer must be an owner on the actual subscription, in order to purchase an RI using that subscription. 
  • The HBv2 series of VMs can be purchased as an RI reservation for five years.
    • The five-year term offer is exclusive to the HBv2 series. No other series of VMs supports this term limit currently.
    • Deny requests for five-year RI reservations for other SKUs, and refrain from escalating these requests to Engineering.

Note: When a subscription is selected to purchase Azure Reserved Instance, the subscription’s Friendly Name is shown, not the Offer Name. For example, if a customer changes the friendly name of their Microsoft Azure Enterprise subscription to “My subscription X,” then when this subscription is selected to purchase an Azure RI, “My subscription X” is displayed, not “Microsoft Azure Enterprise.”

  • For Cosmos DB customers, please follow the process documented in TSG 2995104.
  • SUSE is not available for Fairfax customers. 
  • Cloud Partner Portal (CPP) Subscriptions
    • The difference between RIs and CPP is that RIs are based on the VM itself, whereas CPP is based on hours. Each CPP unit provides customers 744 hours/month that could be spread across multiple virtual machines that were the same instance, operating system, and in the same datacenter. RIs are based on the instance itself and does not take the OS into consideration.
    • RIs in EA programs can be paid for using Azure Monetary Commitment whereas CPP was an Azure Plan offering requiring a separate invoice. When the customer makes a purchase for Azure RIs, the amount will be deducted from their Monetary Commitment in that given month.
    • There is no automated migration of CPP to RI. Customers with CPP hours will be asked to complete the remaining months of their CPP term. Once completed, the customer can choose to purchase RIs or pay their applicable hourly rate for continued use of that VM.
    • Customers with CPP licenses can purchase RIs. In cases where a customer has both a CPP subscription and an RI for the same machine, the RI will be used first. For any usage/instances beyond the quantity of RIs the customer has available, that usage will then deduct from the customers CPP hours available in that given month. Only after RI and CPP have been used will we then charge the customer their hourly list rate. We should advise customers not to buy RIs for VMs that they currently have CPP hours for unless they are fully utilizing their CPP hours and need to deploy additional virtual machines.
    • CPP expires on volume licensing pricelists on December 31st, 2017.

Understand Reservations Lifecycles

  • Purchasing a reservation provides a discount on the matching resource for a duration of 1 or 3 years.
  • Customers don't need to assign the reservation to a resource (VM or a SQL Database). Matching resources that are already running or ones that are newly deployed, will automatically get the benefit.
  • If a customer does not have enough quota (regional core quota, specific VM size quota, or SQL quota) during the time of reservation, they will need to open a new service request to have the quota increased. See https://docs.microsoft.com/azure/azuresubscription-service-limits for more information. 
  • Post purchase, the reservation discount is automatically applied to the matching resource.
  • At the end of 1 or 3 years, the reservation will expire. Unless the customer has auto-renew turned on.
    • If auto-renew is turned on you should note the following:
      • Public facing guidance about this feature can be found here.
      • Customers will receive an email before the RI expires. This email will also include the price in which the customer will be locked in. How this email is sent depends on the type of channel customer. If the customer is a web direct (WD) customer the email will be sent to the email used during purchase. If the customer is in an EA, the Enrollment Admin and selected notification users will receive the email.
      • For CSP customers, the current design does not include any upcoming expiration notifications. The CSP partner nor the customer will receive an email.
      • You cannot auto-renew a reservation after it has expired.
      • During the renewal process you will be able to update the term.
      • There is a limitation today where if you attempt to auto renew a reservation within one day (24 hours) of the RI expiring, it will fail with an obscure error (Example: "Another action in progress"). If the customer receives this message, ask them to repurchase the sku, following the normal purchase process. There is also a link in the Details blade that will direct them into the “create experience” where they can re-purchase an expired RI.

        Note: This is a known issue and engineering is working to fix. If and only if you have informed the customer of how the feature works and they are still unable to re-purchase should you engage engineering. In that case, you can submit a Sev 3 IcM only.
    • If auto-renew is not turned on note the following:
      • The resources (VMs, SQL, etc.) will continue to operate the way they do and will be charged on-demand once the reservation has expired.

Refunds, Exchanges, Charges, and Invoices

Exchanges: The current mechanism for exchange is that we cancel the earlier RI (and credit the amount) and have the customer purchase the new RI of the required type. It doesn’t get exchanged automatically.

  • Customers can self-serve exchange a reservation via the Azure portal.
  1. Select the reservations you want to refund and click Exchange.
  2. Select the SKU you want to purchase and provide the quantity. Make sure that the new purchase total is more than the return total. Determine the right size before you purchase.
  3. Review and complete the transaction.

Note: Self-service exchange errors may occur under the following circumstances:

  • Self-serve exchanges must be equal or greater in monetary value at the time of purchase.
  • Customers will get an error if they mix different types of RIs, e.g., VM and SQL.
  • Customers will get an error if the new purchase is less than the return.
  • It is only possible to Exchange multiple RIs for 1 new RI in one transaction.
  • It is not possible to Exchange multiple RIs for new multiple RIs in the same transaction.
  • To exchange reservations purchased for VM sizes that do not support premium storage (F1) to the corresponding VM size that does support premium storage (F1s), go to reservation details and click Exchange. 

Note: Such an exchange will not reset the term or lead to a new transaction.

  • When a PAYG account with RI is added to an EA (not SOT), and the customer is looking for a refund under the CC but the RI charge is under the EA commitment, the only option is to return and repurchase under the EA where the prorated charges apply. If the PAYG account with RI is added to an EA (not SOT) but not looking for a refund under the CC, then they can continue using RI under EA without any issues.
  • RI invoices can only be retrieved via the CST tool. By expanding the invoice section within CST and clicking the indicated icon, you will be provided the option to select Download Invoice. 
  • A customer can download their own RI invoice by going to the Ibiza portal navigating to Cost Management and selecting invoices.
  • It is possible for engineers to process a partial exchange (i.e. Return 8 quantities out of 10)
  • All RI cancelations need customer approval before engaging the EA team.
  • There is a return/cancellation cap of $50,000 USD (or equivalent in local currency) per customer per year (not calendar year).
    • If the cap is exceeded, the user will get an error. You should never proactively offer  to provide an exception to this rule. Instead, explain how the limit is enforced, and ensure that the customer is aware of and following the policy. If, after explaining this, the customer remains dissatisfied, do the following: 
      1. Reach out to AEO for assistance via an SDC collaboration task.
      2. Provide a valid reason why AEO should consider an exception to the existing policy.
      3. If AEO determines that an exception should be considered (for example, due to a duplicate purchase, system error, request within 5 days without utilization, etc.), clearly document that reason and include it in the task to avoid back-and-forth communication or denial of the request.

Note:  As a goodwill courtesy, the RI Business Group (BG) is currently waiving the 12% early termination fee (ETF) listed for RI cancelations. The RI ETF waiver is in place until further notice, however the overall policy is still in place and may be activated by the RI BG in the future.

Scoping Limitations

There are three scope options for reservations: shared scope, single scope and single resource group scope. Shared scope applies the reservation to entire enrollment for EA customers and to the entire account for credit card customers. Single scope applies the reservation to a single subscription provided by the customer. Single resource group scope applies the reservation discount to the matching resources in the selected resource group only.

Priority 

When applying Reservation discounts, the Resource Group discount is applied first, then single subscription Reservation, and then shared scope Reservations. So if there are any RIs at Resource Group scope for a VM, then that is applied first. 

If there are any subscription level RIs already applied to VM's within a Resource Group, upon purchasing Resource Group scope RI's, those will automatically come off the VMs and let the resource group RIs apply when the new Resource Group RIs are sufficient for the VMs' usage in the Resource Groups.

Virtual Machines Limitations - How to troubleshoot RI benefit that are not being consumed: 

Note 1: VMs deployments with premium storage use the same meters as VMs deployed without premium storage but the reservation SKUs for these VMs are different. This leads to issues where customers look at the meter being used and then purchase the SKU name indicated by the meter, in such scenarios the customer does not get reservation discount.

  • Example: DS1 VM use the same meter as D1 VM but these two types of VMs have different reservation SKUs. Customers look at their usage file and think that they used D1 instead of DS1 and purchase reservations for D1. Reservation purchased for D1 does not apply the discount on DS1, and vice versa. How can the customer get the right VM size: https://docs.microsoft.com/en-us/azure/virtual-machines/windows/prepay-reserved-vm-instances#determine-the-right-vm-size-before-purchase

Note 2: A-series, Av2-series, and G-series are not covered for RI benefits. You should not engage Engineering via IcM for these VMs not being available to the customer. 

Note 3: Promo VMs do not get reservation discounts.

Note 4: Reservation discount is applied only to the region for which the reservation is purchased

Note 5: Within the size series group, the number of VMs the reservation discount applies depends on the VM size selected when purchasing a reservation. This will also depend on the size of the VMs that are running. The ratio column listed here compares the relative footprint for each VM size in a group of series. Use the ratio value to calculate how the reservation discount applies to the VMs running.

Example #1: One hour of RI purchased for the D32 VM equals to 16 hours of the D2 VM if the ISF is on and can be applied to both VMs.  A usage report example for 2 VMs (D32 and D2) running for 24 hours:

  • D32 = 23.5hours of RI allocation showing charges for 0.5hour for the same VM. So, 0.5*16 = 8
  • D2 = 8hours of RI allocation

Example #2: If a customer has a RI for E16s VM with a ratio of 8 and the flexibility feature turned on, and then the customer decides to change from the E16s VM to E20s VM, as shown on the table.

The RI will also apply to that E20s VM but, with a ratio of 10. Having said this, the RI will only apply to 80% of the costs of that new VM (asuming that is the only matching VM with RI the customer has on their environment), and the 20% remaining compute hours will be charged at the list price of that VM.

By adding an RI with a ratio of 2 would cover the rest of the machines. Multiple RIs can be applied to a single VM on the assumption there is only one E16s on the environment, therefore it would cover all machines. When the flexibility is turned on, the RI will be applied to all matching VMs randomly. The customer cannot decide which VM the RI will be applied to and at what percentage.

  • Capacity Priority: This reserves the data center capacity for your deployments, offering additional confidence in your ability to launch the VM instances when you need them. Capacity priority is only available when reservation scope is Single subscription.

Exchange and Return 

RI Exchange is a two step process. The first step led by support cancellation and the second step consisting of the customer re-purchasing the required reservation.

  • Customers can return multiple existing reservations and purchase a new reservation of the same type. For example, a VM reservation cannot be returned to purchase a SQL reservation. 
  • RIs do not get exchanged automatically. It is the customer's responsibility to immediately re-purchase. An exchange is processed as a refund and re-purchased – separate transactions for the cancelation and the new purchase are created. The prorated reservation amount is refunded for the reservations that was traded in. 
  • There is no 12% penalty when cancellation is done with intent to exchange.
  • Only reservation owners can process an exchange. Learn how to add or change users who can manage a reservation.
  •  The new reservation purchased as part of an exchange will receive a new term starting from the time of exchange.
  •  The reservation property such as size, region, quantity, and term can be changed.
  •  The new purchase total should equal or be greater than the returned amount.
  •  There is no penalty or annual limits for exchanges.
  • RI refund approvals are strictly managed by the Azure EA Portal Support team. Offering a refund on an Azure subscription is independent to RI refunds and ASMS empowerment on Azure sub does not change.
  • If the customer is requesting a refund or exchange, the EA team should be engaged via a collaboration task. Follow the guidance outlined in the Engagement and Escalation Path section below. 
  • If a customer requests a return/refund, the refund goes back to the previous term, causing the customer to have a positive value (Monetary Commitment), which will result in a forfeit of funds. These funds are not eligible for transfer to the current term. In these instances please note the following: 
    • Before Azure Operations will process the return, ASMS must confirm with the customer that they understand and acknowledge the refund process and are okay with forfeiting their funds. 
    • Once the customer confirms that they would like to move forward, proceed with engaging the EA team following the guidance outlined in the Engagement and Escalation Path section below.

      Note: For more information, see Self-service exchanges and refunds for Azure Reservations.
  • If a customer has multiple exchanges/returns under a single SR, only one collaboration task is needed. 

When a ticket is created for an exchange, inform the customer of the best aligned exchange policy below and set expectations on how long the refund will take. When a customer requests an exchange, they should immediately repurchase so that there are no benefit gaps.

How the Exchange and Return transactions are processed: 

Note: In all instance below the overall process could take 7-10 business days. Support will need to cancel the existing reservation and consider the service request created date to calculate the refund.

First, Microsoft cancels the existing reservation and refunds the pro-rated amount for that reservation. If there is an exchange, the new purchase is processed. Next, Microsoft processes refunds using one of the following methods, depending on the account type and payment method:

  1. Refund processing for Enterprise Agreement customers:

If the original purchase was made using a monetary commitment, then the money is added back to the monetary commitment. Any overage invoices since the original purchase are re-opened and re-rated to make sure that the monetary commitment is used. If the original purchase was made using overage, a credit memo is issued.

  1. Refund processing for CSP customers and Web-Direct Channel customers with invoice payment method:

The original reservation invoice is canceled and a new invoice is created for the refund. For exchange the new invoice has both the refund and the new purchase. The refund amount is adjusted against the purchase. If a reservation is only refunded, then the prorated amount stays with Microsoft and it is adjusted against a future reservation purchase. To receive a check for the refund, open a support request.

Global Financial Services (GFS) follows the same process with check/wire refunds for all Web Direct and CSP customers, no matter the product. In these scenarios, engage the GFS team and provide justification of the check request using the guidelines outlined in the TSG 2647567. 

  1. Refund processing for Pay-As-You-Go customers using a credit card for the original purchase:
  • The original invoice is canceled and a new invoice is created.
  • The money is refunded to the credit card used on the original purchase.

  • Warning: This transfer can be easily done in CST without the need to create the JSON. Follow the above steps for the RI Ownership Transfer Process. If your Destination Tenant is in Modern, use GTM Transfer. If your Destination Tenant is in Legacy, use Non-GTM Transfer.
