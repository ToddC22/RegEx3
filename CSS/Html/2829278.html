Summary

Who should follow this procedure?
  • Frontline support engineers (SEs).
What is this procedure about?
  • This procedure describes how to use the case ticketing system to determine whether a caller or a web-submitted case is entitled to support.
  • The purpose of entitlement is to:
    1. Protect customer information.
    2. Preclude unauthorized persons from receiving support.
    3. Prevent “social engineering” attempts to trick you into revealing confidential information.
When should this procedure be followed?
  • When opening a completely new case received from a caller directly.
  • When a caller contacts Frontline SE on a direct line for a new or existing case -- instead of calling Commercial Customer Service technical routers (CCS TR) and being transferred to an engineer.
  • When handling cases with specific exceptions that require Frontline entitlement as described in 2755056 Procedure: M365: Supporting cases transferred from tech routers.
  • Note: When calling back a customer on an existing entitled case, confirm you are speaking to the correct admin handling the case, as described in Supplemental notes in this article.
  • When handling cases that are related to SFBO Customer Proprietary Network Information (CPNI)/ Public Switched Telephone Network (PSTN) PIN for both web and phone (regardless if it’s from Tech Router).

Assumptions 

  • Entitlement begins with an anonymous caller requesting support. Do not assume the caller is authorized until you complete the entitlement process below.
  • The current ticketing system for M365 is Rave, majority of the case entitlement information can be confirmed in Rave. 
  • Status of the subscription does not affect entitlement process.
  • Most phone cases routed to Frontline SEs are already entitled by CCS TR and do not require Frontline entitlement, except as described in 2755056 Procedure: M365: Supporting cases transferred from tech routers.
  • A support engineer should still perform the same entitlement process for all cases transferred from Service Desk to ensure support is engaging the appropriate administrators.
    • If the customer is an admin:
      • And the case has correct tenant info, proceed in supporting the customer.
      • And the case does not have the correct tenant info, close the current case, either:
        • Create a new case on behalf of the customer on the customer tenant.
        • If you do not have the appropriate permissions level to create cases, engage your SME/TL/TM to perform above action.
    • If the customer is not an admin:
      • Redirect the customer to their admin and close the case.

Procedure

Entitlement requires you to confirm that callers are eligible and qualified to receive Frontline SE support as indicated by customer records in the support ticketing system.

There are two ways for customers to request support:

Web cases: when customer creates a web support request through logging into the M365 Admin Portal, there are three potential modality options: Phone, Email or Chat. When working on a web case, support entitlement should focus on verifying if the case contact info matches the admin’s record on file.

IVR cases: when a caller calls into support, an IVR phone case will be created in Rave.

  • If Rave recognizes the phone number, a corresponding customer profile will show up in the case. Support entitlement can follow the web case entitlement, focus on verifying if the case contact info matches the admin’s record on file.

  • If Rave does not recognize the incoming phone number, no tenant info is available. In this scenario, when support gets to below screen, ask about the Company details, select an Admin contact based on the conversation with the caller. If the Admin contact, first and last name in Rave match caller’s info, associate this tenant with the case. After case association, support will gain full tenant access in Rave, re-confirm case entitlement by going through the complete entitlement checklist in the article. If the caller is not able to pass entitlement, follow the procedures described in Troubleshooting entitlement including use LogMeIn. If all these actions fail, support should ask caller to work with an admin within the company to create a request through M365 Admin Center.

Detailed entitlement procedures

Use this section if you need to refer to details about using specific tools or if you’re having trouble with verifying the required entitlement information listed above. 

To verify admin info:

  1. Verify that the requestor matches the admin role in ‘Customer Profile’ tab in Rave. Click the admin’s name to see more details such as alternate email address, or phone.
  2. The nature of the service request needs to match the admin type. If a billing admin asks to make changes in Exchange, contact company’s Global Admin to receive written confirmation before fulfilling the request.   
  3. Complete Entitlement checklist and enter confirmation details in case notes.

To verify subscription type – Trial or Paid & SKU

In Rave, when you navigate to Company profile, there is a second tab about subscription where you can find details about subscriptions.

To verify CPNI/PSTN PIN-related request:

If a service request is related to SFBO CPNI/PSTN PIN, such as the service request has “PSTN” in the display name, refer to 3122455: Authenticating admin for requests that subject to SFBO CPNI/PSTN PIN requirements.

To verify Premier status:

  1. Each tenant has special tags in Rave ‘Customer Profile’ tab for support to quickly identify the type of the customer. Follow article 4339885-Office 365 Premier Support in Rave to support Premier request.
    • If a case is marked for Premier General, it will be visible under the Quick facts the tag Premier customer.
    • If a case is marked for Premier Escalation, it will be visible under the Quick facts the tag Premier customer and near the case number is a Gold Diamond.
To verify Partner status:
Partner entitlement is typically handled by CCS TR. If a partner bypasses CCS TR and contacts a Frontline SE directly, refer to the following:
  • 4462573: Cloud Partner Entitlement Policy

Entitling generic admins

Generic Admin accounts are identified by username only and may be used by multiple persons. A sample of a generic admin account is when the username matches 'admin@' or 'administrator@'. All other admin accounts are considered specific to a given user.

  1. On the Administrators tab.
  2. confirm telephone number (if available), email address, and the alternate email address.
  3. Enter confirmation details in case notes.

Entitling non-admins

Entitled customer admins may request Frontline to contact or work with a partner or other non-admin to resolve an issue. This is permissible only under the conditions described in non-admin options below.

Important: In accordance with internal security guidance, limit your communication to only explaining the non-admin options. Do not advise, request, or recommend any options. Carefully document non-admin customer requests and related decisions in case notes in the case. 

In this scenario, any admin type may open a case to request approval for a non-admin to work on a project but it still needs to contain documented approval from a Company Administrator.

Non-admin options
To delegate tasks to a non-admin, the entitled customer admin must choose one of the following options:
  • Unlicensed admin. Customer admin may create an unlicensed admin account for the person working on their behalf. This account can be restricted to one of the more restricted admin types in order to prevent unwanted access to customer information and services.
    • Note: The non-admin does not have to be a named user of the customer’s subscription. For more information about admin accounts, see 2861104: Providing support to non-admins.
  • Written authorization. Customer admin may approve the non-admin to work on their behalf. Written authorization from a Company admin must be one of the following:
    • A Company admin makes an entry into the open case, using their Microsoft 365 admin center access to the case.
    • A Company admin replies to an email message sent by you from within the ticketing system.
Note: The customer’s email message must be sent by a Company admin and must document the following:
  • The name and User ID of the customer admin who will supervise the non-admin’s work.
  • Non-admin contact’s full name.
  • Non-admin contact’s telephone # and O365 email address.
  • A statement of scope of support that the non-admin will provide.
Note: While working with an approved non-admin, the Frontline SEs must include the customer’s Company admin on all email messages with the non-admin.

Redirecting non-entitled admins

  • Explain the support guidance as described in 2861104: Providing support to non-admins.
  • Provide support only to customer admins who are identified as administrators in the case ticketing system.
    Note: Strictly avoid providing any name or contact information.
Troubleshooting entitlement

This section includes solutions for specific scenarios or “corner cases.”

Caller cannot provide required entitlement information

  1. If you cannot verify the caller or if you suspect that you are not communicating with an authorized customer admin, stop providing support and prevent any access to account information.
  2. Tell the person requesting support that they must become an administrator on the account before support can continue.
  3. Inform an admin on the customer account of the unauthorized attempt to obtain technical support on the account.
  4. For additional information, refer to 2861104: Providing support to non-admins.
Important: Limit your engagement to asking only direct questions and then crosschecking the answers with the appropriate records in the case ticketing system. Strictly avoid giving out any personally identifiable information (PII) such as the name of an admin or any other details.
 

Caller or admin not appearing in tools

  1. If the caller is not in Rave because their tenant was just created, use LogMeIn to watch the customer sign in, following the procedure defined in 2774213: Procedure: M365 TS: LogMeIn Usage Guidance.
  2. If the caller is verified as an admin in Rave or by using LogMeIn but does not appear in the case ticketing system, choose Enter Manually in the Administrator Email filed when creating a case in Rave, complete the rest of required information accordingly. This does not apply to non-admins.
  • If the caller is not an admin but requests for an admin elevation (2887514: O365 Elevation Procedure), transfer the case to Office 365 data protection team. 
Case ticketing system is down:

Refer to the following:
  • 2879066: Office 365 primary tools down procedure

Supplemental notes

Verifying customers in callback model. When calling back customers on an existing case that has already been entitled, you need to confirm that you are speaking to the correct admin handling the case. Use the following required guidelines:

  • State your name and provide the case number.
  • Depending on how the call is answered, ask for or confirm first and last name of the entitled admin/case creator.
    • If answered by receptionist or answering service, ask for the admin by first and last name.
    • If answered by the admin and only the first name is used, confirm last name.
      • Example: “Thank you John. Just to confirm, can I also have your last name?”
  • Never share any account information with an unentitled caller.
  • If the contact number in the case is different than the number stored in customer records in the case ticketing system, ensure that the admin confirms the number.
