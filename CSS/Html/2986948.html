INTRODUCTION

This article is intended to help engineers troubleshoot nondelivery reports (NDRs) that are received when Office 365 users send mail to external recipients or when external recipients send mail to Office 365 users.

The following table lists Simple Mail Transfer Protocol (SMTP) codes that customers may encounter in Exchange Online or Exchange Online Protection, the reason that the NDR is occurring, and recommended troubleshooting procedures.

If an explicit SMTP bounce code is missing from the following table list, please note the following: Any bounce code in the range between x.7.500 - x.7.799, or containing an antispam code like “ASxxxxxxxx” should be escalated to the appropriate Antispam team, not to Transport.

This response code usually appears in the SMTP send log on a hub computer in which the SMTP outbound tries to proxy through the FFO FD to a remote system by using STARTTLS. 451 4.4.5 Address domain different from previous accepted address. All recipients must be on the same domain ATTR46. For more details, please refer to  https://support.microsoft.com/kb/3169958

The message could not be unambiguously attributed to any known sender tenant as originating from Office 365 when the email is received. ATTR46 is a corner case where sending MTA did not bifurcate mail to different recipient domains and is sending it to O365 as is. In this case, we are already attributing the mail as incoming to a specific domain which is a tenant and some of the recipients are not on the same domain (even if part of the same tenant). As a result, only the recipients on different domains are deferred temporarily which SHOULD cause the sending mail server to retry the mail only to them on the next retry as per SMTP RFC.

We expect then they would be delivered accordingly on subsequent retries. You could investigate why sending mail server did not bifurcate the mail in the first place and create an escalation to bring this case to Engineering’s attention.

However, if the mail here was supposed to be attributed as originating from a known tenant, please work with the customer and investigate why the email was not recognized as originating from the sending tenant. This is usually due to an invalid configuration of the tenant’s inbound connector. Please check  https://support.microsoft.com/kb/3169958  for more details.

451 4.7.500 Server busy. Please try again later from [XXX.XXX.XXX.XXX]. (ASXXXXXXXX)

Office 365 does not have sufficient reputation for this IP address to allow it to send the volume of email. This issue is transient, and it will fix itself. There are at least two levels of throttling, one for short term (currently about an hour) and one for longer term (currently about 24 hours).  Think of this as a form of grey-listing.

If this is for a customer’s on-premises server or partner, the first step is to verify that they have a proper inbound connector configured. Throttling should not take place if connectors are properly configured.

If this is for an external party sending to Office 365, or if the issue persists, lookup the IP in ViewPoint (and possibly Senderbase.org), and escalate for override.
