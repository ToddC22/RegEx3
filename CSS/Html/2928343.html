What is this article about?

This article describes the Public Switched Telephone Network (PSTN). In simple terms, PSTN is a global network, combining all mobile and landline phone numbers in the world.

Skype connects to the global PSTN network to enable some of Skype’s main features:

  • Calling landline/mobile phone numbers.
  • Receiving calls from landline/mobile phone numbers.
  • Sending SMS messages from Skype.
  • Using Skype To Go service from a landline/mobile phone.

An overview of rates and charges applied when using paid features, can be found on the rates page.

2.2 Initial troubleshooting for all PSTN issues

The initial troubleshooting steps should always be followed before troubleshooting PSTN issues:

When a user comes with a call quality issue, you should check and troubleshoot possible problems related to the internet connection including but not limited to:

Backoffice code

End reason

6430

RTP channel died

6433

Connection lost between SIP and RTP

6519

Timeout, while waiting for client to choose RTP proxy

6534

Client hang up while no packets from RTP

6536

Hang up by RTP: no media from client for 15s

No code exists

The user hung up and no data for >5 seconds

These end reasons indicate that there might be a significant connection issue at customer's side. Proceed with checking the customer potential connection issues by reviewing the troubleshooting steps available in KB 2925256 Technical Guide: Skype: Connection issues.

If the customer continues to experience call quality issues, review the following list:

Ask if the user’s Skype to Skype calls have quality issues.

Check if the customer experiences poor quality only when using a paying feature or the issue is also met when doing Skype to Skype calls. Problems with internet connection on the customer’s side can affect Skype to Skype call quality.

Ask if the user has quality problems only when calling a specific destination (for example, USA) and if calls to other destinations work properly/do not have quality issues?

Having quality problems when calling one specific destination might indicate that the problem is specific to one provider.

Ask if calls to reported numbers have quality issues using different modalities?

Request that the customer try to reproduce the issue using an alternative calling method or procedure:

  • Calls to landlines and mobiles/sending SMS
    • Try from another Skype client, on a different platform.
    • Try while being connected another internet connection / provider.
  • Calls to Skype numbers
    • Try from a regular landline and mobile.
  • Skype To Go calls
    • Try calling the Skype To Go number from another landline and mobile, from another carrier.
    • Try calling the destination number directly from Skype.
  • Skype SMS
    • Try sending a SMS from a regular mobile to the device which do not receive Skype SMS.

Ask the user about the quality while using other calling paying features.

  • Calls to landlines and mobiles:
    If the customer also has a Skype Number, check the call quality they are experiencing when they are getting calls to their Skype number.
  • Calls to Skype numbers:
    If the call quality isn’t optimal when the customer receives calls to their Skype Number, check the call quality they are experiencing when they are calling mobiles and landlines directly from Skype.
  • Skype To Go calls:

If the call quality isn’t optimal when the customer uses their Skype To Go number, check the call quality they are experiencing when they are calling the destination numbers directly from Skype.

Note: You should always make sure that the problem is not connected to an ISP issue on the customer’s side before creating SkypeOut/SkypeIn quality reports for the PSTN Team to investigate.

  • Is the customer online?
    To make or receive calls, customers must be online (exception for Skype To Go calls)
  • Is the customer’s account blocked in Backoffice?
    If the account is blocked, the customer might be able to sign in into Skype but will not be able to use paid features.
  • Did the customer purchase the correct paid feature?
    Check Backoffice to make sure the customer has purchased the correct feature, the subscription is still valid, and the customer still has enough credits to use the features.
    • Calls to landlines and mobiles or sending SMS messages:
      • Does the customer have enough Skype Credits? (Initial fee + first minute fee)
      • If the customer has a subscription, is it still valid? Have they used up all their minute allowance?
    • Calls to Skype numbers:
      • Does the telephone company or provider support calls to the number? If it involves international calling, is it part of the operator calling plan?
    • Skype To Go calls:
      • Does the customer have enough Skype Credits? (Initial fee + first minute fee)
      • If the customer has a subscription, is it still valid? Have they used up all their minute allowance?
      • Does the telephone company or provider support calls to the number? If it involves international calling, is it part of the operator calling plan?

2.3 Troubleshooting calls to landlines and mobiles

1. Was the number dialed correctly?

Make sure the customer is correctly dialing the number they want to reach; double check in Backoffice. In case the customer isn’t sure on how to dial numbers, refer them to:

  • How do I make a call on Skype?
  • Calling mobiles and landlines in Skype for Windows RT
  • Calling mobiles and landlines (Windows Phone 8)
  • Calling mobiles and landlines with Skype for Android – Calling mobiles and landlines (iPhone)
  • Calling mobiles and landlines (iPad)
  • How do I call mobiles and landlines from Skype for Linux?

Note: Backoffice end reason codes 6101 and 6102 describe a normal hang up.

2. Is the number called blocked?

If the number is flagged as blocked in Backoffice (errors 10403/9407), the issue should be reported as follows:

  • 10403: Number is blocked by the PSTN team. Requests to check if the number can be unblocked must be sent via the Star TFS process via Tier 2.
  • 9407 and other blocked cases: Please follow KB 2927565 Procedure: Skype: Blocked Account, PSTN number, IP Address or device ID to determine if the PSTN number can be unblocked.

3. Is the destination included in the price list?

If a certain destination isn’t listed on the following page, calls to a number of that location will not be possible: Rates Pages.

Additionally, check Backoffice for the following end reasons in Backoffice:

  • 9500: No price for destination
    Issue to be escalated to the Carrier Relation team via Tier 3.
  • 9402: Not enough funds on account (db checks that there is at least one minute worth)
    The customer has an active subscription and calls are rejected as the subscription does not cover them. The issue can be reported to the Carrier Relation team via Tier 3.
  • Landline number detected as mobile number

Typically, the case when a customer tries to call a landline number and it is being detected as mobile in Backoffice (or vice versa). As a result, the customer’s active subscription might not cover the calls. The issue can be reported to the Carrier Relation team via Tier 3.

4. Can the number be called using Skype?

  • India to India
    Calling within India from Skype to mobiles and landlines is no longer available. Details can be found     here.
  • Toll-free number
    Calls to Toll-free numbers are only supported in the following countries: Supported countries: Australia, Austria, France, Germany, Estonia, Netherlands, Poland, Taiwan, UK, US.
  • Personal numbers
    Calls to Personal numbers are only supported in Estonia.
  • Special service numbers
    Calls to Special service numbers are only supported in Australia.
  • Satellite numbers
    Supported satellites: AERO, B, B HSD, BGAN, BGAN HSD, GAN/FLEET/SWIFT HSD, M, INI M/GAN/FLEET/SWIFT.
  • VoIP numbers
    Calls to VoIP numbers are only supported in Japan.
  • Freephone numbers
    Calls to Freephone numbers are only supported in Switzerland.
    (or International Network Numbers +883)
  • iNum numbers
    Calls to iNum numbers are not supported.
  • Emergency numbers

Calls to emergency numbers are only supported in few countries. Details can be found on skype.com website.

5. Caller ID testing

Does the customer have a Caller ID set up? If a customer is having an issue calling a particular number without an active Caller ID set up for their account, then ask the customer to set one up by following the steps in What is caller identification and how do I set it up?

If the issue is still occurring, make sure to copy the attempt with a valid caller ID from Backoffice when escalating the case.

6. Escalation from Tier 1 to Tier 2

If the issue isn’t resolved, collect all the troubleshooting steps results and provide the following information:

  • Username of the customer.
  • Destination and PSTN number having an issue.
  • Short and clear problem description (including SR number from ASD) such as “Colombia, 573002770507: Call fails with end reason 10503; SR 123456”.
  • Whether described problem has always persisted (from the moment when the paying feature was purchased) or has appeared recently.
  • Most recent attempts by the user (example cases) with Date and Time in GMT time zone (Examples should not be over seven days old).
    Note: if proper troubleshooting has been conducted, test attempts will be surfaced in the list and should be flagged as CS action.
  • Results of all the troubleshooting steps taken:
    • Call quality/ISP issue? Pass/Fail + Details of steps a, b, c and d
    • Customer online? Pass/Fail
    • Customer account blocked? Pass/Fail
    • Customer purchased the right paying feature? Pass/Fail
    • Was the number dialed properly? Pass/Fail
    • Is the number blocked? Pass/Fail
    • Is the destination listed in the price list? Pass/Fail
    • Can the number be called using Skype? Pass/Fail
    • Caller ID testing? Pass/Fail

Do not escalate without documenting each of the points listed above. Please provide relevant details, for each of the steps.

7. Blocked numbers - End reasons 9500/9402/Landline numbers detected as mobile number (Tier 2)

Those cases must be escalated to Tier 3 without the result of prefix testing. Tier 3 advocates will then report the issue to the proper team within Skype (Fraud or Carrier Relations team).

  • Note: End reason 10403 calls must be reported to the PSTN team via the Star TFS process by Tier 2.

8. For all other cases, Prefix testing (Tier 2)

Important: Tier 2 advocates need to be provided with access to carry out prefix testing. If an advocate tries to perform prefix testing but does not have the correct access, an error code 13403 will be displayed in BO. In this case, Tier 2 should send a list of the Skype accounts that need access for prefix testing to T3SkypeCSS@microsoft.com.

  • Note: Access is provided to Skype accounts, not LDAP accounts.

Perform the prefix tests using the SkypeOut Troubleshooting Tool in Backoffice:

[Asset 4019667]

Check if the user is having an issue calling a specific number with or without active Caller ID (*). Based on that, activate/deactivate your Caller ID and make the tests as follows:

  1. Select the Test button to test a specific number. A pop-up window loads containing the information about providers currently defined for this destination as well as Test Prefixes to force SkypeOut calls via these specific providers.
  2. Initiate a test call using each of Tier 1 providers by clicking on the corresponding Test link.
  3. If none of the calls via Tier 1 providers were successful, then try Test Prefixes for available providers Tier 2, Tier 3, etc. until first successful call.
  4. Copy paste from Backoffice results of your attempt using test prefixes to STAR TFS report (under section Results of troubleshooting (Both using test prefixes and landline)) along with description (PASS/FAIL and symptoms) for each test result.

  • Provider 105 Tier2 Test Prefix +999228907447782333123: PASS
    2015-03-24 09:13 447782333123 United Kingdom - Mobile - Hutchison3G EE SkypeOut 00:12 €0,066 0.090 0.160 Regional pricelist EU EUR 6101: user hang-up (Skype side) 304 105 -

* If the user is having an issue calling particular number without active Caller ID, then also perform testing with active Caller ID by initiating a test call using each of Tier 1 providers (step two). Copy paste from BO results of your attempts using test prefixes to STAR TFS report (under section "Results of testing with Caller ID (if the user is having an issue calling particular number without active Caller ID)“) along with description (PASS/FAIL and symptoms) for each test result.

9. Pre-escalation checks (Tier 2)

  • No duplicates – One issue for one number.
  • Check known issues beforehand and inform customer accordingly if it is connected to his issue.
  • In case of de-escalation from PSTN Team, contact customer to ask for more information.

10. Escalation to the PSTN Team

Escalate the issue to the PSTN team using the Star TFS form “Skype Bug”.

  • For the TP site, use this form.
  • For the MHL site, use this form.
  • For Concentrix site, use this form.
  • For the TekExpert site, use this form

Note: Allow two business days for the PSTN team to update the bug.

  • Once the ticket has been created, it will be assigned to a PSTN Quality team member who will investigate the reported issue.
  • PSTN Quality team will post their findings/solutions in Star TFS ticket and Star TFS ticket will be assigned back to the advocate if more information is needed. The advocate will gather the additional information and update the ticket as necessary. Please respond to the request within 24 hours.
  • If the issue is resolved the PSTN Quality team will close the ticket.
  • If problem still persists and customer comes back, the advocate will reopen the ticket (in case ticket has been already closed) or assigns it back to PSTN Quality team member (in case ticket is still open) and should add the problem description with fresh examples of faulty calls to comments. Further investigation will be done by the PSTN team.

Make sure to provide all the information below in the TFS form:

  • Username.
  • Receiver's number.
  • Short and clear problem description.
  • Most recent attempts by the user (example of failed calls) with Date and time in GMT time zone. Examples should not be over seven days old.
  • Results of troubleshooting (Both using test prefixes and landline).
  • Whether described problem has always persisted or has appeared recently.
  • Results of ISP troubleshooting (if issue is related to Call Quality).
  • Results of testing with Caller ID (if the user is having an issue calling particular number without active Caller ID).

2. Privacy settings check
Customers should enable their privacy settings to receive calls from people who are not in their contact list. For detailed steps on how to set up privacy settings, see Why am I not receiving calls to my Skype Number?

3. Backoffice check

  • Is the number correctly assigned to the username. Search either by username or Skype number, to make sure the Skype Number is still active and correctly assigned to the right username.
    • The same username might have several Skype Numbers assigned to it.
    • The customer might be signed into another Skype account than the one which holds the Skype Number.
  • Incoming calls to Skype Number. Search by username, select Calls to online number and review the call end reasons.

4. Test dialing the Skype number
In addition to customer’s call testing done in the mandatory troubleshooting steps detailed above (step 1, point c. and d.), make sure you try dialing the number from office landline where caller ID is hidden. Check again the Call to online number section of Backoffice for the call end reasons, after your test.

2.5 Troubleshooting Skype to Go

Skype To Go issues can be classified into two categories:

  • Issues calling the Skype To Go number or dialing the Access number. These type of issues are mostly due to the PSTN carrier of the user.
  • Once connected to Skype To Go, issues calling the destination number. These type of issues are always due to Skype: It is the same as PSTN troubleshooting.

1. Was the number dialed correctly?

For more details on how to use a Skype To Go number, see:

  • How do I set up Skype to Go?
  • How do I use my Skype to Go numbers?
  • Did the customer dial their Skype To Go/Access number properly from their landline or mobile?
    • Customer can check the format of their Skype to Go / Access number from their account page.
    • Explain to the customer how to use the correct number format.
    • If the customer does not remember their PIN code when dialing their Access number, advise them to reset it by connecting to their account page.
  • Is the destination number dialed properly?
    • Explain to the customer how to dial the destination number properly. When greeted by the voice menu, they should select option two and enter an international number starting with the country code (for example, +1 for the US or +44 for the UK) followed by the number itself.
    • If the customer is sure to dial the number properly, check in Backoffice to confirm.
    • Numbers in Nigeria, Morocco and Egypt cannot be called using Skype To Go. For more information, redirect customers to What is Skype to Go?

2. Is the destination number called blocked?

If the number is flagged as blocked in Backoffice (errors 10403 / 9407 / etc.), the issue should be reported as follows:

  • 10403: Number is blocked by the PSTN team. Requests to check if the number can be unblocked must be sent via the Star TFS process via Tier 2.
  • 9407 and other blocked cases: Please follow KB 2927565 Procedure: Skype: Blocked Account, PSTN number, IP Address or device ID to determine if the PSTN number can be unblocked.

3. Is the destination number included in the price list?

If a certain destination isn’t listed on the following page, calls to a number of that location will not be possible: Rates Pages. Additionally, check Backoffice for the following end reasons in Backoffice:

5. Caller ID testing

Does the customer have a Caller ID set up? It must be enabled for the system to recognize the customer and connect the call. If the customer doesn’t have an active Caller ID set up for their account, then request the customer to set one up by following the steps in What is caller identification and how do I set it up?

If the issue is still occurring, make sure to copy the attempt with a valid caller ID from Backoffice when escalating the case.

6. Backoffice check

In Backoffice, you can see the following:

  • Registered phones: Phones recognized by the system and from which calls can be made without having to dial the PIN number.
  • Access number: For more information about Access numbers, see What is a Skype To Go Access number?
  • Contact numbers: Phone or Skype name entered as contacts, which can be directly called using their specific Skype To Go number.

From this page you will be able to:

  • Disable the feature, however the customer can reinstate the functionality from their account page.
  • Manually remove registered numbers from the list, as well as the selected Access Number.
  • Manually Reset PIN and request the customer to set a new one from their account page.
  • Removing all user settings will permanently delete all Skype To Go settings for the user account. Use this button only in case the customer is aware tis action cannot reversed.

  • If problem still persists and customer comes back, the advocate will reopen the ticket (in case ticket has been already closed) or assigns it back to PSTN Quality team member (in case ticket is still open) and should add the problem description with fresh examples of faulty calls to comments. Further investigation will be done by the PSTN team.

Provide all the information below in the TFS form:

  • Username.
  • Short and clear problem description.
  • The number you are calling from (registered phone number).
  • Skype Access Number/Direct Dial Number.
  • PSTN number (number you are calling).
  • The telephone company or provider used by the caller.
  • Dates of failed calls (not older than one week).
  • Time of failed calls in GMT time zone.
  • Results of ISP troubleshooting (if issue is related to Call Quality).

2.6 Troubleshooting sending SMS messages

1. Was the SMS sent correctly?

Make sure the customer is following the correct process for sending an SMS; double check in Backoffice. In case the customer isn’t sure on how to send an SMS, refer them to:

  • How do I send an SMS from Skype?
  • Sending SMS text messages in Skype for Windows RT
  • Sending SMS text messages on Skype for Android
  • Sending SMS text messages (Mac)
  • Sending SMS text messages (iPhone)
  • Sending SMS text messages (iPad)
  • How do I send SMS text messages from Skype for Linux?

2. Does the Skype platform of the user support sending SMS?

3. Is there an issue with the receiver’s mobile device or network coverage?

  • Is the receiver’s phone not switched off?
  • Bad network coverage?
  • Does SIM card have enough capacity?
  • Is the phone SMS compatible phone?
  • Is the receiver roaming abroad?
  • Location of the user may be problematic.
  • To send SMS to someone located in the USA, customer must make sure the recipient’s carrier supports receiving Skype text messages. The carriers supporting receiving Skype SMS text messages are listed below:
  • Alltel
  • AT&T
  • Cellular One Dobson
  • Cellular South
  • Centennial
  • Cincinnati Bell
  • Nextel
  • Ntelos
  • Sprint
  • T-Mobile
  • Unicel
  • U.S. Cellular®
  • Verizon Wireless
  • Virgin Mobile USA

4. Sender ID issue?

To set up Sender ID, refer the customer to:

  • How do I show my mobile number when sending SMS text messages from Skype?
  • If the Sender ID verification message is not delivered to the customer, it may be because the carriers filter and restrict SMS which:
    • Come with an alphanumeric Sender ID (Skype username).
    • Come from the verification code sender Skype.
  • If the carrier doesn’t apply such restrictions, collect information and report the issue.
  • SMS sent to mobile phones in Canada, China, Taiwan and the USA will not display the customer’s Skype Name or a validated mobile number. Instead, they display generic, pre-defined numbers as the sender ID.
  • SMS text messages sent to mobile phones in Russia, a generic number is displayed unless the customer has set their mobile number as the sender ID.
  • If the Sender ID keeps not showing properly, even though it should, report it to the PSTN team to confirm it is (or not) an expected behavior. Provide the following information:
    • Receiving number
    • Expected Sender ID
    • Received sender ID
    • Date & Time in GMT time zone. Examples should not be over seven days old.

5. Is the SMS received missing some characters?

If the content of the SMS received isn’t correct, or displays incorrect characters, report the issue while providing the following information:

  • Receiving number
  • Text sent
  • Text received

6. Backoffice check

  • Check the SMS section for any issues. See table below for known issues.
  • Problematic SMS should not be older than seven days, after that period, the message will no longer be delivered.
  • As part of the troubleshooting, ask the customer to send a new SMS and review the outcome in Backoffice.

7. Escalation from Tier 1 to Tier 2

If the issue is not resolved, collect all the troubleshooting steps results and provide the following information:

  • Username of the customer.
  • Receiver’s mobile number.
  • Name of the receiver’s (current) mobile network provider/carrier (If the user did not provide you this information, then do not use International numbering site to get information as this site does not give information in case the number was ported from one operator to another).
  • Was the receiver roaming when the problem occurred?
  • Did the receiver ported the number to a new provider (and if so, the new provider's information).
  • Short and clear problem description (including SR number from ASD) such as: “Colombia, 573002770507 Operator XYZ: SMS delivered in Skype but not received; SR 123456”.
  • Whether the issue has always existed (from the moment when the paying feature was purchased) or has appeared recently.
  • Most recent attempts by the user (example cases) with Date and Time in GMT time zone. Examples should not be over seven days old.
    Note: if proper troubleshooting has been conducted, test attempts will be surfaced in the list and should be flagged as CS action.
  • Results of all previous troubleshooting:
    • Call quality/ISP issue? Pass/Fail + Details of steps a, b, c and d
    • Customer online? Pass/Fail
    • Customer account blocked? Pass/Fail
    • Customer purchased the right paying feature? Pass/Fail
    • Was the number dialed properly? Pass/Fail
    • Does the Skype platform of the user support sending SMS? Pass/Fail
    • Is there an issue with the receiver’s mobile device or network coverage? Pass/Fail + details
    • Sender ID issue? Pass/Fail + details
    • Is the SMS received missing some characters? Pass/Fail
    • Backoffice check? Pass/Fail

Do not escalate without documenting each of the points listed above. Please provide relevant details, for each of the steps.

8. Pre-escalation checks (Tier 2)

  • No duplicates – One issue for one number.
  • Check known issues beforehand and inform customer accordingly if it is connected to his issue.
  • In case of de-escalation from PSTN Team, contact customer to ask for more information.

9. Escalation to the PSTN team via Tier 3

Escalate the issue to Tier 3 using KB 2907961 Procedure: Skype: Consumer escalations, making sure to provide all the information below:

  • Username.
  • Receiver’s mobile number.
  • Short and clear problem description.
  • Example cases with Date and Time in GMT time zone. Examples should not be over seven days old.
    • If SMS is received incorrectly. For example, a character is missing include:
      • Text sent
      • Text received
    • If Sender ID for received SMS is shown incorrectly include:
      • Expected Sender ID:
      • Received Sender ID:
  • Name of the receiver’s current mobile network provider/carrier. (If the user did not provide this information, do not use International numbering site to collect the information as this site does not give information in case the number was ported from one operator to another).
  • Whether or not the receiver was roaming (if yes, this could cause the text to fail).
  • Whether the receiver ported the number to a new provider (and if so, the new provider's information).
  • Whether the receiver's device is text compatible.
  • Whether the receiver’s SIM card was full at the time of sending the text.
  • Whether described problem has always persisted or has appeared recently.
