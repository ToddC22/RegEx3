ASMS – Cloud Solution Provider (CSP) Offer – 145P

Submit a ModTrack request to update this TSG.

Overview

This TSG provides guidance on how to support Cloud Solution Provider (CSP) partners.

This offer allows for transactions across Microsoft cloud services (e.g., Azure, Office 365, Enterprise Mobility Suite, and Dynamics CRM Online) through a single platform, enabling partners to do the following:

  1. Own the customer lifecycle and relationship end-to-end.
  2. Set the price of their offer.
  3. Establish terms.
  4. Directly bill customers.
  5. Provision and manage subscriptions.
  6. Attach value-added services.
  7. Be the first point of contact for customer support.

Additional information about this offer can be found in the Online References section of this TSG.

For more information about Azure CSP Sponsorship, please see TSG 3014508.

For more information on the plans available to CSP Partners, see TSG 4530078.

Policy

CSP cases which come in with the correct entitlement should be worked by SEs/TA/PTAs that handle Premier, Pro-Direct, and Partner volume.

If you receive a case with such an entitlement and it was incorrectly assigned to your queue (e.g. by the Duty Manager (DM)), move it to the respective ASMS Premier queue (See TSG 3189895 for specific queue names) and email the DM asking them to assign it to a Premier/Pro-Direct/Partner resource.   5. Security Requirements for CSP

Microsoft is introducing a set of mandatory security requirements for partners participating in the Cloud Solution Provider (CSP) program, Control Panel Vendors, and Advisor partners.

The requirements are:

  • Enable Multi-Factor Authentication (MFA) for all partner users (administrators and end-users) of partner tenants at no cost via Azure Active Directory baseline policy.
  • Implement the secure application model to integrate with Partner Center API and other APIs using app + user authentication such as Microsoft Graph or automation using tools such as PowerShell.

The new security requirements terms will be included to the CSP Program Guide (August 1, 2019) and Microsoft Partner Agreement (available on September 1, 2019). The same security requirements will be enacted for the Advisor program as well.

Partners who fail to abide by these security practices and obligations will not be able to transact in the Cloud Solution Provider program or manage customer tenants leveraging delegate admin rights, once these partner security requirements are enforced.

We are in the process of establishing an enforcement date for the requirements and will notify partners of the date with detailed information.

Customer Experience and Provisioning

All CSP partners have a tenant in Azure Active Directory (AAD). The process of becoming a CSP partner creates three groups in the partner tenant: AdminAgents, HelpDeskAgents and SalesAgents. The partner administrators populate these groups with employees (users).

When a CSP partner makes a sale, they provision a new subscription for their customer. The customer may already exist in AAD or this may be a new Microsoft cloud customer. In the latter case, the customer is created in AAD and the subscription is provisioned. This could be a seat-based subscription such as Office 365 (O365) or Enterprise Mobility Suite (EMS) or, from July 2015, an Azure Pay-As-You-Go subscription. The operation of provisioning new subscriptions is available to partner-users who are members of the AdminAgents or SalesAgents groups.

1. Admin on Behalf of (AOBO)

  • When a partner provisions a subscription, they are automatically provided with the following administrative permissions:
    • The partners AdminAgents group is added to the customer’s TenantAdmin role.
    • The partners HelpDeskAgents group is added to the customer’s UserAdmin role.
    • For Azure, the partner’s AdminAgents group is made the Owner of the subscription.
  • This configuration is modeled by the creation of a Foreign Principal Object (FPO) in the customer directory, representing each of the partner agent groups, and the FPOs are inserted into the customer admin roles. Members of the partner AdminAgents group can then operate on all the partners’ customers as a tenant admin and on all the partner’s Azure subscriptions as a subscription owner.
  • A partner AdminAgent accessing the Partner Center portal will see a list of their customers. The agent can select a customer and links are made available to allow the agent to navigate to the admin portals for all purchased services (e.g. the O365 admin portal, or Ibiza for Azure.) When the agent navigates to these portals, they will be in the customer’s context and see only those subscriptions purchased from the CSP partner by that customer.
  • Larger CSPs may want to integrate Directory Management with their own customer management systems. AAD PowerShell lets the AAD Graph API support the partner authentication model, providing scriptable/programmable access to the directory. Similarly, partners who want to call Azure PowerShell and REST APIs will need the same capability. Azure PowerShell has added a “­–tenant” parameter to support this.

2. Multi-tier and Delegated Admin provisioning

  • Partners are split into two categories; Partners that deal directly with their customers and those that work through their own reseller channel (Distributors.) In this model the Distributor partner acts as the CSP and provisions the customer subscriptions and is responsible for paying the Microsoft bill.
  • Resellers (which are also provisioned in AAD as a CSP or Advisor partner) can send an AOBO request to the customer admin. If the customer decides to grant the reseller access, their agent groups are configured in the same way as the CSP partner. The result is that both partners (Distributors and Resellers) have AOBO rights for the customer tenant.
  • To obtain Azure, the Distributor will use Azure PowerShell to add the reseller partner’s AdminAgents group as a subscription Owner.
  • Contributor - Contributors can manage everything except access.
  • Owner - Owner can manage everything, including access.
  • Reader - Readers can view everything but are not eligible to make changes.

7. Billing/Usage Reports

  • Partners manage the billing relationship with their end customers and send their own bills to the customers. Customers will not have access to any billing details from Microsoft, unless provided by the partner. ASMS has no visibility into the billing of the customer.
  • The partner is also responsible for paying Microsoft for these subscriptions. A multi-subscription, multi-customer aggregated bill is sent to the partner each month.
  • Partners have access to the usage report from the CPC portal. They will need to follow the below steps to obtain this information:
  1. Search by the partner's Tenant GUID.
  2. Click the arrow to go into the partner's profile.
  3. Within the profile, click the Billing tab.
  4. Within the Billing tab, click on Invoices.
  5. Select the appropriate usage file by clicking on First_Party_Azure for the relevant month.
    Note: CSP's usage file is referred to as a "recon file" and contains multiple subscriptions.
  6. Within the appropriate file, filter on the relevant subscription.

Note: The MPN ID that displays in the recon file references the partner's subscription. If a partner has concerns about discrepancies of the MPN ID in their recon file, ask them to also check the subscription in the Partner Center, where they can update the MPN ID. Partner unable to access end customer's tenant due to Multi-Factor Authentication (MFA)

If A CSP Partner receives the error message "Sign in Failed. Sorry, we had some trouble signing you in. Click 'Try again' to try again." while attempting to access their end customer's tenant/subscription that is using an Admin Agent account, it is possible that the Partner tenant users do not have MFA enabled. 

In this scenario, attempt the following troubleshooting steps: 

  1. Advise the customer to work with their Global Admin on the partner tenant, to enable MFA for the Admin Agent account. In return this should give the Partner the ability to manage customer's subscriptions.
    • Provide the customer with steps on how the to enable MFA. Steps can be found here. 
  2. If the customer is facing issues setting up MFA after attempting step 1, then engage the Cloud Identity Team (AAD- Authentication Queue). See TSG 2832928 for additional guidance. 

Root Cause of issue: If the customer tenant has MFA setup (which is added security for all customers logging to the tenant) but the partner Admin Agents who have delegated access to the customer's tenant do not have MFA setup, then login request to the customer's tenant for Admin Agents will get denied access. This happens because Admin Agents do not meet the AAD security standards which are set on the customer's tenant and all login attempts from any user who does not have MFA setup would be denied.

Unable to see subscription in Azure Portal

Customer
If the customer is unable to see the subscription, follow the troubleshooting steps below:

  1. Verify if the user is added to an RBAC role. Engage your TA who can assist in obtaining this information or request the partner to provide a screenshot from the portal if possible.
  2. Verify if the user is using the correct authentication.
  3. Verify that the user is looking under the right tenant. 

If the customer is still unable to see the subscription after attempting the above troubleshooting steps, engage Cloud Identity. 

Partner
If the partner is unable to see the subscription in the portal, attempt the following troubleshooting steps:

  1. Check with partner/customer to see if the reseller relationship is successfully established. This could be verified in CMAT under CSP relationship for customer Tenant

    1. Use Jarvis Operation to access AAD Developer Acceleration

  5. Remove this user and then try to access the subscription in https://portal.azure.com.

3. If the above two steps fail, check if the partner is able to access the subscription to see if they see a FPO (Foreign Principal) role in the Azure Portal under Access control (IAM).
