1. Program-level out of scope guidance

Note that the following topics are considered out of scope at the program level for BOS:

  • Adult advertisers.
  • Language translation and localization.
  • Work requiring advocates to log into non-Microsoft products and services (such as Google Ads) due to privacy concerns.
  • System/tool limitations inherent to each service.
  • Making any modifications directly to advertiser accounts without explicit direction or approval from requestor.
  • Providing service for accounts who are currently in bad standing with Microsoft Advertising (for example, credit hold and QBR penalties).
  • Any services that are not listed on the AES Menu of Services, but out-of-scope requests are considered on a case-by-case basis and may be handled with a stated turnaround time that is outside the normal. See KB 4473076 AES: Policy: Microsoft Advertising: Out-of-Scope Request Guidelines for guidance.
    • Note: Out of scope requests are considered on a case by case basis and may be handled if the following requirements are met:
      • You believe can execute the request based on detailed instructions provided by the requestor.
      • You have bandwidth to complete the request, adjusting SLA as required.
      • There are no tools or system blockers.
  • Any customer in non-supported AES markets.

2.2 Request acknowledgement

You should send a request acknowledgement email within 24 hours of ticket submission if:

  1. The request will take longer than 24 hours from submission of the ticket to process.
  2. You need more information or clarification to complete the request.

It is not always necessary to send an acknowledgment email if you know that you are able to complete the request within 24 hours of submission and you already have all the information you need to do so. The Request Acknowledgment Email should contain the following:

  • Request acknowledgement – A general acknowledgment of the request
  • Clarification/Information request (Optional) – Any clarification or information needed to complete the service.
  • Service Level Agreement (SLA) – The amount of time needed to complete the work.

Best practice: It’s always a good idea to thoroughly review a request to ensure that:

  • All the information needed to complete the request is provided.
  • You believe that the request can be completed within SLA.

3. Emails, attachments, and child tickets

The following provides guidelines on how emails, attachments, and child tickets should be handled for requests from account managers/coordinators vs. services (frontline) advocate requests:

Return to top

3.1 Email communications

Account managers/coordinators:

Only the original ticket owner should be communicating with the requestor to prevent any confusion.

Handling multiple-tactic requests: If multiple-tactic requests are received in the same email, a new parent service request (SR) should be created for each tactic immediately. However, communications relating to each individual tactic should be sent from the main SR to avoid sending multiple emails to the requestor. Proper communication between the main SR owner and the other PT owner should be followed so that pushbacks and other information are communicated in a timely manner.

Example: When the request is to propose and then upload Sitelink Extensions, you are responsible for creating a new, separate ticket, and provide the new case number for the upload portion of the work to the requestor. You must send a separate email communication and proper documentation in the ASD Case Notes.

Services (frontline) advocates:

All communications should take place in ASD case notes per child ticket.

Inform the Services (frontline) advocate that an update is waiting for them by adding a brief note in the parent ticket, and advise them that the AES ticket has been updated. Do not ask questions or engage in discussion via parent ticket notes. When you update ASD notes, don’t delete the existing notes. Place your latest notes at the top of the case notes thread, to allow for easier visibility when reviewing in the tool.

Note: To properly document the ticket activity, separate the email communication and the ticket activity documentation, still following the cadence discussed above. This will make the communication professional, as well as avoid any confusion of information. See section 10, Documenting requestor engagement, for further details.

Return to top

3.2 Attachments

Account managers/coordinators:

Proposal file: The final output (proposal) file, which is to be sent to the requestor, must only be sent through the parent ticket via email attachment, or the Very Large File Share if greater than 10MB.

Downloads/working files: If multiple BOS advocates are involved with any given request, their work should be attached to their respective child tickets via email to capattach@microsoft.com. This includes but is not limited to proposal, skipped, or error files.

Attachments with any customer data and personal data must be encrypted.

Services (frontline) advocates:

Proposal file/download/working files: The final output (proposal) file, which is to be sent to the requestor, should be sent via capattach@microsoft.com so that the requestor can pick this up from the tool. Files greater than 10MB should be shared via the Very Large File Share. This includes but is not limited to proposal, skipped, or error files.

Note: For Monthly Invoice, any information containing customer legal documents should be shared using the Data Transfer Management (DTM) link.

Return to top

3.3 Child tickets

Account managers/coordinators:

Create child tickets for every +1 account number/XID request you receive from an Account Manager/Account Coordinator (for example, if you are asked to work two XIDs, you would handle one XID in the parent SR and one XID in a child SR).

The child tickets should be created immediately upon receipt of the parent SR request. This will ensure the SLA is being tracked for each XID individually and ensure accurate SLA reporting.

The owner of the parent ticket is responsible for collating the work that is attached in child tickets to create the final output (proposal) file.

If you review the parent SR request and you are unable to begin processing the request without further clarification from the account manager/coordinator, then you may temporarily delay the creation of child SRs until the necessary clarification has been provided. Document the reason behind the delay of the ticket creation. See section 10, Documenting requestor engagement for further details.

If only selected accounts need further clarification with the requestor, you should only delay the child ticket creation for the affected accounts. Then continue to create child tickets and work on the remaining accounts. This will ensure request continuity and accurate SLA reporting.

Services (frontline) advocates:

N/A. The request is submitted via a single child ticket. If the frontline advocate submits multiple requests in one child ticket, please follow the above AM/coordinator guidelines and ensure that we are creating extra tickets on behalf of the frontline advocate to document the back office work motions. Do not pushback for this reason.

4.1 ASD coding guidelines

Opening a case:

  1. Go to the QVIEW tab in ASD.
  2. In the QVIEW dropdown, select your site-specific view.
  3. Locate the Owner Name column and select the Sort button to display the full list of assigned tickets. Once you’ve found your name, select the checkbox to the right to display only those tickets assigned to you.
  4. Find the ticket that you will work on and check the Customer Email column to see who submitted the request.
    • If it is a Parent ticket and the email address ends with @microsoft.com, the request is from an internal Service Team member (for example, an Account Manager/Account Coordinator) who is representing the advertiser (such as, v-john@microsoft.com or johnsmith@microsoft.com).
    • If it is a child ticket, the request came from a vendor Service Team member (for example, MAST Service or Support advocate) who is representing the advertiser (the alias in the Customer Email column will be an external alias, such as, john@customeremail.com).
  5. Right-click the ticket you wish to work and select Edit to open the AES incident form.
    • Note: If the ticket is not in editable, navigate to the bottom of the ASD form and select Edit.
  6. Check that the SR number is already created and present on the bottom portion of the form.
  7. Complete the incident form (ticket). See steps below for how to do this.

Completing the incident form (ticket):

   1. Going in order from top to bottom, complete the required fields of the incident form (ticket).

  • You will not be able to move to the next field until you fill out the form in the proper order.

Important: Be sure to select Product and Advocate Type first! This will ensure that you are taken through the correct coding workflow for Back Office Support.

   2. Select the following options in the Reason for Contact section:

  • Product: Microsoft Advertising
  • Advocate Type: BackOffice
  • Program: Advertiser Enablement Services
  • Customer Issue: Select the appropriate code to match the request.
  • Topic: Select the appropriate code to match the request.
  • Scenario: Select the appropriate code to match the request.
  • Problem: select the appropriate code to match the request.
  • Solution Delivered: While the ticket is being worked, select Solution in Progress. Once you have completed the work and are closing the ticket, update with the appropriate code that describes what service you completed. If you execute more than one service for a request, create and code separate tickets.

Note: The appropriate request code is outlined under ASD Coding Guidelines in this article.

   3. When filling out the fields of a child ticket, use details in the Parent ticket to complete the form:

  • Primary Target Market: Fill out the market that’s noted in the parent ticket.
  • Account Number Available?: Select Yes
  • Account Number: Copy and paste the Account Number noted in the parent ticket.

    4. Follow these protocols when filling in the greet customer and collect info section of the ticket:

  • Subject: Follow your site-specific procedures for completing the subject line field.
  • Status:
  • Open: Keep the ticket open while it’s being worked.
  • Closed resolved: Select this option when you are ready to close the ticket and you’ve successfully finished the work for the requestor.
  • Closed unresolved: Select this option when you are ready to close the ticket and you were not able to successfully finish the work for the requestor.
  • Queue: The queue may be updated in this dropdown as needed. Follow your site-specific procedures.
  • Issue specific details required: Leave blank
  • Scheduled follow up: Leave blank
  • Bug ID: If you’ve escalated to any of our partner teams for technical assistance and you’ve received a Bug ID, paste this information directly in this field.
  • DTM shared with internal users: Do not remove the email address cssba_msft@msncs.com. Then, make sure to add your support’s @msncs.com credential to share documents with them. Use a semicolon ";" to add more than one user to the case. See the Customer data security requirements section of this article for details.

4.4 Campaigns and ad groups treated

Within the ASD form, the Number of Accounts, Campaigns, and Ad Groups Treated fields should be used to identify how many accounts, campaigns, and ad groups have been worked on per ticket.

Accounts Treated is the most important of these fields, and should always be filled out in the parent ticket in the format of the total accounts treated minus one. This number should match the number of child tickets that are created. For example, if you have one parent ticket and 20 child tickets, the number in that field should be 20.

These fields are mandatory. For data accuracy and calculation purposes for campaigns and ad groups, we ask that a minimum value of 1 be entered. This is not applicable for Accounts Treated where there are no child tickets, this should be left at 0.

For campaigns and ad groups:

  • If a request is made at CID level and involves multiple accounts, you will still need to update the campaign or ad group per the below instructions.
  • If a request is made at the campaign level (for example, sitelink extensions), the Ad Group Treated field should be set as 1. This is the same when handling requests at the account level.
  • If a request is made at the ad group level, the Campaign Treated field is based on the number of campaigns associated with the requested ad groups.
  • For BOS services, such as invoice account setup and reporting requests, which do not involve work at either the campaign or ad group level, populate both fields with 1.

We also have maximum values, which we should not exceed:

  • Campaigns should not be entered higher than 10,000.
  • Ad groups should not be entered higher than 200,000,000.

5.3 Email font, font size, and font color to use

Per Microsoft compliance, all email communications must be formatted as follows:

  • Font: Segoe UI
  • Font Size: 12
  • Font Color:
    • Generic: Gray
    • HTML color code: 696969
    • Microsoft Word version Microsoft Office Professional Plus 2016: White, Background 1, Darker 50%
  • The same font, font size, and font color should be used throughout the entire email.
  • Bold, italics, ALL CAPS, or underline text must only be used when highlighting important information such as the tactic, CID, XID, or the SR number.
  • Proper line spacing must be observed.
  • The Microsoft Image (logo) should be included as it is our corporate style. This applies as well to the gray text.
  • Remember to change any information highlighted in yellow or green before sending the message to the submitter.
  • Customize the template, if needed, to avoid sounding robotic.

6. Following-up with requestors

6.1 Use of Microsoft Advertising advertiser enablement services forms

If a request has been submitted which is based upon the Microsoft Advertising Advertiser Enablement Services Form (see KB 2023008 AES: Process: Microsoft Advertising: AES Requests), and you do not have enough information to complete the service request, send a follow up email to the submitter to ask for further clarification.

6.2 Use of Microsoft Teams

You may encounter scenarios where it is most efficient to communicate with the submitter via an online instant messaging tool, such as Microsoft Teams. In such instances, you must copy and paste the conversation into the ticket case notes once the conversation has ended.

When to use Teams:

  • When after reviewing the ticket assigned, you have questions and clarifications that we need answered to proceed with the request. You can chat immediately with the requester to get a quick answer.
  • When after 24 hours of sending the acknowledgment email with pushback and clarification items, you have not received any reply from the requester.

Guidelines around using Teams:

  • For multiple SRs, same request, same requester: To avoid too many contacts and requesters getting multiple messages, a POC will be assigned. The POC will be responsible for contacting the requester through Teams to clarify the needed information.
  • Keep the communication professional. Comical font types, sizes, and colors should not be used.
  • Do not communicate through Teams if the requestor’s status is:
    • In a Meeting
    • In a Call
    • In a Presentation
    • Do not Disturb
    • Offline

To comply with privacy and security policies, the following outlines the approved usage of IM:

Permitted use of IM:

  • You can use Teams with requesters, but both parties must use their Microsoft alias or v-account.
  • You may use IM for quick questions or clarifications on a ticket (such as campaign budget, over-length text ads, negative keywords, time zone, and targeting) or to reach out when the submitter is nonresponsive to emails.

Not permitted use of IM:

  • You are not allowed to use IM to receive or send files, or share their desktop to share files that contain personal data (including but not limited to, XID, and SAP IDs).
  • You cannot use Windows Live IM to chat with submitters.
  • Submitters should not initiate an IM to express urgency or open tickets with you. If this occurs, notify a supervisor.

Best practice: You can send a quick recap to the submitter by highlighting the agreed instructions/clarification for the request.

See section 10 for how to save the Teams conversation.

6.3 Total case ownership

In a Total Case Ownership model, the submitter should only be in direct communication with one person, the case owner. Internally, if work is distributed to additional advocates to meet SLA, the submitter should not be made aware. The following provides guidelines when handling very large requests, additional information and status updates, and unresponsive requestors.

Very large requests

When handling very large requests, it is especially important that you provide submitters with a sense of ownership and confidence in our ability to process their work. It is also critical that the information and updates we provide to requestors will allow them to clearly and easily communicate updates with their customers and plan for service completion around the SLA we provide. Use the AES_Work_In_Progress template.

For all updates relating to large request handling, ensure you follow this guidance:

  1. Inform the requestor why more time will be required (if applicable) and provide an updated estimate for how long it will take to complete the work.
  2. Work with the requestor to determine the cadence of updates to be provided (if deemed appropriate by you and the requestor) and establish if a single output file is to be provided or whether the current/complete work should be provided at each update stage.
  3. Provide completion status updates every 48 hours (such as, “We have processed X out of XX many accounts”). If the request itself should take more than one week, and the requestor is aware of this SLA from the start, then updates should only be provided once per week.
  4. Always set a next action for when you will follow up again, regardless of the SLA.
  5. Always ask clarifying questions as soon as possible, to avoid additional delays in processing.
  6. Inform the Microsoft BOS Technical Advisors (sbosesc@microsoft.com) of the encountered scenario for visibility.

Additional information and status updates

You may encounter scenarios in which they need to follow up with requestors to either:

  1. Ask for additional information.
  2. Provide status updates.

For either scenario, follow-up should take place within two business days or within a time period agreed upon by the requestor (for example, in one week if the Account Manager is away for vacation).

Note: Use the AES_Out_Of_Office template. In the event that a requestor is out of office for a prolonged period of time (one week or greater), review their out of office message to see if there is an alternative person that you may be able to work with until the requestor returns from leave.

Unresponsive requestors

In the event that a requestor becomes unresponsive after two follow up attempts (spanning four business days), send a closing email on the fifth day to the requestor stating that they did not hear back from them and they can re-engage with AES for any service request they have. The two follow up attempts should take place on the second and fourth business days.

Note: Use the AES_Unresponsive Response email template from ASD. If the request is only completed partially, provide the survey link. If no work has been done yet, the survey link should be removed.

6.4 Requests from non-Microsoft domain emails

As a reminder, the BOS team is an internal support team and only offers support to members of the Microsoft Advertising salesforce and other frontline advocates servicing Microsoft Advertising customers. If a request has been received from a non-Microsoft domain email address (such as @contoso.com), it will be closed automatically as Cancelled when it is not possible to know who the requestor is. If this happens, inform the requestor of the following:

“Unfortunately, we cannot accept requests from non-Microsoft email domains. Please resubmit your request using your Microsoft email address and we will be more than happy to help. Otherwise, you can contact Microsoft Advertising Support."

Where it is possible to identify the requestor from the non-Microsoft domain as a member of the Microsoft account team, then update the alias in ASD to their Microsoft email and proceed with the request. Also let the requestor know that they submitted with their personal email and to ensure for next time that they are not logged in with this before submitting.

Note: In instances where the requestor added a non-Microsoft domain email address, either in the “To:” or in the “cc:”, you must inform the submitter that the non-Microsoft domain contact was removed and ask them to provide the Microsoft email address of this contact to include them on the next email communication.

7. Using the standardized Output template for AES services

Use the following Output Template and follow the instructions provided below when sending a requestor an output file for any services requiring to use it, for example, proposals, skipped, and error items.

  1. Open the Output Template and select Save As to create an editable version of the document on your PC.
  2. Update the template’s title depending on the standard naming convention per service. For example, Advertiser Name_XID_Service Name_Date.
  3. In the Special Requests tab, copy and paste any special requests or instructions provided in the submission template.
  4. In the Proposal Overview tab, complete all fields (such as Advertiser Name, Date, and Account Number(s)). Then delete rows for each Proposed Change not included in this proposal.
    • Note: You can hold the Ctrl Key and select the row numbers to highlight and delete all rows to be removed at once.
  5. If more than one account is included in the output request, right-click on the Proposal tab, select move or copy, and then check the create a copy box. Repeat for each proposed account.
  6. Right-click each Proposal tab and select Rename to change the tab name so it represents the proposed change (like Keywords).
  7. In each proposal tab, copy and paste the output file worksheets: Paste column headers into Row 3, with output data underneath. Repeat for each proposed change.
  8. Important reminder! Update all column headers to reflect the correct import-friendly headers contained in the Editor Import Template file. This file can be found in the Microsoft Advertising UI, on the Import Campaigns > Import from File section.
  9. If any errors were encountered during upload or implementation, copy and paste the error report on the Error Report tab. This includes but is not limited to “skipped items” during the import process.
    • If assignment produced no errors, delete the Error Report tab.
  10. Important reminder! Delete the User Instructions tab before sending the file to the SR requestor.

Note: This output file is the only output document you need to provide to your requestors, and it replaces any legacy output files you may previously have used. The Proposal Overview tab contains prepared scripting for requestors to use when sharing the output file with their customers. The scripting here doesn’t necessarily reflect the process work that has been completed by BOS.

8. Customer data security requirements

Customer data protection is important to Microsoft and every effort must be made to protect personal data and its collection or processing using appropriate security measures.

All classifications of data require a 'medium' level of data protection, which should be achieved through encryption any time the content is emailed, whenever possible. High Business Impact (HBI) content must be locked regardless of storage or transit -- most internal systems are only Medium Business Impact (MBI) secure by default and emails are susceptible to interception.

Standard AES tactic work shared via output files does not need to be password protected. This does not contain personal data.

See KB 3174641 Microsoft Advertising: Handling Personal Data team guidance for details on data classification.

  1. First, KB 4056786 Microsoft Advertising: Data Transfer Management must be used to collect documentation containing MBI/HBI from customers (for example, monthly invoice setup documents, financial audit documents and bank statement screenshots).
  2. Any documents containing personal data or account data must be encrypted with a password before they are relayed via email to any other internal team or system. This usually includes all MBI and HBI documentation. As a best practice, all MBI and HBI data should be encrypted both in transit and at rest. Options for encryption:
    • For Microsoft Office documents:
      1. Access File from the ribbon.
      2. Select Protect Document, and then select Encrypt with Password.
      3. Enter the password "CSSBA" twice to enable encryption.
    • For all other documentation types, use 7-Zip:
      1. Under the Encryption section, enter the password "CSSBA" twice.
      2. Set the Archive Format to zip.
      3. Select Add to archive. ​
  3. Send passwords in a separate email to the recipient as needed. Partners will be notified in advance that documents may be locked and what the password should be; however, it may be necessary to remind them the document is password protected and how to open it.

WARNING:

You must encrypt all customer documents to legally comply. This includes the Customer Setup Form (CSF), Financial Audit Statements, Legal Documents, Tax Certificate, or any documents containing customer information.

10. Documenting requestor engagement

You are required to document all interactions with the submitter of a case, including any next-steps or commitments made to the submitter. Ticket case notes should contain enough detail to allow you (or others) to gain a complete and accurate understanding of the interaction and next steps without needing to review the entire history of the ticket.

Document within the ticket notes should include:

  • Update Provided - The update you provided to the submitter.
  • Expectations - The expectations you set with the submitter.
  • Miscellaneous - Any miscellaneous information that may be of benefit to the advocate who owns the ticket. If the submitter provides new information which was not originally shared with the existing ticket owner, make sure to document it in the ticket notes so the existing ticket owner is aware.

10.2 Chat transcript attachment

The following provides guidelines on when and how to attach a chat transcript on parent or child tickets via ASD.

When to attach:

If you were able to converse with the requestor through the Skype/Teams tool, they are required to attach the chat transcript in real-time to each of their child/parent tickets via ASD. This is to either relay any clarification/pushback regarding the request or alert if there was an additional request or special instruction provided by the requestor.

How to attach chat transcript:

To properly document every interaction or communication that transpired between the advocates and the requestors using the Skype/Teams tool, you must always attach the chat transcript on each of their tickets provided that they meet the conditions stated above. Here is the step-by-step procedure on how to attach chat transcript via ASD:

  1. Open the ticket in ASD.
  2. After the Skype/Teams conversation has ended, select all texts by pressing CTRL + A, then press CTRL + C to copy.
  3. Select Case Notes in ASD.
  4. Press CTRL + V to paste the chat transcript in the Chat Transcript section.
  5. Update the Next Steps/Resolution Summary section, if necessary.
  6. Select Complete in ASD once done.

10.3 How to separate email communication and case notes in ASD for frontline service or support requests

To properly document the ticket activity, separate the email communication and the ticket activity documentation (case documentation). This will make the communication professional as well as avoid any confusion of information (for example, multiple ticket stamps). See below is the step-by-step process:

  1. Open and edit the assigned L1 or child ticket.
  2. Edit any necessary fields.
  3. Go to the Case notes section on the left navigation panel in ASD.
  4. Copy and paste your email communication above the L1 notes, using a separator if needed.
  5. Select Complete.
  6. Reopen and Edit the ticket again.
  7. Select the Case notes section, delete the old email communication and replace it with your new case notes.
  8. Select Complete.

Note: When updating ASD case notes, email communication should be retained in a thread form, except for the case documentation. Make sure to place your latest communication at the top of the email thread to allow easier visibility when reviewing in the tool. Any communications will not be removed since these had been previously stamped in the ticket.

11. Reopening tickets

If a requestor has contacted BOS regarding a closed ticket, the existing ticket should be reopened, but only if it has been closed for less than seven business days. If the ticket has been closed for more than seven business days, a new parent ticket needs to be created.

Important: You are responsible for creating the new parent ticket and shouldn’t ask the submitter to create a new request.

12. Conducting proactive outreaches for advocate errors

If an error is identified while reviewing tickets and it directly impacts the client’s Microsoft Advertising account (for example, account-related issues such as keyword capitalization preferences and non-ticket related issues such as coding), you will be assigned to conduct a proactive outreach and offer to correct the mistake.

The following messaging is used by a team manager or to coach communication with the requestor as the first step of the proactive outreach:

“Hi {Requestor Name},

I’m {Name} from the Back Office team. As part of our quality monitoring program, we review tickets on an ongoing basis to ensure our advocates can meet the needs of requestors.

We have found an error on your request {Ticket Number}. The error is {insert detailed, specific outline of the transgression}. The impact to your client is {detail impact]. To resolve this issue, we recommend {insert detailed recommended restitution steps}. Would you like us to proceed?

If you do, we will have an advocate assigned to your request and have the issue correct as soon as possible.

Thank you,

{signature}”

Note: In instances where an advocate forgot to include supplementary information, such as a skipped filed for upload requests, the missing information can be provided to the requestor without asking for the issue to be resolved. In instances where only minor rework is required, the error can be corrected and proactively resolved, instead of asking the requestor if the rework should be done.

12.1 Ticket creation and follow up cadence

A new parent ticket is expected to be created for the proactive outreach by the team manager or coach, even if the original request was documented in a parent or child ticket. Re-assignment of the ticket to an advocate should only take place after a request to have the mistake corrected is received.

Note: Whenever possible, it is recommended that the new parent ticket be assigned to the advocate working on the original request. This will enable them to learn from their mistake.

If no response is received after the first contact attempt, one additional follow up should be sent 48 hours from the time of the initial proactive-outreach email. The following message can be used in a follow up communication:

“Hi {Requestor Name},

I wanted to reach out once more to see if you would like assistance in fixing the error identified by our quality monitoring program.

Please feel free to reply or reach out to AES using {Segment Specific Alias} and we’ll be more than happy to help.

Thank you,

{signature}”

If no response is received after an additional 24 hours, close the new ticket.

13. Escalating to Global Support and Mission Control

13.1 Escalations for Global Support

If a technical issue is encountered for a Microsoft Advertising account or a Microsoft Advertising Tool (such as Editor, Intelligence, or UCM), escalate to Global Support by submitting a ticket to adsupport@microsoft.com and inform your manager/supervisor of the issue encountered.

Notes:

  • Managers/Supervisors should inform mailto:sbosesc@microsoft.com of the encountered issue for visibility.
  • Escalations should be raised from the ticket-level you are already working from, depending on the requestor. Do not create additional child tickets.
    • Example: An Account Manager (AM) request would be escalated from within the parent SR, while a CSS L1 request would be escalated from within the child SR.
  • For CSS L1 request, only one UCM reference number can be associated with it.

13.2 AdSupport escalation steps and email template

The following provide the step-by-step guidelines when escalating a tool issue to AdSupport:

  1. Inform your Team Manager or Solutions Coach of the issue.
  2. Once given an approval from your Support team, open your ticket in ASD and make sure that the ticket is also open in CAP.
  3. Select Edit, locate the Issue- Specific Details Required? beneath the Queue, then choose Yes. [Asset 4495501]
  1. A new section that will appear on the bottom of the left pane named issue – specific details. How to fill out the issue – specific details section and what should be coded:
    • Issue Category: Global Support Escalations
    • Issue Type: Tool Technical Issue
    • Tool Customer is having issues with (for example, Microsoft Advertising, UI, Intelligence, Editor, Ad Preview): Indicate the most appropriate.
    • XID: Microsoft Advertising Account Number
    • Error message or describe technical issue (specify if it timed out, froze or closed abruptly): Indicate the most appropriate.
    • Are the errors/issues intermittent or consistent: Select the most appropriate.
    • Time stamp when technical issue last occurred: Use this time and date format – MM/DD/YYYY HH:MM PST.
    • Screenshots or attachments (for example, Diagnostic file for Editor): Be specific with the name of your attachment.
    • SessionID/RequestID if the errors are in UI: Provide the information based in the Error in Microsoft Advertising UI.
    • Email address of the User: Provide your v-dash email (example, v-josmith@microsoft.com).
    • Describe what the customer is experiencing: Be specific with the technical issue being experienced.
    • Steps taken to reproduce the issue: Be specific with the steps you’ve taken when you encountered the error.
    • Steps taken to resolve the issue: Be specific with the steps you’ve taken to resolve the issue on your own.
    • Other findings from the investigation: Include other details that can be beneficial for the Global Support Team (for example, Microsoft Advertising help page).
    • Specify what help you need with this issue (if applicable): Include all items where help is needed.
  2. After filling out the issue – specific details section, proceed to the ucmt escalation section. How to fill out the fields in the ucmt escalation section and what should be coded:
    • Tier-2 Escalation Required?: Yes
    • T2 Receiving Team: Microsoft Advertising and Tools
    • T2 Sub-Team: Technical Support
    • Customer Support Region: Select the most appropriate.
    • Delivery Market/Language: Select the most appropriate.
  3. Once done, select Save, then select Transfer.
  4. The Transfer Template & Queues pop up window will appear. How to code per field:
    • Transfer Type: SMB Escalations to Tier 2 UCMT
    • Transfer Destination: UCMT Escalations
  5. Select Apply Transfer, then Complete.
  6. Since the ticket is open in CAP, just select the Refresh button (a page icon with two green arrows curved in a circle).
  7. Once refreshed, you will notice that the Owner will be changed to NONE.
  8. Select Refresh occasionally until the ticket is back to your bucket. This should be in less than two minutes.
  9. Once the ticket is back to your bucket, re-open the ticket in ASD.
  10. Select Edit, then select Email.
  11. On the left pane, make sure that the email addresses in the To: field is adsupport@microsoft.com
  12. Include your support team and your own v-dash account (for example, v-josmith@microsoft.com) on the CC: field.
  13. Don’t change anything in the email’s subject line.
  14. Follow the format below for the email body:

“Hi Global Support Team,

This is a help needed request for a technical issue encountered with {specify the name of the tool}.

Please see below details:

Account Number/s: xxxxxxx

Customer ID:

Issue: xxxxxxx

Date/time when the issue started: MM/DD/YY HH:MM (PST)

Repro steps:

  1. Xxxxxxx
  2. Xxxxxxx
  3. Xxxxxxx
  4. Xxxxxxx

I have attached the screenshot of error and the {upload/error/CAB} file that might be helpful with the investigation.

No. of advocates affected:

Affected alias/es: v-alias@microsoft.com)

Regards,

{Advocate Signature}”

14. Very large file share: Granting permissions and file sharing

In cases where a requestor and a BOS advocate need to share files that exceed 10 MB with one another, the Very Large File Share (VLFS) SharePoint site should be used.

Each partner BOS site will have at least one designated individual who is responsible for granting Account Managers/Coordinators and frontline services advocates access to the VLFS SharePoint site, should they need to share file(s) with BOS advocates. Contact mailto:sbosesc@microsoft.com if you require access to the VLFS SharePoint.​​

Note: Due to the SharePoint size limitations, BOS advocates are required to delete hosted files within one week of ticket closure.

Granting permissions

If you are one of the designated individuals responsible for granting access to the VLFS SharePoint, you may take the following steps to do this:

  1. On the VLSF SharePoint Site, go to the People and Groups page.
  2. Under Groups, select BackOffice Users.
  3. Select New.
    • Once you have selected the new link, a popup window will appear that will allow you to enter the aliases of any Account Managers/Coordinators or frontline advocates that require access to the VLFS SharePoint.
  4. In the Add People… field, enter the email aliases of the Account Manager(s)/Coordinator(s) that require access, or the partner credentials of the frontline advocate.
  5. Select the Share button to grant access immediately.
    • Note: If the frontline advocate cannot be granted access, the frontline advocate can request that their SME/Team Lead upload the file on their behalf.

Important reminder!

The advocate should check with their team manager that the submitter already has access to the Very Large File Share link before sending the link, as this will avoid back and forth communication with the submitter and impacting the SLA.
