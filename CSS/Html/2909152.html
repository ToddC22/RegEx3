Overview

Azure DevOps (formerly Visual Studio Team Services) allows customers to plan projects, collaborate, and manage code online from anywhere. Azure DevOps exists as a free standalone service, which does not require an Azure subscription for free usage and has its own portal.

This TSG provides guidance on how to assist customers who may contact ASMS with questions about this offer.

Azure DevOps Organizations are available for free to all customers at http://start.visualstudio.com/ and includes the following:

  • Unlimited private projects

  • 5 Repos and Boards (Basic) users

  • Unlimited Visual Studio (VS) Subscribers

  • Unlimited Stakeholders

  • Monthly amount of Build and Deployment Hosted Agent (e.g., 1800 minutes/month)

  • Monthly amount of Cloud-based Load Testing (e.g., 20,000 virtual user minutes/month)

Azure DevOps Organizations that stay within these constraints will not require an Azure subscription for billing. However, Azure DevOps Organizations that need to go beyond these free limits will need to be linked to an Azure subscription. For more information, see Set up billing for your Azure DevOps organization. For Azure DevOps pricing details, see Pricing for Azure DevOps.

Policy

Support Boundaries

  • The ASMS team supports billing dispute questions. Other Azure DevOps questions should go to that specific support team. See the Engagement and Escalation Path section of this TSG to determine the appropriate support team.

  • Azure DevOps is available under Azure support contracts and customers with paid Azure technical support can submit a SR via Azure portal in their preferred language.

  • Azure DevOps has its own SLA and support availability, so the standard Azure SLA for tickets and availability does not apply to the Azure DevOps team. For more information, please see http://www.visualstudio.com/en-us/support/support-overview-vs.aspx.

Customer Experience and Provisioning

Use the table below to see what experiences the customer may encounter via self-serve.

Disconnecting an Azure DevOps account from an Azure subscription

Customers can self-serve this scenario from the Management Portal.

The Remove billing action will result in the removal of the resource from the subscription and the Azure DevOps organization will downgrade to a free state.

Note: For more information see Removing a billing subscription.

User wants to change paid users in Azure DevOps

Customers may have an issue where all available paid Azure DevOps users have been added to the Azure DevOps account, but the Admin wants to remove a user and add another user into the account without purchasing more Azure DevOps users.

The owner of the Azure DevOps account or a member of the Project Collection Admins group can complete this action by doing the following:

  1. Navigate to <acct>.visualstudio.com/_user

  2. Remove one user and assign it to another user.

If the customer has attempted these steps and needs additional help, send a Problem to the Developer VSTS Depth queue.

Note: Select Azure DevOps as the routing product to engage VSTS Developer Depth queue.

Azure SA who created the account has been removed and lost access to the Azure subscription

The Azure Service Admin is the owner of the Azure DevOps account, so if this individual is removed, the Azure subscription that is linked to the Azure DevOps account cannot be accessed by the Azure DevOps account owner (i.e., Azure subscription admin has been removed). 

In these instances, the Azure Account Admin (AA) will need to add the Azure DevOps Account Owner as an Azure subscription owner, contributor, service admin, or co-admin.

For more information, see Adding a user to make purchases for Azure DevOps.

Note: If the AA is unable to perform this action, escalate to Azure DevOps via a DTS ticket in mseng.visualstudio.com (team project “VSOnline”).

User wants to move the Azure DevOps account from one Azure subscription to another Azure subscription

If a customer wants to use a different Azure subscription to pay for Azure DevOps services, instruct the customer to follow these steps outlined at https://docs.microsoft.com/en-us/vsts/billing/change-azure-subscription.

Note: If the customer experiences issues, instruct the customer to open a case with Azure DevOps support as this is out of scope for ASMS.

Azure DevOps account owner has left the company and the connected Azure account also needs to be transferred

This is a self-serve action that is out of scope for ASMS. The customer will need to disconnect the current connection between Azure\Azure DevOps and then reconnect to the new Azure sub.

Note: If the customer runs into issues while taking this action, instruct them to open a new case with Azure DevOps support.

 

Support Scenarios

Below is a list of scenarios in which the customer will contact support. Use this table to determine how to best assist the customer.

Canceling an Azure subscription with a connected Azure DevOps account

In these scenarios do the following:

  1. Inform the customer that the Azure DevOps account will be downgraded to a free state upon canceling.

  2. Once confirmed follow the standard Azure Cancellation workflow.

Note: A Subscription Cancelation will result in a downgrade of the Azure DevOps account to a free state.

For more information, please see TSG 2646764.

Azure DevOps account users lose access when the Azure subscription gets canceled

When an Azure subscription becomes disabled, Azure DevOps users may lose access as well. The Owner reactivates the Azure subscription, but when the Azure DevOps user goes back to Azure DevOps users Hub, the user still sees the user status is disabled.

This status is due to a system delay, in which the customer would need to wait until the re-enablement changes fully propagate through the system.

Note: Changes go through multiple hubs (CP to Azure RDFE to VSTS Metering Service to VSTS Account Services) before a user can see the impact of changes.

If more than one hour has elapsed and the issues persists, ASMS should follow the regular ASE engagement process for sync issues that lead to a wrong subscription state.

Send a Problem to the Developer VSTS Depth queue if additional help is required.

Authentication options for managing an Azure subscription

Customers may ask if they can use an OrgID?

Yes, for more information, see https://www.visualstudio.com/en-us/get-started/setup/manage-organization-access-for-your-account-vs.

Existing Azure DevOps user can no longer access the Azure DevOps account

Your customer may state that an Azure subscription is disabled and paid Azure DevOps users (i.e., other than the Azure account admin) cannot access the account.

Inform the Azure DevOps user that the owner (i.e., account admin) of the Azure subscription used for billing must reactivate the Azure subscription before the Azure DevOps user can access the Azure DevOps account. Developer VSTS team may sometimes assign a Problem or Task to ASMS.

Accept the Problem or Task from the Azure DevOps Depth queue if required to validate the subscription status and help the customer with payment.

Raising the soft cap for paid Azure DevOps users

The Service Admin or co-admin can buy Azure DevOps users up to soft limits that the UI of the Azure Management Portal enforces, which can be increased by the Azure DevOps Depth queue when needed.

In these instances, send a new Problem to the Developer VSTS queue for assistance.

SLA violations

Customers may contact support because they are unhappy about system features, availability, or performance.

In these instances, do the following:

  1. Contact Azure DevOps for usage info (see Engaging Azure DevOps below) related disputes.

  2. Send new Problem to the Developer VSTS queue for technical help.

  3. Then issue a refund within guidelines of Azure DevOps SLA: https://azure.microsoft.com/en-us/support/legal/sla/azure-devops/v2_0/

Charge is disputed

Customers may have questions/concerns regarding the purchase of Azure DevOps users or the usage of additional services (Build, Build and Deployment, or Cloud-based Load Testing) on their invoice.

In these instances, ASMS will assist the customer with the billing dispute and contact Azure DevOps for usage info (see Engaging Azure DevOps below).

Technical questions should be routed to the Developer VSTS queue via a new Problem.

Engagement and Escalation Path

Engaging the VSTS Depth queue

  1. To engage the VSTS Depth queue in MSSolve use the Developer VSTS queue (Problem). SLA is 1 business day, so if the Azure DevOps customer wants faster service, they can open a new case for Azure DevOps support and pay for faster service:

  2. Go to http://www.visualstudio.com/support-overview-vs

  3. Click the Azure DevOps (formerly VSTS) link.

  4. Scroll to and click Create an incident under Premium technical support.

  5. Do not transfer cases to Developer VSTS for issues not handled by ASMS. Customers should open new support cases using Premier Online, or the process above.

Engaging Azure DevOps

You may need to engage Azure DevOps for some scenarios such as billing disputes or verifying customer usage claims but first you must gain access to Azure DevOps:

  1. Go to //MyAccess to request both READ and WRITE access to the required Azure DevOps by searching for DevDiv Work Items and request access.

  2. Once access is approved, use this link: https://mseng.visualstudio.com/DefaultCollection/VSOnline/_workItems.

Note: This server requires 2-factor authentication. You will be required to sign in with a physical or virtual smart card, or your phone… even when connected to CorpNet. For more information, see https://microsoft.sharepoint.com/sites/itweb/securelogon.

  1. Open a new tab in the same session, paste the following into the address bar and click Go. This will create a new partially filled out DTS work item: http://aka.ms/wasms-vso-dts

Note: Ensure that the area path is changed from VSOnline\VSTS\Shared Cloud Services\Commerce to AzureDevOps\VSTS\Shared Cloud Services\Commerce.

  1. Complete the work item, including:

  • Case #

  • Azure DevOps account name (like Contoso.visualstudio.com).

  • A clear description of the issue, with as much detail as possible. For example, “On September 23 at approx. 2pm EST, customer queued 5 builds that never started but ate up ZZZ minutes of his paid build time. Please confirm this claim so we can process a refund.”

  1. Save the work item:

When you are ready to send it to the Azure DevOps Commerce team, save it once to trigger an email to that team and send it to their triage, then change the [STATE] field to 2 - CSS Triage and save again.

Note: No action will be taken on the request until this is done, as the work item must be moved to 2 - CSS Triage or it will remain invisible to Azure DevOps Support.

  • Azure DevOps Commerce team will comment the work item in the History section and\or email you. They will then assign the work item back to you with questions or a resolution. You should check on the work item periodically to see if it has been updated. You may also want to copy a link to the work item for future reference after submitting it.

  • You can also select the follow button in the TSF ticket to be emailed each time the ticket is updated by anyone.

  1. If the work item is sent back with questions, answer them in the HISTORY section then assign the work item back to the person who assigned it to you. For added visibility, you may wish to send them a courtesy email as well telling them you’ve done this. You can see who assigned you the work item on HISTORY.

Set these fields when sending it back:

  • STATE: 5 - PG Engaged

  • Who's Got the Ball? - Product Group

  • Assigned To: <The person who assigned it to you>

  • Once the issue is resolved Azure DevOps Commerce team will assign the work item back to the SE who opened it.

  1. It is your responsibility to close the work item when it is resolved by setting the STATE field to 6 - Closed. This is mandatory.
