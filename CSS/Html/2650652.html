1. Summary

1.1 Content overview 

This article is an overview of MPN technical benefits, how to handle support requests and refer to online submission for Partner Advisory (Technical Presales and Deployment Services, TPDS), On-premises Product support incidents* (from ContactSupport or aka.ms/SupportForBusiness), Signature Azure and Dynamics 365, Partner frontline and the Partner + TechNet Community.

By default, requests should be submitted online and only by phone if online submission is unavailable or not possible (for example, when online submission cannot be accessed or not available, such as when PMC partners cannot use online submission). As an exception the frontline agent can use the TPD internal online submission form to create a request on behalf of a partner. Use aka.ms/TPDForm, enter their email address, description, select Program MPN to submit a request. To learn more, see article 4561797.

If the partner needs Access ID and Contract ID to create On-Premise, Azure or Dynamics 365 support incidents, see public KB 2904733 Microsoft Partner Network Support contract activation and Access ID in Partner Center (aka.ms/MPNContractActivation).

1.2 Basic process

1. Check section 2 - Benefits and Entitlement. 
 
2. Check if the request is either break-fix or advisory. 

3. Check the product type and if On-Premises or Cloud.

4. If required, explain the value of online submission.

5. Check the online submission requirements. Request / create support contracts (if required).

6. Refer the partner request to the relevant online submission portal.

3.1.1 Probing Questions  

Partners are more technical than RSC agents. Therefore, use their knowledge to route the request. Pick the relevant question, rephrase it, and use it to clarify if a request is break-fix or advisory. 

Break-fix

Q. Do you need help with troubleshooting to diagnose and fix a specific technical problem / error / functionality that has failed?
(If the partner answers yes, this is a request for break-fix support. If the answer is “no” continue check for Advisory questions below.)

Advisory

Q1. Do you need advice on one of the following: Product configuration, design, deployment or product information, technical presales/compete to help close a deal?
(If the partner answers Yes, see Q2.)

Q2. Is the product / solution:

a. Cloud / Hybrid / To Be Determined. To be determined is when the decision has not been taken yet.
Refer to Advisory online submission.  

b. On-premises. Only a limited number of On-premises products exceptions are supported. Exception products are Win10, Win server, Skype for Business, SQL, products (see 3094111 http://aka.ms/advpolicy).

If the product is on-premises only and excluded from Advisory scope, refer to the TechNet community.

3.2 Value of online submission

Online submission is the default for all Advisory or Break-fix requests. You only manually create a request if online submission is unavailable.  If you receive push back, you may need to justify the value of Online submission.  

Online submission saves time, effort, and money in several ways:  

  • Time: You can submit requests 24x7 without waiting for the phone lines to open, navigating the phone menu, or waiting in a queue.

Note: Only severity A support incidents are worked 24x7, B or C wait until morning. Advisory is available during local business hours only. 

  • Time: You can supply more information, copy\paste and add attachments (for support incidents) to provide a better-quality description that enables the request to be resolved faster.

For Break-fix technical support incidents:

  • Money: Online Problem type and categories are linked to relevant KB articles and diagnostic utilities. You could solve the problem without using a support incident. 

  • Effort: Online is easy. Support incident Access ID and Contract ID information is only added once and then associated to your Sign-in (until the contract expires or you have no support incidents remaining).

4. Handling support requests

Use https://partner.microsoft.com/support - a single page where all services can be accessed.

4.1 Request screening

When a support request is received, the agent will complete the following: 

1. Search for the organization in Prose or VMT. VMT has associated individuals and partners migrated to Partner Center only show Admin users. 

  • If the individual is associated with the membership, you may discuss the specific benefits that the organization is entitled to. 

  • If the individual is not associated, discuss generic entitlement and continue. 

2. Ask the individual for a brief description of the technical issue and the Microsoft product. This will help you to identify the type of request and how to route it. 

3. Check Benefits Entitlement to assess the partner’s options. You may refer the partner to KB 2987884 Microsoft Partner Network technical benefits summary (http://aka.ms/MPNTech) for the partner to check their own entitlement and learn more about their support options. If they want details to create a support request see KB 2987884 Microsoft Partner Network technical benefits summary.

4. Double-check if the request is Break-fix or Advisory and use probing questions. If a PTR agent is available, transfer if required.  

5. Advise on the relevant benefit based on entitlement and request type break-fix or Advisory. Refer to online submission and explain the value of online submission if required. 

For Advisory and Program Support a Partner Center AAD Sign-in is required. MSA account sign-in will fail. For on-premises technical support the partner must can use an AAD account to add MPN product support. However, they must use a MSA sign in to enable support to be purchased for old products not supported by MPN Product support (aka.ms/n-1).

6. If you use https://partner.microsoft.com/support, search using issue keywords or click Create a ticket in Partner Center.

6.1 On-Premises – Key points

MPN products support incidents are: Partner support Gold 20pack, Partner support Silver 15pack, Partner support MAPS 10pack.
Note: Signature Cloud support is cloud product only – Signature will not work for on-premises products

  • For detailed advice on all Technical support incident submission see aka.ms/MPNTechSupport and then select product version Windows.
  • For on-premises products partners use the Support for Business portal and online submission – see aka.ms/SupportForBusiness
  • If the partner needs to know how many support incidents are remaining, see article 4559866 Informational: Partner: How many Partner support incidents are remaining?

  • The partner needs a credit card and to use an MSA sign-in to buy on-premises technical support incidents (Pay Per Incident, PPI) online. An AAD account eg Partner Center (AAD) cannot be used to purchase paid support due to a bug – see article 4490370 Professional Support (pay-per-incident) support plans are not available for Azure Active Directory accounts

  • The workflow can use a Microsoft account (MSA) or a Partner Center (AAD) account. The account does NOT need to be associated to the partner organization because the partner ID is not required.

  • If the partner is entitled to support (see aka.ms/MPNtech) and the contract is missing, the partner can request contract activation (see aka.ms/MPNContractActivation) and use the Access ID and Contract ID as soon as the contract is created. New support contracts are not shown on the MPN portal for 3-4 days.
  • The partner needs the Access ID and Contract ID to use the support contract on first use. Thereafter, the contract will be associated with the sign-in and remain until it expires after 1 year (or 1 year + 30 days) or all the incidents are used.
  • Partner with their MPN organization migrated to Partner Center can see Access ID and Contract ID in Partner Center - see below.

  • If you receive push back, see Value of online submission

Support entitlement

Pay Per Incident (PPI) support is entitled for products in mainstream and extended support (see Microsoft product lifecycle policy)

The MPN product support incidents are entitled for the most recent product (N). They are entitled for the next most recent product (N-1) only if the product is in mainstream support(Search product lifecycle). For most products, only the most recent products is supported e.g. Windows server 2019 is the most recent server version, the next most recent version is Windows Server 2016 and it ended mainstream support October 2020. Older Windows Server versions eg 2012 are not entitled to support. Therefore, partners need to use Pay Per Incident for Windows Server 2012. See aka.ms/N-1

7.2 Break-fix cloud summary

For Cloud service eg Azure, Dynamics 365 etc Partners use Delegate administration to create support requests for their customers. Delegated administration is where a partner has a relationship with a customer, the partner user creates a support request On behalf Of (OBO) their customer. The request is created on the customer tenant, by the partner user.

Setup delegated administration

1. Enable Delegated Administrative Privileges (DAP) at the customer level

      a. Enabled by default for CSP and Indirect provider – see Azure CSP Delegate administration

      b. For Advisor partners: Select Customers, Request a relationship and email the customer global admin – More information

2. Enable the partner user to support your customer by assigning the admin agent security role for users on your partner tenant

Create support request

3. The partner admin agent user opens the customer in Partner Center and selects the relevant workload eg Azure, Dynamics 365 and creates a new request

[Asset 4535868]

4. Start with the Partner dashboard > Support > Customer requests. 

5.  The relevant products portal is opened and a support ticket is created on the customers tenant. 

6.  Depending on the product, a support contract may or may not be required.

  • Support contracts are required for Dynamics 365 (Power Platform and Lifecycle Services and Azure. You need the Access ID and a Contract ID, the first time you use the benefits. This is a one-off activity because the support contracts will be associated to your sign-in account for future requests (until benefits expire or no support incidents remain). To see your Access ID and Contract ID in Partner Center or request contract activation see 2904733. Competency partner should use Signature Cloud support incidents.
  • Support contracts are NOT required for Microsoft 365 admin center requests eg Office 365, Enterprise Mobility Suite, Microsoft Intune or Windows.

7.3 Office 365, Intune, Enterprise Mobility Suite

The section below is based on the public KB aka.ms/MPNTechSupport, then select Office 365, Intune, EMS - Step by Step: Office 365, Intune, Enterprise Mobility Suite

An Advisor Partner activates "Partner features" and requests Delegated Administrator Privileges (DAP) via email. The customer global admin approves the relationship to enable DAP at the partner level.

CSP, Indirect Provider and Indirect Reseller have a relationship with all their customers and therefore DAP is enabled at the customer level, by default. If the Indirect Provider chooses not to enable DAP for the Indirect Reseller; the Indirect Reseller may request a relationship to enable DAP for themselves.

The Advisor or CSP Partner needs to assign a DAP security role at a user level, to the relevant trusted employees. The DAP security role is Admin Agent (Partner Center) or Full Administration (Microsoft Admin Center).

The Partner selects the customer in Partner Admin Center or Partner Center. When a service request is created, the customer is opened in a new session, under the delegated administrator context. Service request creation is the same as normal, and the request is created by the partner on behalf of their customer.

1. Requirements

1.1 Delegated Administrator Privileges (DAP)

Delegated administration is where a Microsoft Partner Network Partner has a relationship with a customer and creates a support request On behalf Of (OBO) their customer.
The Advisor Partner activates "Partner features" and requests Delegated Administrator Privileges (DAP) via email; the customer global admin approves the relationship to enable DAP at the partner level. The advisor partner sends the customer a delegate administration offer via an email or when they send a trial invitation or purchase offer (enable “Include authorization for delegated administration”). See Help for partners
