Procedure: TR: Extended Security Update (ESU) Call Handling

[Asset 4489629]

An Extended Security Update is an agreement which allows a customer to receive assistance with a specific product beyond the End of Support (EoS) date for that product. This support model replaces the previous Custom Support Agreement (CSA) model. 

ESU replaces CSA for new EoS products only

ESU replaces CSA for all new products that hit EoS moving forward, products which hit EoS prior to the launch of ESU may still have active CSA contracts in place.

Moving forward, any products which reach their EoS date and for which Microsoft will still allow customers the option to purchase continued assistance will follow the new ESU model outlined in this article.

The ESU agreement itself is not a support contract, it allows a customer to continue to receive security updates and makes the customer eligible to receive assisted technical support using standard support entitlement options.

Therefore, to receive assisted technical support for ESU products past their EoS date, the customer must have an ESU agreement for the specific problem product involved AS WELL AS an active support account (Professional / Premier / Unified / SA).

Separate ESU required for each product / edition

Having an ESU does not make a customer eligible for support of every ESU-enabled product (it’s not an “all products” offer). Customers choose the specific products they want an ESU for, and the customer must have a line item (in the ESU customer list) for the problem product they are calling about.

For server products, the customer must have a line item for the product and the edition.  E.g. having a SQL Data Center line does not allow support for SQL Standard, and vice-versa.  If the customer wants ESU for both editions in this example they would have the option to purchase ESU support for both.

Who

  • Commercial CS Post-Sales Advocates (aka Tech Routers)

What

  • This article details how to verify a customer’s entitlement for ESU products, and then create and route their cases

When

  • The steps outlined in this article should be used whenever a customer calls for support with one of the following products for which ESU support is available:
    • SQL Server 2008 or SQL Server 2008 R2
    • Windows Server 2008 or Windows Server 2008 R2
    • Windows 7

How to Identify ESU Products in Service Desk

The Service Desk SAP field Product Version will include “ESU” at the end of the product name, which will allow advocates to identify that an ESU is required to open a support ticket.

Additionally, not all versions qualify.  For example, only Windows 7 Professional and Enterprise editions have an ESU option.  Other products, such as Windows 7 Home, are not eligible for ESU support and will not be included in the Service Text SAP fields.

Identify In-Scope Support Requests

Note

As of July 1, 2020, the following products offer ESU support, and therefore are supported via the steps outlined in this article:

  • SQL 2008/R2
  • Windows Server 2008/R2
  • Windows 7

Other products that reach their EoS date may also move to this model (though the decision is on a product by product basis, not all products will have ESU offered); therefore, the steps below are purposely created generically rather than specifying particular products. They key to identifying that this article should be followed is the inclusion of “ESU” in the product name.

  1. Follow standard Service Desk processes to assist customers calling in to open a new technical support case.
  2. When populating the product SAP fields, if the product name includes “ESU” this indicates the product has passed its EoS date and an ESU is required to receive assisted support from Microsoft.
    • E.g. for SQL 2008
Product Family Product Product Version
SQL Server SQL Server 2008 SQL Server 2008 Enterprise ESU
SQL Server 2008 Standard ESU
SQL Server 2008 R2 SQL Server 2008 R2 Enterprise ESU
SQL Server 2008 R2 Standard ESU

Verify ESU

1. To verify that a customer has an ESU, visit this ESU Customer List.

Important

The ESU customer list link above was updated for the inclusion of Windows 7 (January 13, 2017).  If you have favorites//bookmarks/shortcuts to the list, they must be updated to point this list.

2. Ask the customer if they have their Extended Security Update Agreement Number,

  1. Use this specific scripting:

    "<Problem Product Name> has reached its end of support date. In order to receive assisted support moving forward, you must have an Extended Security Update agreement in place.

    If your organization has such an agreement for that product in place, and     you have the agreement number handy, I can quickly pull that up.

    If you're not sure, or if you don't have the agreement number, I can also try searching by organization name."

    URGENT

    It is essential that agents ask for the ESU agreement # in an manner which does not suggest that it is required in order to open a support case.  Feedback has been received that some customers infer that is the message from Tech Routers, so this specific scripting has been implemented to prevent that assumption on the customer's part.

  2. If the customer says they do have the agreement number, ask if it is a 7-8 digit number, or a long alpha-numeric string (36 digits including hyphens).
    • For 7- or 8-digit Agreement Number, collect the entire number for searching in the step below.
    • If the customer has the 36-digit alpha-numeric agreement type (actually a Tenant ID), ask them for the first 7 digits before the first hyphen and use that value when searching in the step below.
  3. If the agreement number is available, you can quickly search the list by using CTRL+F to search for that number.
  4. You can also use CTRL+F to search for the customer’s organization/company name from the list (however this is less efficient as acronyms, abbreviations, etc., may be included in the list’s org name)

3. When reviewing the ESU support lines for a customer, it’s essential to apply the following rules:

  1. Server Products
    • Both the initial release and R2 are covered by the same line item, if the customer has either then they are eligible for support (e.g. while Service Desk differentiates initial and R2 releases the ESU customer list does not)
    • The customer must have the product and the applicable edition (e.g. Standard, Enterprise, Datacenter)
    • If the customer’s organization needs support for multiple editions of the same product, they have the option to purchase ESU support for each edition for which it’s required separately
  2. Windows 7
    • There are line items for “Windows 7 ESU” and “Windows 7 ESU for M365” products on the ESU list
    • As Tech Routers cannot differentiate whether an instance of Windows 7 came with M365, as long as the customer has a Windows 7 option continue with support case creation

4. If it’s determined that the customer has an ESU, then proceed to case creation.

  1. Otherwise, set the customer’s expectations that assisted support for their issue from Microsoft is no longer available, unless an Extended Security Update is purchased for their product.
  2. If the customer want information on purchasing an ESU, refer them to the Lifecycle FAQ-Extended Security Updates

Create Case

  1. After the customer’s ESU is verified, case creation follows standard guidelines.
  2. Some sort of support account is required that will be used for entitlement of the customer’s case (normal Professional/Premier/Unified/SA options apply)

Important

The ESU itself is not an entitlement option.  It only determines that a customer is able to use some actual support agreement to open new technical support cases for the ESU product.  Entitlement will still be via the Access ID / SA ID as normal, the ESU agreement number is not used in Service Desk entitlement.

  1. Create the Service Desk case populating the remaining required fields and following standard Service Desk procedures.

Tip

Support Topic and Support Subtopic may be grayed out and unavailable, as all issues for a product may route to a single queue.

Transfer Case

  1. Verify if the default transfer queue is the Global Misroute Queue (TIP:  It should not be, as applicable routing rules should be in place)
  2. If the default queue is anything other than the Global Misroute Queue, transfer the case to the queue displayed.  No Manual Override is required.
  3. If the default queue is the Global Misroute Queue, the use Manual Override to route the case.

[Asset 4512953]

  1. Click on Manual Override and search for the queue name below for the problem product:
    • SQL 2008 & 2008 R2:  MSaaS SQL Core
    • Windows Server 2008 & 2008 R2
      MSaaS Windows User Experience Vendor Tier 1 - CVG NA
    • Windows 7
      MSaaS Windows Devices and Deployment Vendor Tier 1 - CVG NA
  2. Select that specific queue name (for SQL Server 2008, for example, do not choose the option that ends in "Tier 1")
    [Asset 4512955]
  3. Click Next
  4. Click Submit to transfer the case

TAM Questions about ESU

If a TAM contacts customer service with questions about the ESU process, including how to get their customers added to the ESU list, provide them with the applicable email alias(es) below to send their questions to:

  • Windows 7 Client ESU issues
    • EOSquestions@microsoft.com
  • SQL Server and Windows Server 2008/2008 R2 ESU issues
    • Eos2008@microsoft.com

IMPORTANT: These are internal email addresses and should never be provided to customers/partners.
