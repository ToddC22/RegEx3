Getting Started

In order to help our customers fix a problem quickly, we must be able to understand and describe the problem they are experiencing.

Use the following table as a conversation guide to learn about what the problem is AND what it is not. The word “object” as a place holder for: an operating system (OS), an application, a device, a PC, a Server, a service, a DLL, a driver, etc.

IS

IS NOT

WHAT

What specific object(s) have the problem?

What is the specific problem?

What similar object(s) could reasonably have the problem, but do not?

What other problems could reasonably be observed, but are not?  (e.g. does this happen on all versions of the object?)

WHERE

Where is the object when the problem is observed (geographically)?

Where is the problem on the object?

Where else could the object be when the problem is observed, but is not?

Where else could the problem be located on object, but is not?

WHEN

When is the problem first observed (clock and calendar time)?

When since that time has the problem been observed? Any pattern?

When, in the object’s history or life cycle, was the problem first observed?

When else could the problem have been observed, but do not?

When since that time could the problem have been observed, but was not?

When else, in the object’s history or lifecycle, could the problem have been observed, but was not?

EXTENT

How many objects have the problem?

What is the impact and/or size of a single occurrence of the problem?

How many objects have the problem?

What is the trend? (spreading? Increasing or decreasing in frequency? Does it start off with mild symptoms and then become worse?)

How many objects could have the problem, but do not?

How many problems could there be on each object, but are not?

What could the trend be, but is not? (number of occurrences of the problem?)


Once you have a well understood problem description, update the support case's title to clearly reflect what the problem is.


Pro-tips

 Be careful with “intermittent” and “random”. Try to include bounds to the timing.

 When you ask the customer to be specific about how often or when an issue occurs you will often get resistance. Try to get them to open up with “I just want to get a rough estimate of when the issue started. Do you remember the first time the issue occurred?”. Using the “rough estimate” phrase allows the customer to provide an answer without feeling like they have committed to an “exact” answer.

 When recording rough estimates, use the “~”. The tilde is also used to denote when something is an estimate, such as ~24 meaning approximately 24.

 Some issues start with less impactful symptoms and then get worse. It these scenarios you can outline the problem in phases like:

 Phase 1: initial symptoms

 Phase 2: additional symptoms

The server hang issue

Here is an example problem description that does a good job of conveying the problem:

Good Example

The customer has 3 servers in their Texas location that are experiencing an intermittent hang. The issue started occurring on Jan 12th . The issue occurs on one of the servers at least once a day, but the issue has occurred up to 5 times on a single server in a single day. The issue started with TXSVR2 and over the course of about 2 weeks it has spread to the other two servers. All the servers are running Windows Server 2012 R2. All the servers are acting as file shares for user data. When the server hangs:

When the server is having the issue (a.k.a. while it is “in state”).

 WE CANNOT

 RDP to the server

 Prompted for credentials by the Remote Desktop Client, but the connection never completes

 Access file shares

 When accessing shares hosted on the impacted server, the connection starts, but after a long duration (~2 minutes) the connection will fail with a “Timeout” error message.

 WE CAN

 Ping the server

 Use Computer Management to remotely connect to the target server

 We can connect to the RPC endpoint mapper port using PortQry.exe -n <ServerName> -e 135 and it does display all the registered RPC endpoints.


The customer has other servers with the same OS and same work loads in their site in EUROPE, but they are not getting any reports of the issue. When the issue occurs, all the users (~2400) lose the ability to access key business data stored on the servers. The issue only occurs during local time business hours, never between ~7PM to 6AM. Seems to be correlated to load. We have not found a way to force the issue to occur.

Bad Example

The server hangs.


Note that the good example addresses key points of what is and what is not broken on the server when the issue is occurring. It also outlines where, when, and to what extent the issue is occurring. It included timing information about how frequent the issue occurs and what the impact is to the customer when it happens. Not every point in the table above makes sense for every case, but including as much details as you can allows you and the engineers you work with to understand what tools and troubleshooting is going to possible for a given scenario.
