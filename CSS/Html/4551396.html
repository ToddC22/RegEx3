ConfigMgr clients become Inactive in console due to outdated Defender

Also described as:
ConfigMgr Client stops communicating with site​
ConfigMgr Client stops processing machine policy​
​
Additional symptoms:​
​Querying MSFT_MpComputerStatus class hangs​
Policyagent.log doesn't reference machine policy for days/weeks​
Defender is not in use or hasn't been updated in months, typically due to use of 3rd party AV​
Windows Defender AV service will likely be stopped​
Appears to be some relationship to OS upgrade - some report 1903 to 1909 was when behavior was first seen​
Sometimes ccmexec service fails to stop
CCMExec has a dependency on successfully querying the Defender MSFT_mpComputerStatus class. When querying, it expects an error returned or, 0 - success. At some point Defender installation path and WMI namespaces changed. When ccmexec attempts to query the MSFT_MpComputerStatus Defender class, we hang because no zero or error is returned, which ultimately results in CM client becoming non-functional. This likely occurs because Defender has become extremely outdated.
​
Over time Defender has become a core Windows component versus an external installation. ​
​
Previous default install path of:​
C:\Program Files\Windows Defender​
Has become:​
C:\ProgramData\Microsoft\Windows Defender\Platform\<Build>\​
​
Previous related namespace/class​
ROOT\Microsoft\protectionManagement​
ROOT\Microsoft\protectionManagement:MSFT_MpComputerStatus​
Has become ​
ROOT\Microsoft\Windows\Defender​
ROOT\Microsoft\Windows\Defender:MSFT_MpComputerStatus​
​
Currently, the easiest method to confirm issue exists is to run the following PoSH statement:​
Get-WmiObject -Namespace Root\Microsoft\Windows\Defender -Class MSFT_MpComputerStatus​
​
If this hangs, you have likely confirmed the issue exists and can kill the PS window and proceed with data collection and workaround.​

Security has confirmed that in a normal scenario with Windows 10 systems, using 3rd party AV the Windows Defender AV and other services should not be manually stopped/disabled or via policy. Basically, they will run in a passive mode but need to be running to be maintained, in order to take over if the 3rd party AV is ever no longer used. Please see the Warning on the link below:

Workarounds:
​
Rename C:\ProgramData\Microsoft\Windows Defender\Platform\4.18.1910.4-0\mpclient.dll (and/or C:\Program Files\Windows Defender\mpclient.dll if it exists), and reboot​
This will allow CM client (and 3rd party AV) to be functional but will prevent Defender from assuming default AV role if 3rd party AV is removed in the future​.
or​
Uninstall 3rd party AV​
Update Defender to latest version for appropriate platform (this might require running updateplatform.exe manually - possibly from MU - as CM client remains non-functional)​
Reinstall 3rd party AV​
​
Ex:
Update for Windows Defender Antivirus antimalware platform - KB4052623 (Version 4.18.1911.3) ​​
https://www.catalog.update.microsoft.com/Search.aspx?q=4052623​

Some customers have also added/updated the following Registry keys as a successful workaround:

    [HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows Defender]

    "DisableAntiSpyware"=dword:00000000

    "DisableAntiVirus"=dword:00000000

    "PassiveMode"=dword:00000002

   

    [HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\WdBoot]

    "Group"="Early-Launch"

    "Start"=dword:00000000

   

    [HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\WdFilter]

    "Start"=dword:00000000

   

    [HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\WinDefend]

    "Start"=dword:00000002


There may be challenges in renaming the mpclient.dll file. ​
Depending upon permissions, you may have to grant explicit permissions (Full) to the administrator or manipulate TI perms, or potentially rename from CMD prompt running as Local System.
​
It has been suggested that we might utilize Movefile.exe from Sysinternsals. Ref:​
https://docs.microsoft.com/en-us/sysinternals/downloads/movefile​
​
You may be able to move or delete the file since Movefile puts the file in PendingFileRenameOperations regkey to be processed at next boot, then reboot.​
​
MoveFile usage​
The included MoveFile utililty allows you to schedule move and delete commands for the next reboot: usage: movefile [source] [dest]​​
Specifying an empty destination ("") deletes the source at boot. An example that deletes test.exe is:​
movefile test.exe ""  ​
​
Another utility mentioned, although it isn't supported by us, is RunAsTI. Ref:​
https://github.com/jschicht/RunAsTI​
