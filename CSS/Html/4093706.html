Known Issues

You should make sure the customer is working on Windows 1703 (RS3) where many fixes were implemented, and desired features are added.

See following info in compatibility issues and other limitation (note DA and CSC):

https://docs.microsoft.com/en-gb/windows/security/information-protection/windows-information-protection/limitations-with-wip

WIP is NOT protecting against malicious users

One must remember the goal of WIP – to prevent the accidental sharing of enterprise information with private applications. It is trying to keep honest users honest. In a world where users work on private devices with their enterprise data and the other way around WIP is trying to prevent that the user accidentally copy from for example the cooperate Excel table to the private Hotmail email account.

Application connection to an IP rather than a name (Chrome/ Firefox not working)

There are apps that are special, and customers always run into issues with them. Any application that uses HTTPS and does not use our stack to connect to the server looks to WIP as if it uses the IP address only rather than the FQDN. As a result, the access to the IP address resource might be blocked. Examples are: Firefox, Chrome, WhatsApp desktop app.

OneNote notebooks can't be created/ viewed from UNC paths/ DFSN paths

OneNote PG:

We support opening of existing notebooks on network shares that are marked as WIP, but we don’t support creating new notebooks.  Local/UNC notebooks go through our legacy storage code that none of our modern clients support, and it introduces both technical and usability complications with WIP, so we don’t support it.  For migration + backwards compatibility, we allow opening of existing UNC notebooks, but they will behave as personal notebooks.  But not new, as this would further proliferate the confusion.  The supported WIP scenario is to create/move work notebooks on SharePoint/ODFB which is fully managed.

Document can’t be changed from Enterprise to Personal

Policy can be set to “Hide Overrides” which will remove the personal option for work files in Explorer (RS2+) and Save As (RS3+).  However, a determined user could find ways around this – we don’t try to protect against malicious authorized users (see above).

 

“Hide Overrides"

WIP looks for inappropriate data sharing practices and stops the user from completing the action. This can include sharing info across non-corporate-protected apps and sharing corporate data between other people and devices outside of your organization. “

See: https://docs.microsoft.com/en-us/windows/threat-protection/windows-information-protection/protect-enterprise-data-using-wip

 

DFS path can’t be accessed

There are cases where customer notices that DFSN paths can’t be accessed/ are not seen as enterprise. Accessing the fileserver directly works. For the enterprise context to be identified by WIP the accessed path (\\contoso.com\dfsRoot\share) has to match with the enterprise namespace defined in the WIP policy (contoso.com) and the enterprise IP address range defined.

In this case the domain name contoso.com is resolved to a list of IP addresses by DNS, which are the DCs of the domain. All these IP addresses must be specified in the enterprise IP address range in the WIP policy.

Some file extensions are not encrypted

You will find that some file extensions are not encrypted. This is expected, and the list of extensions is hardcoded. However, there is a setting in the policy that can be adjusted:

https://docs.microsoft.com/en-us/intune/windows-information-protection-policy-create#add-encrypted-file-extensions

In the EnterpriseDataProtection CSP you can set:

Settings/SMBAutoEncryptedFileExtensions

Added in Windows 10, version 1703. Specifies a list of file extensions, so that files with these extensions are encrypted when copying from an SMB share within the corporate boundary as defined in the Policy CSP nodes for NetworkIsolation/EnterpriseIPRange and NetworkIsolation/EnterpriseNetworkDomainNames. Use semicolon (;) delimiter in the list.

When this policy is not specified, the existing auto-encryption behavior is applied. When this policy is configured, only files with the extensions in the list will be encrypted.

Supported operations are Add, Get, Replace and Delete. Value type is string.

https://docs.microsoft.com/en-us/windows/client-management/mdm/enterprisedataprotection-csp

IE favourite bar does not work with encrypted (work) URL files

This is a known issue. You have to go to Edge to make this work.

Enterprise network domain names are NOT optional

It was previously documented that the setting in the policy for enterprise network domain names (or enterprise names) is optional.

This is wrong. This name along with the IP address ranges is needed for the firewall to identify the target address. If not specified, the firewall will prevent access to the home network routers http management site (for example on 192.168.1.1).
