ASMS – Usage Validation and Billing Verification

Submit a ModTrack request to update this TSG.

Overview

This article outlines how to handle customer questions about unexpected Azure usage and charges. Customers will often open support cases with the ASMS team because they have received their Azure invoice and see unexpected usage and charges on their bill. Use this article to inform your approach to these customers.

Note: Be sure to check Message Center for current alerts and issues. Customer usage and billing complaints are sometimes based on currently ongoing issues, and additional guidance might be available there.

Examples of what your customer might say:

  • I didn't change my usage patterns, so the extra high charges must be a bug or Microsoft's fault!
  • Microsoft needs to refund me because there is no way I used this much Azure!
  • Why does my bill show such high charges?
  • Why is there a sudden spike in usage this month? What caused it?
  • I am an internal user with the Azure Internal Registration System (AIRS), where is my usage report?
    • Send these customers to the detailed usage page on the AIRS site.
  • ASMS handles usage validation cases (not the AIRS team).
    • Engage your technical advisor (TA) or partner technical advisor (PTA) prior to engaging the concerned team to investigate.
    • Create a task to the TA or PTA using the below template:
      Task Title: Internal 15P Subscription Usage Report

      Account Owner Live ID:

      Subscription GUID/s:

      Date Range: Write out the date (i.e March 3, 2016 to May 5, 2016) rather than the short date (i.e. 3/3/2016 to 5/5/2016)

      Service:

Policy

The most important aspect to these cases is to set customer expectations appropriately. In these instances, you should note the following:

  • It is unlikely that Microsoft will be able to determine the root cause of the increased usage and associated charges besides what is already visible in the detailed usage data, which is available for the customer to download from their portal.
  • Microsoft does not scan or monitor customers' Azure deployment, virtual machines, networks, data transfers, etc., due to security concerns and customer privacy. For Microsoft to proactively scan customer deployments would be a major security and privacy concern, so we don't do it with the customer's best interests in mind.
  • While Microsoft can help educate customers on how to monitor their Azure usage, it is ultimately the responsibility of the customer to monitor their own usage.
  • There are a few things that Microsoft can check to shed some light on the increased usage, but the investigation can typically take 30 days or more.

If you encounter a case with an issue that is older than 90 days (making a Master Data Services (MDS) analysis impossible), consider giving your customer a Customer and Partner Experience (CPE) credit.
(The steps to do so are covered in the Consider a CPE Credit section.)

Support Scenario

1. Gather Your Customer’s Information

Gather all relevant information from the customer pertaining to the higher-than-expected usage or charges, including but not limited to:

  • All subscription-related data such as WLID, account ID, subscription ID, subscription GUID, etc.
  • The name of the Azure service or deployment that is showing the higher-than-expected usage or charges.
  • The exact date and time range of the higher-than-expected usage or charges.
  • Your customer’s invoice(s) showing the contested charges.

2. Triage Your Customer’s Request

Determine if this is a usage validation or billing verification request.

Billing Verification

Example: Your customer’s verbatim states “My invoice says I have been charged $30 for 750 hours usage on VM-xxx, but the usage file for the same month only displays a total of 350 hours usage for VM-xxx, what is the reason for this discrepancy?"

Review the usage file to ensure the validity of the customer’s claim using standard troubleshooting questions.

For example:

  • Do you also see the discrepancy that the customer is seeing?
  • Do the usage and invoice reports match and show identical information?
  • Are the VM SKUs the same across the reports?
  • Are there multiple instances of the same SKUs?
  • Is there a disagreement between the price your customer is getting charged vs. the price quoted on the Azure portal?
    • If this is your situation, engage your TA and the Product Marketing Manager (PMM) for the associated service.
      • If the price is wrong on the portal, the PMM will submit a change to get the price fixed on the portal.
      • If the issue is with pricing of the service on the bill, work with PayOps to resolve it. (See the PayOps Engagement TSG (TSG 2651579)).

If everything seems correct, but there is still an error in billing, engage PayOps using the PayOps Engagement TSG (TSG 2651579).

Usage Validation

Example: Your customer’s verbatim states “In last month’s usage file, I see charges or usage for Storage displaying over 1 TB per day for 2 days in the billing cycle, but on the other 28 days, it is 100 MB. I only had 100 MB data storage and did not make any changes to my services or storage.”

For these types of requests, engage the ASE team.

Engaging the ASE Team

The ASE team can attempt to obtain further compute, storage, and bandwidth usage verification data.

IMPORTANT: See the SQL Azure Billing Team (SQL Billing Validation Only) and Web Apps sections for important exceptions to filing an IcM.

File a Customer Reported Incident (CRI) to ASE and include the following details:

  • The name of the service and the date and time range of the disputed usage.
  • The total monetary value of the higher-than-expected charges.
  • Confirmation on whether the problem is still occurring, or if this was a past issue which is no longer occurring.
  • Attach the full Usage Report without editing the full version.
    • Note: Be sure to include the aggregated usage report along with the Recon file for CSP usage and billing validation.
  • Call out the VM or Storage account in question:
    • Potential questions for ASE:
      1. Was there a known Live Site Incident (LSI) at the general time of the customer's high usage?
      2. Was a "Failed Inner DC Whitelist issue" at fault?
      3. Can any names of the customer deployments/services/database tables/VM names/etc. be identified to help narrow down the issue?
  • Wait for the ASE response.

Note: See TSG 3093263 (ASE Engagement) for additional details.

3. Consider a CPE Credit

Due to the Azure platform limitation of collecting customer-specific usage info, it is unlikely that we will be able to give a definitive root cause of what caused the higher-than-expected charges. Based on the results from the previous steps, there may be general info we can give the customer, and we may also be able to engage other partner teams to help determine why the usage was so high even though we consider it valid. Possible scenarios for excess charges include a misbehaving database table or a loop in one of their website’s ASP.NET pages.

Our main goal is to satisfy the customer, so if you sense that they will not be satisfied with the information (or lack of information) we can provide, you can consider a one-time CPE credit:

  • Calculate the amount of the credit based on the total amount of the higher-than-expected charges.
  • Follow TSG 2646823 (How to Handle Azure Credit Requests) to process the credit.

4. Instruct the Customer on How to Monitor Their Usage in the Future

As explained above, Microsoft does not collect specific usage data from customer deployments. Customers may request the IP address involved in high data transfers, or usernames and passwords of users logging into their VMs and running up charges, etc. We do not have this information! This is up to the customer to monitor. Below are some useful tips for customers to monitor their deployments in the future.

  • Help your customers manage costs with Cost Management and Azure Budgets. More details are available in the following links: 
    • Manage costs with Azure Budgets
    • Create and manage Budgets in Cloudyn
    • Tutorial: Create and manage Azure budgets
  • For storage usage, the customer can enable Storage Analytics. Enabling Storage Analytics logs gives customers the ability to have per-transaction logging. The logs are quite detailed, and customers can do end-to-end tracing and debugging on their own. More details are available in the following links:
    • MSDN Storage Analytics Overview
    • Using Logs to Track Storage Requests
    • Storage Analytics Log Format
    • Monitor a storage account in the Azure Portal
  • For network-related usage, the customer can use a network capture tool such as Microsoft Network Monitor or Fiddler.
    • Network Monitor
    • Fiddler

Note: Fiddler is not supported by Microsoft.

  • For issues related to Virtual Machines with a Windows OS image, the Host OS Event Logs can be used.
    • Windows Event Logs
  • For Platform as a Service (PaaS) deployments, enable diagnostics in the application.
    • Enabling Azure Diagnostics in Azure Cloud Services
  • For Infrastructure as a Service (IaaS) deployments, enable diagnostics in ASP.NET applications.
    • Enabling Tracing in ASP.NET Web Services
  • Enable Internet Information Services (IIS) Logging in IIS:
    • Enable or Disable Logging in IIS 7
    • Enhanced Logging for IIS 8.5
  • Azure App Service:
    • Enable diagnostics logging for web apps in Azure App Service

'Poolmanager' Resource Group Not Visible for Customer

You may receive a case where your customer cannot find the resource group Poolmanager and the corresponding storage accounts md-xxxxxx on their portal.  However, this is the expected behavior when using Managed disks.
 
When Managed disks are deployed, the customer can view disks for their VMs, however, they cannot see the actual storage accounts & resource groups, because the storage accounts and corresponding resource groups are managed in Microsoft Internal Subscription.
 
Managed disks means the customer is using disks, but doesn't need to worry about disk availability and storage accounts as this is managed internally. Unmanaged disks can be found within the storage account in a subscription normally but with Managed disks, the actual storage account must be viewed by using "Disk Export".
 
Note: Managed disks are the same as Unmanaged disks, they are page blobs and have to be put into a storage account in a resource group.
 
For additional guidance on Managed Disks, see the following documents.
  • Azure Managed Disks FAQs
  • Introduction to Azure Managed Disks
 

Engagement Paths

MSaaS Support Portal

MSaaS Support Portal is an internal web site which lists all support teams and queues. Use this tool when searching for the correct queue to send a customer's case to. 

  • If you are unable to find the correct queue using the MSaas Support Portal, please follow TSG 3104062 (Misroutes) to engage the Duty Managers (DMs) for assistance.

Note: Find more guidance on the MSaaS Support Portal.

SQL Azure Billing Team (SQL Billing Validation Only)

Before escalating, check the SQL team OneNote for the solution - How to solve a billing dispute

Note: Do not assign SQLVM billing IcM tickets to the Azure SQLDB team, the impacted service and owning service should not be “Azure SQL DB” as they do not own the SQLVM services. All VM billing validation requests need to go to ASE.

If you are unsure if you should escalate to the Azure SQL DB queue, please verify with a TA before creating an IcM ticket.

Go to IcM.

  • Impacted Service: Azure SQL DB
  • Impacted Components: Billing
  • Owning Service: Azure SQL DB
  • Owning Team: Billing
  • Severity rules: Normally Severity is 3 for billing CRI investigation. You can raise a Severity 2 if you suspect there is a widespread billing issue impacting many customers.
  • Customer Subscription ID:
  • Server Name: This is the format xyz123.database.windows.net. Note: MySQL or SQL running on Azure VM (*.cloudapp.windows.net:port_number) are not the same product as SQL Azure Db.
  • Server Region: Determined by doing a ping from command line anywhere, so this operation can be performed by SEs.
  • Database Name: The specific database that the customer is reporting a billing issue with.
  • Usage report: Obtained from the above steps listed in this TSG.
  • Date Range of the issue: This is the start and end dates for when the customer has a concern about the usage or billing.
  • What is the specific issue reported by the customer?

Be as specific as possible. Ask specific questions about what you notice on the customer’s bill.

Web Apps
